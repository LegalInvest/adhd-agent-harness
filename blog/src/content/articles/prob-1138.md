# ChatGPT 之于 ADHD，就像 human-in-the-loop 监督 之于 LLM——但有人用错了

> 从同构视角实测 ChatGPT：它到底补上了哪一层执行功能？

## 从两个相似的崩溃现场开始

ADHD 人群最常崩溃的场景不是「想不出」，而是「一个人坐到桌前，却启动不了」。想法很多、能量很强，但执行功能——启动、维持、排序、切换、终止——像缺了一层操作系统。于是项目烂尾、计划流产、自我问责变成自我攻击。另一边，做 Agentic Harness 的工程师也怕另一种崩溃：LLM agent 生成速度快、能力广，但在高风险场景下会幻觉、会偏离目标、会把中间步骤当作最终答案。两边的解法听起来完全不同，但结构惊人地一致：都需要一个**外部在场 + 人在回路**的约束层。

这不是浪漫比喻。ADHD 的神经生物学特征之一是多巴胺能功能失调，尤其是中脑皮质多巴胺分支功能低下，直接影响执行功能、行为计划和目标导向行为（来源：a dynamic developmental theory of attention-deficit/hyperactivity disorder）。LLM 则没有稳定的内在目标保持机制，它在 token 空间中是「生成即存在」，需要外部反馈、奖励信号或人类检查点来把生成锚定到真实目标。两者都是**高产但缺调度层的生成核心**。

## ADHD 侧：ChatGPT 到底补上了哪一层？

ChatGPT 对 ADHD 的帮助，不是因为它比人聪明，而是因为它提供了四种外部执行功能：任务分解、日程外包、知识召回和虚拟在场。

最常被低估的是**身体在场效应（Body Doubling）**。ADHD 患者在有他人在场时工作，专注度和执行力会显著提升，因为社会存在感提供了隐性问责，并激活与多巴胺相关的动机通路（来源：ADHD Task Managers That Work: Top AI Tools 2025）。AI 助手可以作为身体加倍伙伴，与用户并肩工作（来源：Using an AI Assistant to Manage ADHD: A Practical Guide）；Focusmate 则用算法匹配虚拟身体加倍伙伴，把寻找伙伴的门槛降到几乎为零（来源：The Best AI-Powered ADHD Productivity Tools in 2026）。一项资料甚至提到，定期问责检查可把目标达成可能性从 25% 提升至 95%（来源：ADHD Task Managers That Work: Top AI Tools 2025）。

另一层是**把模糊意图切成可执行步骤**。Goblin Tools 的 Magic ToDo 能把「清理厨房」这种让 ADHD 大脑瞬间冻结的任务，分解成具体、可执行的子任务，还能调节「辣度」控制粒度（来源：12 AI Personal Assistants Built for ADHD Brains (2026 Rankings)）。这种分解直接降低了任务启动门槛，绕过了多巴胺不足导致的动力缺乏（来源：Harnessing Artificial Intelligence to Live Better with ADHD - CHADD）。

再到更复杂的调度层，**Motion** 这类工具自动根据任务、会议和截止日期创建每日计划，并在日程被打乱时重新安排，消除「下一步该做什么」的决策负担（来源：11 Best ADHD Productivity Apps for Fluctuating Energy）。**Saner.AI** 则通过本地记忆和知识召回，减少 ADHD 用户因搜索循环和标签切换而产生的认知负荷（来源：Best AI Tools for ADHD Productivity in 2026）。

这些工具的本质是同一个：把大脑内部不稳定的执行调度层，外化成一个持续在线的、可交互的「副驾驶」。

## LLM/Agent 侧：为什么高风险 agent 必须 human-in-the-loop

把视角换到 LLM/agent 工程，问题几乎镜像。LLM 在开放生成环境中表现出色，但它缺乏稳定的长期目标保持、事实核查和自我纠错能力。这正是为什么高风险场景——医疗、法律、金融、自动驾驶、代码部署——必须引入 human-in-the-loop（HITL）监督。HITL 不是效率的拖累，而是把模型的生成能力与人类的判断、价值观和领域知识桥接起来的**约束与问责机制**。

这里的关键同构在于：
- ADHD 大脑的「身体在场」对应 LLM 的「人类监督」——都是外部存在提供的隐性问责；
- ADHD 工具的「任务分解」对应 agent 的「step-by-step 验证」——把大目标拆成可检查的中间步骤；
- ADHD 的「日程自动重排」对应 agent 的「re-planning loop」——在环境变化后重新锚定目标；
- ADHD 的「知识召回」对应 agent 的「RAG / 记忆模块」——外化工作记忆，避免幻觉。

两者都需要一个**外部的、可问责的、能重新 grounding 的回路**，否则生成就会脱离真实约束。

## 张一鸣的 harness 与 LLM 的约束系统同构

张一鸣的 harness 系统是这个同构的绝佳案例。他的 ADHD 特质表现为思维极度发散、什么新东西都要研究、永远在思考组织和产品（来源：张一鸣 wiki 资料）。但他没有靠意志力硬扛，而是建立了一套外部系统：OKR 管理系统、Context not Control、用系统和文化管公司，而不是个人集权。

这对应到 LLM/agent 的 harness 设计里：
- **延迟满足感** ↔ 定时 re-grounding：不让模型被短期奖励牵引，而是定期回到长期目标；
- **OKR 管理系统** ↔ 外部目标约束层：把模糊战略拆成可测量的关键结果，像 agent 的 reward shaping；
- **Context not Control** ↔ 人在回路：不直接 micromanage，而是通过信息共享和检查机制让系统自我校准；
- **从用户需求出发，不做主观判断** ↔ RLHF / 人类反馈：把最终评价权交给外部真实世界，而不是模型内部偏好。

张一鸣的 harness 不是「更努力」，而是**让系统替自己管理注意力**。这正是 ChatGPT 对 ADHD 人群应该扮演的角色——不是更聪明的自己，而是替自己维持目标在场的外部执行层。

## 但有人用错了：脚手架 vs 拐杖的边界

同构的另一面是同样的风险。wiki 资料中明确存疑：AI 工具究竟是「外部执行功能层」还是「拐杖」？过度依赖可能导致能力退化，而长期风险尚缺乏实证研究（来源：全局矛盾与存疑）。

用错 ChatGPT 的典型方式有两种：
1. **把它当决策替代者**：「帮我想想我该做什么」，然后被动接受。这相当于把 human-in-the-loop 取消，让 agent 自动驾驶。ADHD 大脑需要外部问责，但问责的终点必须回到自己的判断；否则只是换了一种逃避启动的方式。
2. **无限分解但从不交付**：用 Goblin Tools 把任务切到无限细，用 Motion 把日程排到无限满，但缺乏一个「终止与交付」的外部检查点。LLM agent 也会陷入类似的循环：不断 re-plan，永远不 commit。

正确的用法是**脚手架式**：AI 提供结构，但用户保留最终决策权和交付责任。身体在场效应的意义不是替你工作，而是让你更难放弃；human-in-the-loop 的意义不是替模型思考，而是让模型更难偏离。

## 今天就能试的 4 个行动

1. **开播一个 AI 身体在场会话**：打开 ChatGPT，告诉它「接下来 50 分钟我要做 X，每 15 分钟提醒我一次进度，并在结束时让我汇报结果」。这利用了 AI 作为身体加倍伙伴的潜力（来源：Using an AI Assistant to Manage ADHD: A Practical Guide）。

2. **把一件你拖了三天的任务丢给 Goblin Tools**：用 Magic ToDo 的「清理厨房」模式，把任务切成可执行步骤，并且只选第一步立即做。重点是**启动**，不是完美。

3. **在 Motion 或任何日历工具里设置「外部检查点」**：给高风险任务或容易放弃的任务，安排一个真实的人类或 AI 问责时间，而不是只靠自己记得。定期问责能显著提升目标达成率（来源：ADHD Task Managers That Work: Top AI Tools 2025）。

4. **写一条「人在回路」规则**：对于任何重要决策，强制规定「ChatGPT 可以建议，但我不在 10 分钟内做最终决定」。这防止 AI 从脚手架滑成拐杖，也对应高风险 agent 的 HITL 设计原则。

## 局限与诚实

需要承认：ADHD 大脑与 LLM/agent 的同构目前主要是理论框架，缺乏神经科学或计算机科学的直接证据（来源：全局矛盾与存疑）。AI 工具对 ADHD 的长期效果也尚未被充分验证，个体差异很大，部分用户甚至觉得学习工具本身增加了认知负荷（来源：全局矛盾与存疑）。Motion、Goblin Tools、Saner.AI 等工具的用户反馈积极，但独立临床证据仍然有限（来源：全局矛盾与存疑）。

所以这不是一篇「AI 治愈 ADHD」的乐观宣言。它只是一个更克制的判断：**当我们把 ChatGPT 当作一个持续在场、可问责、可重新 grounding 的外部执行层时，它就像高风险 agent 的 human-in-the-loop 监督一样，补上了生成核心缺失的调度层。用对了，它是脚手架；用错了，它就是拐杖。** 区分这两者的，不是工具本身，而是你是否仍然握着最终交付权。
