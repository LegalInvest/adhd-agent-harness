# 为什么一个人管理多个 AI Agent，本身就是一项执行功能任务？

> AI 圈正在兜售一个诱人的图景:你当指挥官,手下十个 agent 并行干活——写代码的、调研的、剪视频的——你只需「管理」。对 ADHD 用户这个图景尤其诱人:终于不用自己执行了!但这里藏着一个几乎没人说破的反讽:**管理多个 agent 这件事本身——跟踪状态、切换上下文、验收输出、决定优先级——恰恰是一项高强度的执行功能任务。** 你以为外包了执行,实际上把自己提拔成了「执行功能岗」。

把「多 agent 管理」拆开看它到底消耗什么。①**状态跟踪**:五个 agent 各跑到哪一步了?哪个卡住在等你输入?——这是工作记忆负荷,还是并发版的;②**上下文切换**:从验收 A 的代码切到回答 B 的调研问题,每次切换都要重新装载任务背景——这正是 ADHD 最贵的操作,现在每小时发生十几次;③**验收判断**:每个 agent 的输出都要评估质量、决定返工还是通过——持续的小决策,决策疲劳的完美生产线;④**优先级仲裁**:资源(你的注意力)给谁?——恰恰是 ADHD 排序短板的主场。讽刺完成了闭环:**agent 替你做了执行,把执行功能的部分全部留给了你,而后者才是你的短板。** 很多 ADHD 用户的真实体验印证这一点:开三个 AI 对话并行干活的下午,比自己写一下午东西更累,产出还更散。

工程界其实早就为「一个协调者管多个 worker」的负荷问题攒了方案——因为 multi-agent 系统里那个 orchestrator(编排器)也会过载。他们的设计原则,恰好是 ADHD 用户的用法指南:

**一、限制并发数(orchestrator 的第一美德)。** 成熟的编排设计从不无限开 worker,因为协调成本超线性增长。你的版本:**同时运行的 agent/AI 任务≤2**——一个前台(你正在互动的),一个后台(跑长任务的)。「十个 agent 并行」的图景对你不是效率,是精心布置的过载现场。

**二、异步化,消灭「盯着等」。** worker 干活时 orchestrator 不轮询,做完了会回调。你的版本:后台 agent 跑着的时候,**彻底离开它**——别每两分钟切回去看跑完没有(每次「看一眼」都是一次上下文切换税)。设物理计时器,到点统一验收。

**三、标准化验收,压缩决策。** orchestrator 验收 worker 靠预定义的检查,不靠现场品味。你的版本:给每类任务写死验收单(代码:能跑+有测试;调研:有来源+有结论段)——**把「这个输出行不行」从开放式判断压成清单核对**,决策疲劳减半。

**四、状态外置,大脑零库存。** 哪个 agent 在干什么、等什么——写在一张看板上(纸也行),不存在脑子里。orchestrator 的状态表从来在内存里显式维护,不靠「应该记得」。

**五、诚实核算:这个任务值得开 agent 吗?** 委托有固定成本(写清需求+验收返工)。十分钟能自己做完的事,发给 agent 再验收要二十分钟——**orchestrator 不是身份,是一种有成本的架构选择。** 单线程自己做,很多时候仍是最优解,承认这一点不丢人。

## 边界

同构强度 B 级:多 agent 编排的协调开销是真实工程议题,ADHD 的切换成本与工作记忆负荷有扎实文献,但「管理 agent 的认知负荷」尚无针对 ADHD 人群的直接研究,五原则是架构迁移。声明:若并行工作的疲惫感严重且普遍(不限于 AI 场景),值得做正式评估;工具的用法优化不替代治疗,你的并发上限也不是道德指标——单线程不是缺陷,是一种合法的架构。

## 今天就能试的 3 件事

1. **数一数你现在并行的 AI 对话/任务**:超过 2 个,关到 2 个。
2. **给最常委托的一类任务写验收单**:三条以内,下次照单验收。
3. **做一次诚实核算**:今天有哪个任务,自己做其实比委托 agent 更快?下次直接做。

「人人都当 agent 指挥官」的未来也许会来,但指挥官这个岗位的岗位说明书上,第一行写的就是执行功能。**在给自己招聘十个下属之前,先给指挥官配好参谋部**——限并发、异步化、清单验收、状态外置。不然你只是把「一个人的混乱」升级成了「一个组织的混乱」。
