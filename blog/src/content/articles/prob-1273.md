# Perplexity 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Perplexity：它到底补上了哪一层执行功能？

## 问题：当“搜索”变成另一种拖延

你是否有过这样的经历：想查一个简单的事实——比如“Python 的 list.append 返回值是什么”——结果打开浏览器，先看到一篇有趣的科普，又点进一个视频，半小时后你发现自己正在看猫咪打架，而那个问题还躺在标签页里。

对 ADHD 大脑来说，搜索不是工具，是陷阱。它本应是精确的检索行为，却常常滑向无边的信息漫游。

与此同时，在 AI 工程的世界里，LLM 也面临类似的困境：当它被直接抛给用户做问答，没有工具、没有验证、没有外部记忆，它就会自信地编造答案——这就是“幻觉”。

**Perplexity 的诞生，正是为了同时解决这两个问题。** 它把搜索行为变成了一个带验证的 function call：你问一个问题，它调用搜索引擎，提取来源，给出带引用的答案。对 ADHD 来说，这意味着“搜索-阅读-验证”的闭环被压缩成一次操作；对 LLM 来说，这意味着生成核心被接上了外部事实库，幻觉被大幅抑制。

## 同构：ADHD 大脑与 LLM 共享同一个缺陷

ADHD 大脑与 LLM 在结构上同构：两者都是强大的生成核心，但缺乏可靠的内置执行调度层（来源：ADHD 大脑与 LLM 的同构）。ADHD 个体不擅长精确、重复、状态维护类任务——比如记住一个公式、坚持一个日程、维持一个搜索路径。LLM 同样不擅长精确计算、状态追踪和事实核查——它会忘记上下文，会编造来源，会在一段对话中前后矛盾。

解决方案也同构：**给生成核心套一个 harness（脚手架）。**

LLM 侧的 harness 工程，核心就是 function calling：把精确计算外包给工具（如计算器、数据库、搜索引擎），让 LLM 只负责生成意图和解析结果。ADHD 侧的 harness，则是用外部系统补偿执行功能：日程提醒、任务分解、笔记工具——本质上是把“记得做”和“怎么做”外包给环境。

**Perplexity 恰好站在这个交叉点上。** 它既是一个面向 ADHD 的搜索 harness，也是一个面向 LLM 的 function calling 工具。

## 证据：两种 harness 如何工作

### ADHD 侧：Perplexity 作为“搜索执行层”

ADHD 个体的工作记忆缺陷导致频繁的搜索循环和标签切换（来源：Saner.AI）。Goblin Tools 通过任务分解降低启动门槛（来源：Goblin Tools），Saner.AI 通过本地记忆减少搜索循环（来源：Saner.AI），Lex 通过单一指令触发多步骤任务（来源：Lex）。Perplexity 则直接解决了搜索本身的问题：它把“打开浏览器-输入关键词-扫视结果-点开链接-阅读-判断可信度”这个高认知负荷的流程，变成了“输入问题-得到带引用的答案”。

这相当于给 ADHD 大脑装了一个**外部验证循环**。ADHD 的冲动行为本质上是一种验证失败——内部生成核心直接输出反应，没有经过环境核对（来源：幻觉与验证循环）。Perplexity 强制要求引用，就是在模拟那个缺失的验证步骤。

### LLM 侧：Perplexity 作为“function calling 工具”

LLM 的幻觉源于其生成核心缺乏接地能力。Harness 工程通过工具接口、验证循环和记忆系统来补偿（来源：GitHub - ai-boost/awesome-harness-engineering）。Perplexity 本质上就是一个封装好的搜索工具：LLM 调用它，获得带来源的事实，然后基于这些事实生成答案。这避免了纯对话系统因幻觉和缺乏接地而失败（来源：AI Agent Systems...）。

## 人物案例：孔子与老子的 harness 系统

孔子一生与自己的冲动做斗争：他精力旺盛、周游列国14年坐不住，见南子急得对天发誓，骂宰予“朽木不可雕”。他的 harness 是“克己复礼”——用外在的“礼”作为行为边界，每日反省，“吾日三省吾身”。这本质上是一个**外部调度层**：把“我应该做什么”的决策权交给一套固定仪式，而不是交给不可靠的内部冲动。

老子的 harness 则是“无为”——不强行做事，把所有能量收回来聚焦在道上。他不著书不立说，不被名相束缚。这相当于一个**低功耗模式**：减少不必要的输出，只做最核心的生成。

对比 LLM harness：孔子的“日课”相当于定时 re-grounding（如每轮对话重置上下文），老子的“无为”相当于降低采样温度，减少随机输出。Perplexity 则同时实现了这两者：它通过搜索接地（类似孔子的“礼”），又通过约束回答范围（类似老子的“无为”）来减少幻觉。

## 但有人用错了

Perplexity 的流行让许多人误以为它可以直接替代思考。ADHD 用户可能用它来“快速搞定”本该自己消化的信息，比如直接让 Perplexity 总结一本书，而不是读原著。工程师可能用它来“快速验证”代码，而不理解底层逻辑。

这就是**拐杖与脚手架的边界**（来源：拐杖与脚手架）。脚手架是暂时性的，帮助你完成自己无法独立完成的任务，同时培养你的能力。拐杖则是永久替代，让你原有的功能进一步萎缩。

Perplexity 的正确用法是作为**验证层**，而不是**生成层**。你应该先自己思考、自己搜索、自己尝试，然后用 Perplexity 来核对、补充、纠错。如果反过来——让 Perplexity 替你思考——你就成了 LLM 的“人类在回路”，而不是 LLM 成为你的工具。

## 局限与争议

1. **依赖风险**：过度使用 Perplexity 可能削弱内在的搜索和验证能力（来源：拐杖与脚手架）。
2. **证据基础薄弱**：Perplexity 在 ADHD 人群中的效果缺乏严格临床验证（来源：矛盾与存疑）。
3. **注意力分散**：Perplexity 的“相关搜索”和“探索”功能可能引发新的注意力陷阱。
4. **同构性争议**：ADHD 大脑与 LLM 的同构只是一种理论类比，并非经过验证的科学事实（来源：矛盾与存疑）。

## 今天就能试的行动

1. **用 Perplexity 做“事实核对”**：当你写代码、写文章、做决策时，先用 Perplexity 搜索你已有的结论，看是否有矛盾来源。
2. **设置“先思考后搜索”规则**：在打开 Perplexity 前，先写下你自己的答案或思路，哪怕只有一句话。
3. **限制每次搜索的目标个数**：每次只查一个具体问题，查完就关掉，避免“顺便看看”的滑落。
4. **结合 Goblin Tools 做任务分解**：对于复杂任务，先用 Goblin Tools 分解，再用 Perplexity 查每个子步骤的具体信息。
