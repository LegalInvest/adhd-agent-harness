# Otter.ai 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Otter.ai：它到底补上了哪一层执行功能？

## 一个问题：你录了 100 场会，还是什么都没做

ADHD 读者都有过这种体验：会上听得热血沸腾，觉得自己“肯定记得住要点”，结果两小时后就只剩模糊情绪，连 deadline 都忘了。于是你装了 Otter.ai，它把每次会议实时转成文字，你安慰自己“这次不会再丢了”。可一周过去，Otter 里躺着 20 条未读 transcript，你并没有变得更有条理，只是多了一个新的“信息坟场”。

在 agent 工程那端，问题同构：你给了 LLM 一个 function calling，让它去调用计算器、数据库或 API，结果模型拿到返回值就继续“自信发挥”，既没有验证格式，也没有把结果写回状态。调了工具，却依旧幻觉和重复。

这就是我想说的：**Otter.ai 之于 ADHD，就像 function calling 工具调用之于 LLM——但有人用错了**。

## 高产核心与缺掉的调度层

ADHD 大脑和今天的 LLM 在结构上有一个共同特征：都是高产的生成核心，但缺乏稳定的执行调度层。ADHD 患者工作记忆不稳定、任务启动困难、注意力容易被新刺激截走；LLM 则本质上无状态、无内部调度，容易上下文膨胀和推理退化（来源：ADHD 的 AI 工具全景；上下文工程）。

因此，两边的解法结构也类似：都需要一个 **harness/脚手架**——外部的执行功能层来补记忆、拆任务、控上下文、做验证。给 ADHD 搭脚手架，与给 LLM 做 agent harness，是同一类工程（来源：ADHD 的 AI 工具全景）。Otter.ai 扮演的角色，就是其中一支“工具调用”：它把听觉信息精确、重复、持续地转录成文本，补上了 ADHD 最不擅长的“状态维护”环节（来源：ADHD 的 AI 工具全景）。

但注意：**工具≠harness**。工具只是把问题精确地抛给外部系统；harness 负责把外部结果接回主流程、验证、存储、再触发下一步。

## 古代的 harness：孔子与曹操

看两个真实人物案例会更清楚。

**孔子**的 harness 是“礼”与“日省”。他身高一米九、精力旺盛、冲动爱骂人，思维跳跃，一辈子和冲动斗争，到七十岁才“从心所欲不逾矩”（来源：人物案例）。他的系统不是压抑生成，而是靠外在仪式（礼）和每日反思（吾日三省吾身）重新接地。这对应到 agent 工程里，就是**定时 re-grounding 与自我批判/验证循环**——不让模型一路狂奔，而是在关键节点把上下文拉回事实。

**曹操**的 harness 则是**外部调度器**。他自己军事文学超专注，日常行政完全不行，于是把执行功能外包给荀彧、郭嘉、军法系统，用“唯才是举”把合适的人放到合适的工具位上（来源：人物案例）。这几乎就是 function calling 的古代版：核心只做战略生成，精确执行和状态维护交给外部工具。

ADHD 和 LLM 都不缺“曹操式的才气”，缺的是“荀彧式的执行器”。

## 为什么 Otter 会被用错：它只做了“转录”，没做“闭环”

ADHD 一侧的真实证据是：通用聊天机器人会增加“另一个收件箱、另一个 prompt、另一个丢线索的地方”，反而让上下文更碎（来源：上下文工程）。Otter 如果只录音不处理，也同样是另一个信息黑洞。

LLM 一侧的真实证据更直接：复杂 harness 不会盲目执行工具，而是会对输出格式、计算结果做验证；缺乏验证步骤，就会导致“更多幻觉和重复工作”（来源：幻觉与验证循环）。换句话说，function calling 不是让模型“看起来调了工具”就完事，而是要有一个**验证循环**把结果写回状态、再决定下一步。

所以 Otter 的正确用法不是“我录下来了”，而是：
- **捕获**：Otter 实时转录，补工作记忆；
- **整理**：用 Saner.AI 这类知识回忆工具把关键信息从搜索循环里拉出来，或按项目归档（来源：Saner.AI）；
- **拆解**：用 Goblin Tools 的 Magic ToDo 把“把会上的想法落地”拆成可执行步骤（来源：Goblin Tools）；
- **触发**：用 Lex 或类似“单一指令触发多步骤任务”的工具把下一步自动推进（来源：Lex）；
- **验证**：每天一次像孔子“日省”那样的 re-grounding，确认 capture 真的变成了 done。

## 核心观点：把工具放在 harness 里，否则它就是拐杖

我的判断是：**Otter.ai 是一个出色的 function calling 工具，但 ADHD 真正要补的是它背后的 harness 层**。

在 agent 工程中，function calling 只是把精确计算外包给工具；但一个可用的 agent 还需要记忆系统、规划工件、沙盒和验证循环。ADHD 工具也是如此。Otter 解决的是“不要丢失当下信息”，但如果你不把它接入记忆、计划和验证系统，它只是让失败从“我忘了”变成“我录了但再也找不回”。

这里必须诚实提到“脚手架 vs 拐杖”的边界：工具本应是脚手架，帮助你最终建立更稳定的执行功能；但如果它变成永久拐杖，让你连“听会时自己记一句话”的主动性都放弃，就会削弱内在执行功能（来源：ADHD 的 AI 工具全景；矛盾与存疑）。

## 局限与争议

这种同构视角本身是有争议的。多个 wiki 页面把 ADHD 大脑与 LLM 称为“结构上同构”，但这仍是一种理论类比，缺乏严格实证（来源：矛盾与存疑）。此外，多数 ADHD 工具（包括 Otter、Lex 等）的临床证据薄弱，效果多基于用户报告，而非独立临床试验（来源：矛盾与存疑）。我们不应把“结构同构”当成科学定论，而应把它当作一个工程隐喻：既然两边都在补同一层能力，就可以互相借鉴设计。

## 今天就能试的 4 个行动

1. **下一次开会，用 Otter 录完后 5 分钟内提取 3 条行动项并写入任务管理器**。不要保存整条 transcript，只保留可执行的下一步。这就是“function 调用后必须解析返回值”。

2. **选一件你拖了三天的模糊任务，丢进 Goblin Tools 的 Magic ToDo**，让 AI 把它拆到 5 步以内，并给每步一个时间估计。体会“启动门槛”被降低的体感（来源：Goblin Tools）。

3. **如果你是 agent 工程师，给模型加一个确定性工具后，必须同时加一个验证断言**。例如调用计算器后，检查数值范围；调用数据库后，检查返回字段是否完整。不要让模型用“我认为对了”继续跑（来源：幻觉与验证循环）。

4. **设置一个每日 3 分钟的“再接地”仪式**：打开今天的 capture 列表，问“哪些东西还停留在录音/笔记里，没有变成行动或归档？”这就是孔子的“吾日三省吾身”——给生成核心做上下文清理。

---

Otter.ai 不是 ADHD 的终点，它只是 function calling 这一环。真正让 ADHD 大脑和 LLM 都好用的，是那一套能把“生成—捕获—验证—行动”串起来的 harness。工具用对了，才配得上你那台高产的生成核心。
