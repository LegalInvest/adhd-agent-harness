# RescueTime 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 RescueTime：它到底补上了哪一层执行功能？

你打开 RescueTime，看到“今日专注时间：2小时14分钟”，然后呢？你关掉面板，继续刷 Twitter。这场景太熟悉了——工具给了你数据，却没给你方向。

RescueTime 之于 ADHD，就像重锚定 system prompt 之于 LLM——但有人用错了。他们把它当成绩单，而不是方向盘。

## 同构：ADHD 大脑与 LLM 共享的“目标漂移”困境

ADHD 大脑与 LLM 在结构上同构：两者都是高产的生成核心，但缺乏可靠的执行调度层（来源：AI 与 ADHD 的专注力）。LLM 在长程任务中会因上下文溢出而目标漂移，需要重锚定 system prompt 来拉回轨道；ADHD 大脑则在分心走神或超聚焦跑偏中丢失目标——超聚焦让数小时在不知不觉中消失，与时间盲密切相关（来源：超聚焦）。

RescueTime 的自动时间追踪，本质上就是给 ADHD 大脑装了一个外部记忆，记录“你实际做了什么”，弥补工作记忆缺陷和时间盲（来源：RescueTime）。但问题在于：记录不等于重锚定。LLM 的 system prompt 不仅记录上下文，还主动约束行为方向；而 RescueTime 默认只提供事后报告，缺乏实时干预。

## 重锚定 vs 事后复盘：脚手架与拐杖的边界

这里的关键分歧在于：工具是作为可撤除的脚手架，还是永久的拐杖？ADHD 的专注力工具若只提供数据而不引导行动，就可能沦为拐杖——用户依赖它获取信息，却无法内化执行功能（来源：矛盾与存疑）。

真正的脚手架应该像 Focusmate 那样，通过实时同伴问责提供外部结构，弥补任务启动困难（来源：Focusmate）；或者像 Goblin Tools 那样，将复杂任务分解为小步骤，降低启动门槛（来源：任务启动）。这些工具在“当下”干预，而非事后总结。

RescueTime 的 AI 核心在于自动分类和模式识别，但它缺少一个“重锚定触发器”。例如，当检测到用户从编辑器切换到社交媒体超过 5 分钟，系统应该像 LLM 的 system prompt 一样弹窗：“你正在偏离主要目标，需要回到任务吗？”——而不是等到晚上发一份报告。

## 证据：工具有效，但缺临床验证

现有证据主要来自工具机制与 ADHD 理论的对齐，而非独立临床研究（来源：AI 与 ADHD 的专注力）。RescueTime 被推荐用于提升 ADHD 群体的生产力（来源：RescueTime），但缺乏针对 ADHD 的随机对照试验。同样，Brain.fm 的神经锁相技术理论上可增强专注，但缺乏独立验证（来源：Brain.fm）。

Focusmate 基于身体在场效应的广泛验证，AI 匹配提升了可用性（来源：Focusmate），但效果归因于身体在场而非 AI 本身。这提示我们：工具的“AI 成分”可能只是锦上添花，核心机制仍是人类互动。

## 从同构到实操：给两类读者的行动建议

**给 ADHD 读者**：
1. 将 RescueTime 的“实时警报”功能打开（如果版本支持），或搭配第三方工具（如 Focusmate 的 accountability session）实现实时干预。
2. 每天设定一个“重锚定时刻”：用 RescueTime 的每日摘要，但只关注一个指标——“我最重要的任务完成了多少？”，而不是总专注时长。
3. 尝试“任务分解+时间盒”：用 Goblin Tools 拆解任务，再用 RescueTime 的时间盒功能（如果支持）强制切换。

**给 Agentic Harness 工程师**：
1. 在 agent 的 system prompt 中加入“重锚定指令”：每完成一个子任务，要求 agent 输出当前目标与原始目标的偏差。
2. 设计“上下文溢出检测”：当 agent 的对话历史超过一定 token 数时，自动摘要并重写 system prompt。
3. 借鉴 Focusmate 的配对机制：为 agent 设置一个“同伴 agent”定期检查目标一致性。

## 结论：工具是镜子，不是地图

RescueTime 就像一面镜子，照出你的时间去了哪里。但镜子不会告诉你该往哪走——那是你自己的事。ADHD 大脑需要的是实时重锚定的系统 prompt，而不是事后报告。下次打开 RescueTime 时，问自己：我是在用它导航，还是仅仅在看后视镜？
