# RescueTime 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 RescueTime：它到底补上了哪一层执行功能？

你打开电脑，准备写一份方案。十分钟后，你发现自己正在刷手机看猫视频。你骂了自己一句，关掉手机，回到电脑前——然后点开了另一个无关的网页。这不是意志力问题，这是你的大脑在「上下文漂移」：你失去了对「当前任务」的锚定。

ADHD 大脑与 LLM agent 共享一个底层困境：两者都是「高产但缺执行调度层的生成核心」。LLM 在长程任务中会因上下文膨胀而推理退化（来源：上下文工程），ADHD 大脑则因工作记忆弱、时间盲而频繁丢失目标（来源：时间盲）。两者的解法结构同构：都需要一个外部机制来「重锚定」——对 LLM 是定时的 system prompt 刷新，对 ADHD 大脑则是外部工具提供的注意力锚点。

RescueTime 常被当作「时间监控工具」推荐给 ADHD 人群，但它真正的价值在于充当「重锚定 system prompt」。它后台记录你在每个应用上的时间，生成可视化报告，让你看到「时间到底去了哪里」（来源：Revolutionizing ADHD Management with AI Assistants）。这个功能看似简单，却对应着 LLM agent 工程中一个关键设计：**上下文回溯**。当 agent 在长任务中漂移时，工程师会注入一段「你当前在做什么」的 prompt 来重置上下文；RescueTime 的报告就是 ADHD 大脑的上下文回溯——它告诉你过去一小时你实际在做什么，而不是你以为自己在做什么。

但多数人用错了。他们只把 RescueTime 当作「自我审判工具」：每天看自己浪费了多少时间，然后陷入自责。这就像给 LLM agent 只加一个「你错了」的惩罚信号，却不给它「现在该做什么」的重定向。正确的用法是：**把 RescueTime 当作外部调度层的反馈信号**，而不是道德评判。曾国藩的「日课十二条」就是这种用法的古典版本：他每天写日记反省，不是为了一味自责，而是通过记录来「重锚定」第二天的行为。他的 harness 系统——「黎明即起、读书不二、谨言」——本质上是一套定时触发的 system prompt，用最笨最稳的方式抵消自己的冲动（来源：人物案例）。同样，左宗棠的「身无半亩心忧天下」也是一种外部锚点，用宏大的目标来对抗日常的注意力漂移。

RescueTime 的 AI 介入在于自动分类和模式识别（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout），但这恰恰是它的双刃剑。自动分类可能不精准，需要用户手动调整，这对执行功能弱的用户仍是挑战（来源：RescueTime 局限与争议）。更关键的是，过度关注时间数据可能引发焦虑，让用户陷入「监控→自责→更焦虑→更难专注」的恶性循环。这是「拐杖 vs 脚手架」的边界：如果 RescueTime 让你产生了依赖——没有它你就完全失去时间感——那它就是拐杖；如果它帮助你逐渐内化时间觉察能力，那它就是脚手架。目前缺乏长期研究来界定这个边界（来源：矛盾与存疑）。

对比其他工具，Focusmate 通过实时同伴问责提供外部结构（来源：Focusmate），Brain.fm 通过神经锁相音乐影响大脑状态（来源：Brain.fm），它们分别对应 LLM agent 的「外部监督」和「环境调节」。但 RescueTime 的独特之处在于它提供了**可回溯的上下文历史**——这是 LLM agent 中「记忆模块」的同构物。一个没有记忆的 agent 会不断重复错误；一个没有时间记录的 ADHD 大脑也会反复掉进同一个坑。

那么，如何正确使用 RescueTime？以下四条行动建议基于同构视角：
1. **设置「重锚定提醒」**：在 RescueTime 中开启每日/每周报告推送，但不要只看「浪费了多少时间」，而是问自己：「过去这段时间，我的上下文是什么？我实际在做什么？」这相当于 LLM agent 的「上下文日志」。
2. **用数据找模式，而不是找罪证**：关注「我通常在什么时间、什么环境下漂移」，而不是「我今天又浪费了 2 小时」。例如，你发现下午 3 点后注意力容易涣散，那就提前安排一个「重锚定动作」——比如用 Focusmate 预约一个时段，或用 Routinery 设置一个过渡步骤（来源：上下文工程）。
3. **结合「外部调度器」**：RescueTime 是反馈层，不是调度层。你需要一个主动的调度工具（如 Tiimo 的视觉时间表，来源：时间盲）来规划「接下来做什么」。RescueTime 告诉你过去发生了什么，Tiimo 告诉你未来该做什么——两者构成完整的「上下文管理」系统。
4. **警惕「监控疲劳」**：如果 RescueTime 让你焦虑，就降低使用频率。记住，它的目的是帮你重锚定，而不是让你自我惩罚。你可以像曾国藩写日记那样，每周一次回顾，而不是每天盯着实时数据。

最后，诚实面对局限：ADHD 大脑与 LLM 的同构是隐喻，不是严格等价（来源：矛盾与存疑）。LLM 的「注意力机制」基于数学计算，人类的注意力涉及多巴胺、情绪、身体状态等复杂因素。RescueTime 的证据目前多为用户报告，缺乏大规模临床研究（来源：RescueTime 证据）。但作为工程思维的类比，它提供了一个有用的框架：**工具的价值不在于监控，而在于重锚定**。

如果你是一名 ADHD 患者，下次打开 RescueTime 报告时，请把它看作你的「system prompt 日志」——它不是审判，而是帮你找回上下文的地图。如果你是一名 agent 工程师，不妨想想：你的 agent 是否需要更频繁的上下文回溯？也许答案就藏在 ADHD 的 harness 里。
