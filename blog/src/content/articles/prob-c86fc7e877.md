# LLM agent 的 plan collapse 和 looping，能不能把它们的 recovery 策略直接迁移成 ADHD 工具的设计模式？

> 提问来自一位读了「目标漂移」系列的 agent 工程师,问题问得很工程:「LLM agents 常出现 plan collapse(中间步骤偏离最初目标)和 looping(反复调用同一个工具不前进),这和 ADHD 的『目标漂移』『反复打开同一个标签页』相似得让人起鸡皮疙瘩。我们这行已经攒了一套 recovery 策略:重规划(replan)、反思(reflection)、降级(graceful degradation)。**这三样能不能直接迁移成 ADHD 辅助工具的设计模式?**」答案是:能迁移,但「直接」两个字要收回——三个策略各有一处必须改造的地方,而改造点恰恰暴露了人和 agent 最深的一处不同:**agent 的 recovery 可以随时打断执行,人的 recovery 必须先过情绪这一关。**

收敛:本文只回答——**三个 recovery 策略各自的迁移方案(逐个给出工具设计模式)?每个迁移的「必须改造点」在哪?整套迁移的总边界?**

## 穿透:策略可以照抄,触发时机全部要重写

**策略一:Replan(重规划)→ 设计模式:「检查点重排,而非全量重排」。**agent 版:检测到偏航→丢弃当前计划→按当前状态重新生成计划。直接照搬到人身上的样子是「发现下午全废了→重新规划今天」——而这恰恰是 ADHD 用户最熟悉的失败:全量重规划本身是高点火成本任务,且伴随「前面都白费了」的挫败,多数人卡死在重规划这一步(规划的废墟上再盖规划)。**改造点:重规划的粒度和语气。**工具设计模式:检测到偏移时,不重排全天,只问一个问题——「现在到下一个整点,做哪一件?」(唯一动作的复用:把 replan 压缩成单步决策);且界面语气必须是无追溯的(不展示「你原计划 vs 实际」的对比表——那对 agent 是有用的 diff,对人是羞耻的呈堂证供)。**策略二:Reflection(反思)→ 设计模式:「异步反思,绝不实时」。**agent 版:执行中插入自省步骤(「我刚才那步符合目标吗?」),实时纠偏。直接迁移的样子是工具频繁弹窗「你现在做的事符合今日目标吗?」——实测这类设计的下场:要么被用户关掉(打断是有税的,尤其对好不容易进入状态的 ADHD 用户——**实时反思可能打断的恰恰是它该保护的专注**),要么触发焦虑性自查循环。**改造点:反思的时机从「执行中」搬到「执行间隙」。**工具设计模式:反思只发生在天然断点(任务完成时、日程块结束时、回来的时刻——断点本来就要过渡,反思在这里零打断成本);格式是两问:「刚才那段去哪了?」「下一段想去哪?」——过去时+未来时,跳过现在进行时(现在进行时的自查才是焦虑源)。**策略三:Graceful degradation(降级)→ 设计模式:「崩溃日模式」。**agent 版:高级策略失败→自动切换到更简单可靠的备用方案(工具坏了就用缓存,复杂计划崩了就跑保底流程)。这是三者中迁移得最漂亮的一个,因为 ADHD 工具几乎全体缺失这个层:市面工具默认用户每天状态恒定,状态崩的日子,工具的满屏任务和红色逾期反而成为压垮者。**改造点:降级要用户主动一键触发,而非系统判定**(被系统判定「你今天不行」是羞辱,自己宣布「今天切保底模式」是掌控)。工具设计模式:一个显式的「低电量模式」开关——打开后,任务列表折叠为唯一动作、通知降为静默、逾期标红全部隐藏、界面只剩一句「今天只做这个,做完就是赢」——**graceful 这个词的全部人性化含义:降级不是惩罚性的缩减,是体面的庇护所。**

总边界,一条贯穿性差异:agent 的 recovery 由外部检测触发(monitor 看 log 就行),人的偏航检测本身就是被损害的功能(元认知的盲区:漂移时你不知道自己在漂)——**所以迁移过来的整套系统,检测环节不能指望用户自察,要靠结构性锚点**(整点轻提示、日程块边界、以及最可靠的:另一个人——翻译层与 body double 依然是最强的 monitor)。三个策略解决「检测到之后怎么办」,检测本身,仍然要靠 harness 的老三样。

## 验证

可测(工具团队):A/B 三个改造点——全量重排 vs 单步重排的完成率、实时反思 vs 断点反思的留存、系统判定降级 vs 用户触发降级的使用率与情绪反馈。可证伪:若某用户群在实时反思下表现更好(存在:焦虑低、元认知强的亚组),说明改造点的参数应做人群自适应——改造方向不变,幅度可调。

## 决策

做什么(设计者):三个模式照单实现,改造点一个不省;检测环节配结构锚点。做什么(用户):没有这类工具前手动跑同款——偏航后只问「到整点做哪件」、反思只在断点做、给自己预设崩溃日的保底清单。

不做什么:不要做「计划 vs 实际」对比界面(diff 是给机器看的);不要让系统替用户宣布「你不行了」。

先做什么(用户):今天先写好你的「低电量模式」保底清单——三行以内,贴在看得见的地方。

## 边界

Recovery 策略为 agent 工程实践(快速演进);迁移方案为设计提案,改造点基于 ADHD 的打断成本、羞耻敏感与元认知文献(GRADE 中,本文按 B 级功能同构处理)。工具设计不替代治疗;若「漂移」的频率与代价持续失控,请专业评估。

## 今天就能试的 3 件事

1. 写低电量模式保底清单。
2. 今天偏航时,只问一次「到下一个整点,做哪一件」。
3. 睡前用两问格式做一次断点反思:今天那些时间去哪了?明天第一段想去哪?
