# ADHD 职场人视角：为什么「治好 ADHD 的任务启动困难、不会拆解」和「让 LLM 不跑飞」其实是同一道工程题？

> agent 用 planner-executor 循环分解长任务——同一套 harness 思路，两边都成立。

## 你和我，共享同一个困境

想象你是一个 ADHD 职场人：早上打开电脑，脑子里有十个想法在同时尖叫，但手就是抬不起来，不知道该先点开哪个文件。任务拆解？你连第一步都看不清。另一边，你是一个做 agent 的工程师：你刚让 LLM 写一个长脚本，它前几行写得漂亮，但到第 50 行就开始忘记上下文，开始编造不存在的 API，整个任务跑飞。

这两件事看起来完全不相干，但本质是同一道工程题。ADHD 大脑与 LLM 在结构上同构：两者都是高产的生成核心，但缺乏可靠的执行调度层（来源：生成核心与调度层）。你的大脑像一台没有操作系统的超级计算机，而 LLM 则像一台没有调度器的 GPU 集群——算力惊人，但没法把任务一步步落地。

## 同构的脊柱：规划循环与任务分解

ADHD 的核心困境并非智力不足，而是执行功能失效——被描述为“大脑的驾驶座”常常感觉“方向盘后没有人”（来源：生成核心与调度层）。你无法自动把“写季度报告”拆成“打开文档→列出大纲→写第一段→……”，因为工作记忆瓶颈让你在拆到第三步时已经忘了第一步。

LLM 也一样。单独使用时，它缺乏内置的规划与执行编排。现代 AI 编码代理通过“复合 AI 系统架构”来弥补，包括“分离规划与执行的双代理架构、惰性工具发现、自适应上下文压缩”（来源：生成核心与调度层）。这些技术本质上是在 LLM 外部搭建调度层，防止“上下文膨胀和推理退化”——跟你用便签和番茄钟防止自己跑偏，是同一套逻辑。

## 工具即 harness：外部调度层的两条路径

ADHD 侧的工具正在充当这个调度层。Motion 能自动根据任务、会议和截止日期创建并动态调整每日计划，在日程被打乱时自动重新安排（来源：Motion）。它消除的是“下一步该做什么”的决策负担——这正是 ADHD 任务启动困难的根源。Reclaim.ai 则更强调“保护”时间，通过创建深度工作块自动防御会议侵占，减少决策疲劳（来源：Reclaim.ai）。Tiimo 和 Structured 通过视觉化时间线将时间转化为可见元素，直接应对时间盲（来源：Tiimo）。这些工具本质上都是“外部执行功能脚手架”，外化思维，减轻认知负荷（来源：拐杖与脚手架）。

LLM/agent 侧的 harness 工程同样在做这件事。Harness 包括上下文传递、工具接口、规划工件、验证循环、记忆系统和沙箱，决定代理在真实任务中的成败（来源：拐杖与脚手架）。脚手架在首次提示前组装代理（系统提示、工具模式、子代理注册），而 harness 在运行时编排工具调度、上下文管理和安全执行（来源：拐杖与脚手架）。这与 ADHD 工具的工作方式高度同构：都是通过外部系统提供记忆、调度和约束，补偿核心的缺陷。

## 关键判断：脚手架不是拐杖

但这里有一个必须指出的边界。真正的脚手架应帮助使用者“建造”，但使用者仍需自己“攀登”（来源：拐杖与脚手架）。如果工具替代了治疗或自我管理，可能造成依赖（来源：拐杖与脚手架）。同样，LLM 的 harness 如果设计得过于死板，也会让模型失去灵活性。

然而，当前多数 ADHD 工具（如 Motion、Reclaim.ai）设计为长期使用，未提及撤除机制（来源：矛盾与存疑）。这引发了一个争议：这些工具到底是可逐步撤除的脚手架，还是永久拐杖？同样，在 agent 工程中，过度依赖固定的规划模板也可能限制 LLM 的创造力。我的观点是：**好的 harness 必须包含“撤除路径”——无论是 ADHD 用户逐渐减少对 Motion 的依赖，还是 agent 系统在任务熟练后减少 planner 的介入频率。** 没有撤除路径的脚手架，终将成为新的依赖。

## 诚实面对局限

ADHD 大脑与 LLM 同构命题目前仍是理论框架，证据主要来自概念类比和工具案例，缺乏大规模实证（来源：矛盾与存疑）。多数工具（如 Tiimo）的 AI 功能细节不明确，缺乏严谨的随机对照试验支持（来源：主题综述）。多巴胺干预的效果也存在争议（来源：矛盾与存疑）。但即便如此，同构视角提供了一种有力的思维模型，让 ADHD 用户理解自己的困境并非“懒”，而是系统架构问题；也让工程师明白，他们解决的不是冷冰冰的 bug，而是与人类共享的认知瓶颈。

## 今天就能试的行动

1. **ADHD 侧**：用 Motion 或 Reclaim.ai 的免费试用版，把明天的一个模糊任务（比如“准备周报”）输入进去，观察 AI 如何自动拆解和排程。注意自己是否感觉“启动门槛”降低了。
2. **工程师侧**：在你现有的 LLM 代理中，手动添加一个显式的“规划-执行”循环：先让模型输出一个任务分解步骤（planner），再逐步骤执行（executor），并在每一步后检查上下文是否超限。对比不加循环时的跑飞率。
3. **通用**：无论你是哪一类人，尝试用“脚手架思维”审视你依赖的工具——它是否让你变得更独立？如果明天工具消失，你还能完成同样的任务吗？如果答案是否定的，考虑逐步减少依赖，同时锻炼内在的执行功能。
4. **交叉启发**：ADHD 用户可以在 agent 的 planner 输出中寻找灵感——LLM 的分解方式可能比你自己更“认知友好”。工程师则可以观察 ADHD 工具的设计（如 Tiimo 的视觉化时间线），思考如何改进 agent 的上下文可视化。
