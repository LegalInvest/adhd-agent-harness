# Lex 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Lex：它到底补上了哪一层执行功能？

## 问题：为什么 Lex 让我更焦虑了？

你是一个 ADHD 患者。你打开 Lex，输入“写一篇季度报告”，期待它像魔法一样帮你搞定一切。结果它列出了一堆步骤：收集数据、分析趋势、撰写初稿……你盯着这些步骤，反而更不想动了。

你是一个 Agent 工程师。你给 LLM 一个 function calling 工具，说“计算这个项目的 ROI”。LLM 调用了计算器，返回了精确数字。但下一次，它却在没有工具的情况下自己瞎算，结果一塌糊涂。

问题出在哪？

答案：Lex 之于 ADHD，就像 function calling 之于 LLM——但很多人用错了。Lex 的真正价值不是替你生成内容，而是作为外部执行调度层，把你不擅长的精确、重复、状态维护类任务外包出去。同样，function calling 的价值不是让 LLM 自己算，而是让它把精确计算交给工具。

## 同构：ADHD 大脑与 LLM 是同一类“生成核心”

ADHD 大脑与 LLM 在结构上惊人相似：两者都是“高产但缺执行调度层的生成核心”（来源：ADHD 的 AI 工具全景）。

- **ADHD 侧**：你有丰富的想法和创造力，但工作记忆容量有限、任务启动困难、时间感知模糊（来源：ADHD 的 AI 工具全景）。你需要一个外部调度层来分解任务、管理上下文、提供即时反馈。
- **LLM/Agent 侧**：LLM 能生成流畅文本，但上下文控制弱、容易跑偏、无法精确计算。你需要 function calling 把数学、搜索、数据库操作等精确任务外包给专用工具。

这种同构意味着：**两边的解法结构相同——都需要一个“harness（脚手架）”来补上执行调度层**。对 ADHD 来说，这个脚手架是 AI 工具（如 Lex）；对 LLM 来说，这个脚手架是 function calling harness。

## Lex 的正确用法：作为“执行调度层”，而非“万能生成器”

Lex 的核心功能是“通过单一指令触发复杂、多步骤的任务序列”（来源：Lex）。这听起来很美好，但问题在于：**ADHD 用户往往误以为 Lex 应该替他们完成所有思考**，而工程师也误以为 function calling 能让 LLM 自己搞定一切。

实际上，Lex 的正确用法是：**用它来分解你明确知道怎么做、但执行起来困难的任务**。比如，你很清楚“写周报”需要收集数据、整理要点、撰写段落，但你总是卡在启动上。Lex 帮你把这些步骤拆开，并逐一执行——它做的是调度，不是创造。

同样，function calling 的正确用法是：**把 LLM 不擅长的精确操作（如数学计算、数据库查询）交给专用工具**，LLM 只负责理解和生成。

证据支持：
- **ADHD 侧**：Goblin Tools 的“魔法待办”功能将大任务拆为小步骤，用户报告称启动焦虑降低（来源：Goblin Tools）。Lex 的单一指令触发多步骤，减少了决策疲劳（来源：Lex）。
- **LLM/Agent 侧**：在 Agent 工程中，function calling 显著提升了 LLM 在数学、检索等任务上的准确率，因为工具提供了精确执行环境。

## 脚手架 vs 拐杖：边界在哪里？

这里有一个关键争议：AI 工具是脚手架还是拐杖？

- **脚手架**：帮助你完成自己难以独立完成的任务，同时让你学到方法。比如，Lex 分解任务后，你逐渐学会自己分解。
- **拐杖**：让你完全依赖工具，丧失内在能力。比如，你永远让 Lex 替你写周报，自己从不动手。

wiki 资料明确指出：“AI 工具应作为脚手架促进能力发展，而非永久拐杖”（来源：ADHD 的 AI 工具全景）。但矛盾在于：“过度依赖 AI 工具可能削弱内在能力”（来源：矛盾与存疑）。例如，过度使用 Otter.ai 可能削弱主动记笔记的能力（来源：ADHD 的 AI 工具全景）。

我的判断是：**Lex 的正确边界在于——它应该处理执行层面的重复劳动，而非认知层面的创造性思考**。如果你用 Lex 来写一封你完全知道怎么写的邮件，那是脚手架；如果你用它来构思你从未想过的方案，那是拐杖。

## 局限与诚实

必须承认，现有证据主要来自用户报告和概念类比，缺乏大规模对照实验（来源：矛盾与存疑）。Lex 的效果因人而异：对某些人，它可能是救命稻草；对另一些人，它可能增加认知负荷（来源：矛盾与存疑）。此外，Lex 的 AI 能否准确理解复杂意图仍存疑（来源：Lex）。

## 今天就能试的行动

1. **给 Lex 一个“执行型”任务**：不是“写一篇论文”，而是“把‘整理桌面’分解成 5 个步骤并逐一执行”。观察它是否降低了你的启动阻力。
2. **对比“用 Lex 分解”和“自己分解”**：选一个你熟悉的重复任务（如“发会议纪要”），先用 Lex 做一次，再自己做一次。感受差异，判断 Lex 是否真的减少了认知负荷。
3. **在 Agent 工程中，只给 LLM 它不擅长的工具**：比如，给一个计算器工具，但不要给“写邮件”工具——让 LLM 自己写邮件，工具只负责精确计算。
4. **设置“脚手架使用时限”**：例如，用 Lex 分解任务一个月后，尝试自己分解一次，看是否学会了方法。如果完全依赖，停用一周。

## 结语

Lex 不是魔法。它是你大脑的 function calling 工具——帮你把不擅长的执行任务外包出去，但思考和创造仍然属于你。用对，它是脚手架；用错，它是拐杖。选择在你。
