# RescueTime 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 RescueTime：它到底补上了哪一层执行功能？

你有没有这样的体验：明明装了 RescueTime，每天看着时间线上一块块「生产力时间」和「分心时间」，却还是在下一次走神时毫无察觉？或者，你是个正在搭 agent harness 的工程师，给 LLM 配了 function calling，结果它还是反复调用同一个 API 输出垃圾——因为缺少验证循环。

这两类困境共享同一个结构缺陷：**有了工具，但没有利用工具的反馈来校准行为**。

## 同构：ADHD 大脑与 LLM 都是「缺调度层的生成核心」

ADHD 大脑常被描述为“高产但缺乏可靠执行调度层的生成核心”（来源：ADHD 的 AI 工具全景）。它擅长发散、联想、创造，但在精确、重复、状态维护类任务上频频卡壳。LLM 同样如此：能生成流畅文本，但离开 harness 的上下文管理和验证循环，就会陷入幻觉和重复工作（来源：幻觉与验证循环）。

两者的核心缺陷都是**执行调度层缺失**。ADHD 需要外部工具来补偿工作记忆、任务启动和时间盲；LLM 需要 agent harness 来提供上下文传递、工具接口和验证循环。RescueTime 和 function calling 正是各自阵营里的「精确计算外包工具」：RescueTime 自动记录时间使用（精确），function calling 让 LLM 调用外部 API 做计算（精确）。

但问题来了：**只外包精确计算，不建立反馈闭环，工具就变成了拐杖**。

## RescueTime 的正确用法：从记录到反馈的闭环

RescueTime 的常见用法是：装好，让它跑，周末看一眼报告，然后继续该干嘛干嘛。这就像给 LLM 配了计算器 API 却不检查它是否算对——输出可能精确，但行为没有改善。

ADHD 侧的真正需求是**实时反馈**。Goblin Tools 之所以有效，是因为它把“整理房间”这种大任务分解成“捡起衣服”“擦桌子”等小步骤，每个步骤都是即时可完成的（来源：Goblin Tools）。这种设计利用了「小步骤的即时反馈维持注意力」（来源：Goblin Tools）。同样，Lex 的“单一指令触发多步骤”降低了启动门槛（来源：Lex）。

RescueTime 的问题在于：它的反馈是滞后的、聚合的。ADHD 大脑需要的是**当下**的提醒——比如“你已经连续刷了 15 分钟短视频，要切换吗？”——而不是周末的饼图。

LLM/harness 一侧的对应物是**验证循环**。复杂的 harness 不会盲目执行工具，而是实现验证步骤，例如检查输出格式或对模型编写的代码运行测试用例（来源：幻觉与验证循环）。缺乏这些验证会导致“更多幻觉和重复工作”（来源：幻觉与验证循环）。

所以，RescueTime 之于 ADHD，就像 function calling 之于 LLM——都是精确计算外包。但 RescueTime 缺少的是**验证循环**：它没有在行为发生的瞬间给出可行动的反馈。

## 脚手架 vs 拐杖：边界在哪？

“AI 工具应作为脚手架促进能力发展，而非永久拐杖”（来源：ADHD 的 AI 工具全景）。脚手架的特点是：**使用过程中提供支持，使用后能力得到锻炼**。拐杖则是：用了就离不开，不用就瘫痪。

对于 RescueTime，脚手架用法是：设置实时提醒（比如每 30 分钟弹窗问“你在做什么？”），并配合自我反思。这样，用户逐渐学会感知时间流逝。拐杖用法是：只看报告，不改变行为。

对于 function calling，脚手架用法是：让 LLM 调用 API 后，再让另一个模块验证结果，并把验证逻辑逐步内化到 prompt 中。拐杖用法是：不管结果对错，直接输出。

## 矛盾与局限

必须诚实指出：目前 RescueTime 对 ADHD 的效果缺乏独立对照试验（来源：矛盾与存疑）。同样，function calling 的验证循环设计虽在工程中流行，但尚无标准评估框架。

另一个矛盾：个体差异巨大。有些 ADHD 用户发现 RescueTime 的实时提醒反而增加焦虑，就像有些 LLM 在过度验证下会陷入死循环。工具不是万能药，关键在于找到适合自己的反馈粒度。

## 今天就能试的行动

1. **给 RescueTime 加上实时提醒**：设置每 30 分钟弹窗，问“当前任务是什么？”。这模拟了 LLM harness 的验证循环。
2. **用 Goblin Tools 替代纯记录**：把一个大任务（比如“写报告”）丢进 Magic ToDo，得到步骤列表，然后每完成一步就标记。这比 RescueTime 的被动记录更主动。
3. **为你的 LLM agent 添加输出验证**：写一个简单的脚本，检查 API 返回的 JSON 格式，或对代码运行单元测试。这直接对应 ADHD 的“行为验证”。
4. **记录+反思**：每天花 5 分钟看 RescueTime 报告，并写一句话“今天哪个时间段最有效率？为什么？”——这相当于 LLM 的“自我批评提示”（来源：幻觉与验证循环）。

RescueTime 和 function calling 都是好工具，但只有配上反馈闭环，它们才从拐杖变成真正的脚手架。
