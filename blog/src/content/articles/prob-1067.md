# 用 Perplexity 给 ADHD 补上重锚定与目标漂移，是脚手架还是拐杖？

> Perplexity 接进 ADHD 工作流：哪些交给它、哪些必须自己留着。

## 问题：你的“生成核心”又失控了？

你打开浏览器，准备写一份报告。三分钟后，你发现自己正在搜索“为什么猫会踩奶”。五分钟后，你点开了一个关于量子物理的科普视频。半小时后，你焦虑地关掉了 17 个标签页——报告一个字没写。

这不是意志力问题。这是 ADHD 大脑的典型场景：一个高产的生成核心（联想、发散、好奇）缺乏可靠的执行调度层。你的注意力像没有舵的船，被每一个新刺激带偏。这种“目标漂移”在工程领域同样存在：一个长程 agent 在任务中途会逐渐偏离初始指令，产生无关输出。解决方案是**重锚定**——通过外部机制不断把系统拉回主线。

Perplexity 这类 AI 搜索工具被宣传为 ADHD 的救星。但问题是：它究竟是帮你锚定目标的脚手架，还是让你漂移更快的拐杖？

## 同构：ADHD 大脑与 LLM 共享的缺陷

ADHD 大脑与 LLM 在结构上同构：两者都是高产的生成核心，但缺乏可靠的执行调度层（来源：AI 与 ADHD 的专注力）。ADHD 的执行功能障碍包括工作记忆缺陷、时间盲和任务启动困难；LLM 的无状态性和上下文溢出则导致 agent 在长程任务中丢失目标。两边的解法结构一致：**给生成核心套一个外部 harness**。

对于 LLM agent，这个 harness 是编排层（scaffolding）——通过 system prompt 重锚定、记忆模块、任务分解来防止漂移。对于 ADHD 大脑，这个 harness 是外部执行功能工具——通过自动记录、定时提醒、同伴问责来补偿内在缺陷（来源：AI 与 ADHD 的专注力）。Perplexity 处于这个光谱的中间：它既可以是搜索工具（被动接收信息），也可以是重锚定工具（主动追问、对比、总结）。

## 重锚定的两种模式：脚手架 vs 拐杖

Perplexity 的核心功能是**搜索 + 综合回答**。对 ADHD 用户，它有两种使用模式：

**模式一：拐杖——被动信息消费。** 你输入一个问题，得到一段漂亮的总结。然后你被其中的一个链接吸引，点进去，开始无休止的漫游。Perplexity 的“相关搜索”和“来源链接”反而成为新的漂移诱因。这种模式下，工具没有提供重锚定机制，反而放大了分心。

**模式二：脚手架——主动重锚定。** 你带着一个明确的任务目标使用 Perplexity，比如“写报告需要三个数据来源”。你搜索、阅读、追问，每次操作都回到原始目标。你甚至可以用它来检查自己是否偏题：“我现在的思路是否还符合最初的问题？”这种主动的元认知监控，就是重锚定。

关键区别在于：**谁控制锚点？** 如果你让 Perplexity 的信息流牵着走，它是拐杖；如果你用 Perplexity 作为工具来反复校准自己的目标，它是脚手架。

## 证据：哪些工具真的做到了重锚定？

现有 ADHD 工具中，**Focusmate** 通过实时同伴问责提供外部锚定（来源：Focusmate）。其 AI 匹配优化了“身体在场”体验，但核心机制是社会压力而非 AI 本身。**RescueTime** 通过自动时间追踪帮助识别漂移模式（来源：RescueTime），但它只提供事后报告，缺乏实时干预。**Brain.fm** 的神经锁相音乐理论上可维持专注状态（来源：Brain.fm），但缺乏 ADHD 专项研究，且不提供目标锚定。

Perplexity 的独特之处在于：它允许用户**在信息获取过程中随时返回原始问题**。你可以把初始问题作为 system prompt，每次追问都检查是否相关。这与 LLM agent 工程中“在每轮交互前注入原始指令”的做法完全一致。可惜，Perplexity 并未原生提供这种“锚点显示”功能——你需要自己记住目标，或把问题写在便签上。

## 争议与局限：脚手架能撤除吗？

最大的争议是：长期使用 AI 工具是否会阻碍内在执行功能发展？（来源：矛盾与存疑）目前无长期研究证明脚手架可逐步撤除。多数工具（如 Goblin Tools、Saner.AI）设计为永久使用，未考虑撤除机制（来源：矛盾与存疑）。对于 Perplexity，同样存在风险：如果你依赖它来“替你想”，你的元认知能力可能退化。

另一个局限是：Perplexity 的效果高度依赖用户主动使用重锚定功能。对于执行功能严重受损的 ADHD 个体，连“主动使用”本身都是挑战。此外，Perplexity 的搜索回答有时会包含无关信息，反而增加认知负荷。

## 我的判断：Perplexity 是脚手架，但需要你自己搭

我的核心观点是：**Perplexity 作为重锚定工具的有效性，取决于你是否把它当作“对话中的锚点”而非“信息瀑布”。** 它与 LLM agent 的 scaffolding 同构：你需要一个外部的 system prompt（你的原始目标），并在每次交互后检查是否偏离。Perplexity 本身不提供这个 prompt，但它提供了执行重锚定的环境——搜索、追问、对比。

对于 ADHD 用户，这意味着：
1. **使用前写下你的目标**。哪怕一句话，把它贴在屏幕上。
2. **每次搜索后问自己**：“这个答案帮我接近目标了吗？”
3. **用追问来校准**，而不是点击新链接。
4. **设置时间限制**。用 Forest 或番茄钟，防止超聚焦跑偏。

对于 Agent 工程师，这意味着：
- 你的 agent 需要每 N 步注入一次原始指令（类似 Perplexity 的“回到问题”）。
- 提供“偏离检测”机制：当 agent 的输出与目标相似度低于阈值时，强制重锚定。
- 不要依赖模型自身的上下文——它一定会漂移。

## 今天就能试的行动

1. **写一个“锚点便签”**：打开 Perplexity 前，在便签上写下当前任务的一句话目标。贴在屏幕边缘。
2. **使用“追问代替新搜索”**：每次想开新搜索时，先追问当前答案，直到无法再追问。
3. **设置 15 分钟倒计时**：用手机或 Forest 应用，每次 Perplexity 会话不超过 15 分钟，之后强制暂停并检查目标是否达成。
4. **记录漂移次数**：在便签上画正字，记录每次被无关链接带偏的次数。一周后复盘，你会看到模式。

Perplexity 不是解药。但它可以是你的重锚定脚手架——前提是你愿意自己握住方向盘。
