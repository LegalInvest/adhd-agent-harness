# 为什么用 Mem 治 ADHD 的任务启动困难，和给 agent 套 function calling 工具调用 是一回事？

> 有一种启动失败发生在你真的想开始之后:打开任务,发现「做这件事需要的东西」散落四处——上次的想法记在某个聊天里,参考链接存在收藏夹,关键决定在三周前的会议记录里。找齐这些东西要二十分钟,而你的启动能量只够燃烧五分钟。**任务没吓住你,是「取上下文」的成本先把你耗死了。** function calling 体系里有个专门的角色管这件事,叫检索工具——而 Mem 这类自组织笔记,赌的就是把这个角色做到零摩擦。

先看工程侧。agent 执行任务前的第一类调用几乎总是检索:查文档、搜历史、取相关记录——**「先取上下文,再开始执行」是标准序列**。这里有个容易忽略的设计权衡:检索的成本取决于**写入时的组织成本**。传统方案要求写入者预先分类打标(高写入成本,低检索成本);但如果写入者不肯付这个成本(而 ADHD 用户几乎必然不肯),库就会退化成垃圾堆,检索失效,整条链断掉。**现代方案的答案是:写入零组织,检索靠语义**——随手扔进去,找的时候用自然语言问,让嵌入和索引承担组织成本。

ADHD 侧的病灶精确对应。ADHD 用户的笔记史几乎都是同一个循环:满怀热情建立分类体系→坚持九天→开始随手乱丢→系统失效→换一个新工具重来。**问题从来不是不记,是「记的时候还要决定放哪」这个额外决策**——每次写入都要付执行功能税,税收不上来,系统破产。而检索侧的失败同样致命:启动任务时想不起来「相关的东西记在哪」,于是索性凭空开始(丢失历史上下文,重复劳动)或者干脆不开始。

Mem 的设计恰好是「写入零组织+检索靠问」:笔记不用分文件夹,自动按关联组织,找东西直接搜或问 AI。用 function calling 的语言:**它把你的历史自我变成了一个可调用的检索工具**——启动任务的第一步,从「翻箱倒柜」变成一次调用:「关于 X 项目我记过什么?」按这个架构用:

**一、写入协议:凡念头必进,禁止组织。** 想法、链接、会议要点、「回头要处理」——**统统直接扔,不分类、不打标、不纠结格式**。写入的唯一 KPI 是速度:超过十五秒的写入就是在交执行功能税,税会杀死习惯。零组织不是懒,是这套架构的设计前提。

**二、启动序列:先调用检索,再动手。** 任何任务开始前,固定第一步:搜一下任务关键词,看历史自我留了什么——**两分钟的检索,经常省掉二十分钟的重新思考,更重要的是它本身就是个低门槛的「点火动作」**:检索不需要勇气,而检索结果里的旧想法常常自带「对了,当时想到这里——」的续接动力。

**三、给高频项目建「入口笔记」。** 全自动关联也有失灵时,给每个进行中的项目留一条置顶笔记(当前状态/下一步/关键链接)——这是检索之上的缓存层,最热的上下文不该每次都重新搜。

**四、警惕「囤而不检」。** 写入零摩擦的副作用是库膨胀成只进不出的仓库——**检索工具的价值在被调用,不在库存量**。每周问它一次「这周我记了什么」,让库保持活的。

## 边界

同构强度 B 级:检索增强是真实的 agent 架构,ADHD 的组织困难与工作记忆外化需求有文献支撑,Mem 本身无对照研究,「历史自我作为检索工具」是功能映射。声明:笔记系统解决的是信息启动成本,不是启动的神经门槛——上下文齐了还是点不着的日子,回到评估与治疗主线;另外任何云端笔记都别放敏感的医疗与情绪记录,那部分建议放本地或纸上。

## 今天就能试的 3 件事

1. **今天起执行「凡念头必进,禁止组织」**:写入十五秒封顶,连续七天。
2. **给下一个任务加「先检索」步骤**:动手前先问一句「关于这事我记过什么」。
3. **给最重要的项目建入口笔记**:当前状态+下一步+链接,三行,置顶。

启动能量是稀缺资源,别让它烧在「找东西」上——**把组织成本外包给机器,把历史自我变成一次调用就能到场的顾问**。你不是记性差,是一直在用高税率的系统记东西。
