# Routinery 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 Routinery：它到底补上了哪一层执行功能？

你打开 Routinery，设定好“25 分钟工作 + 5 分钟休息”的番茄钟，开始执行。前两个番茄钟还算顺利，第三个番茄钟进行到一半，你突然发现自己正在刷手机——目标漂移了。你责怪自己意志力不够，但问题可能出在工具的使用方式上：Routinery 只是给了你一个时间框，却没有帮你重锚定“当下该做什么”。

这就像你在给 LLM 写了一个长 prompt，但 agent 执行到一半被上下文膨胀带偏，忘记了最初的任务目标——你需要一个重锚定 system prompt。ADHD 大脑与 LLM 在结构上同构：两者都是“高产但缺乏可靠执行调度层的生成核心”（来源：AI 与 ADHD 的专注力）。Routinery 之于 ADHD，就像重锚定 system prompt 之于 LLM——但有人用错了。

## 问题：分心走神与超聚焦跑偏

ADHD 的注意力问题并非总是“无法专注”。有时你会进入超聚焦（hyperfocus）状态，沉浸在一件事中无法自拔，以至于忘记了原本更重要的目标。比如，你本打算写报告，却花了两小时完美排版一个表格。这种“超聚焦跑偏”与 LLM/agent 在长程任务中的目标漂移如出一辙：agent 可能因为上下文中的某个细节而偏离原始指令，输出无关内容。

更常见的是分心走神：你被环境中的噪音、手机通知、甚至自己的思绪带离任务。ADHD 的[[工作记忆]]缺陷使得你难以在脑中保持任务目标，一旦被打断，就难以回到正轨（来源：工作记忆）。LLM 也有类似问题：在长上下文推理中，Transformer 的注意力会随序列变长而熵增，导致注意力弥散，表现下降（来源：工作记忆，引用耶鲁大学研究）。

## 同构的解法：重锚定与外部脚手架

问题的本质是执行调度层的缺失。ADHD 大脑与 LLM 都需要一个外部系统来定期“重锚定”——提醒自己“当前的目标是什么”。

在 LLM/agent 工程中，我们通过定期注入“重锚定 system prompt”来解决目标漂移。例如，每执行几步就向 agent 发送“请回顾你的原始任务：XXX。你现在正在做什么？是否偏离？”这样的提示。这相当于给 agent 一个外部记忆和注意力锚点。

对于 ADHD，Routinery 这类工具本应扮演同样的角色。但很多人用错了：他们把 Routinery 当作一个简单的计时器，只设定了时间块，却没有在每次番茄钟开始时明确“这个番茄钟我要做什么”。正确的用法是：每个番茄钟开始前，花 10 秒大声说出或写下当前任务目标——这就是你的“重锚定 system prompt”。

## Routinery 到底补上了哪一层执行功能？

Routinery 的核心功能是“结构化时间”——将时间分割成可执行的块，并提供视觉进度条。这主要补偿了[[时间盲]]和执行功能中的“时间管理”层面（来源：执行功能）。但对于 ADHD 来说，最关键的短板往往是“任务启动”和“目标维持”。Routinery 如果没有与任务分解、目标声明结合，就只是一个空壳。

相比之下，[[Focusmate]] 通过实时同伴问责提供了更直接的重锚定：当你看到屏幕对面的伙伴正在工作，你会被提醒“我也该工作”。其机制是“AI-Matched Body Doubling”（来源：Focusmate），利用算法匹配伙伴，实现身体在场效应。而 [[RescueTime]] 则通过自动记录时间使用，事后帮你识别漂移模式（来源：RescueTime）。

## 脚手架 vs 拐杖：边界在哪？

这里有一个关键争议：AI 工具是“脚手架”还是“拐杖”？脚手架帮助你在建设时站稳，但最终你要自己走路；拐杖则让你永远依赖。

Routinery 如果只是被动计时，你可能会形成“没有计时器就无法工作”的依赖——这就是拐杖。但如果它被你主动用作“重锚定触发器”，每次提醒你重新声明目标，那它就是脚手架。证据显示，过度依赖 AI 工具可能削弱内在时间感知能力（来源：矛盾与存疑）。因此，使用 Routinery 的关键是：不要让它替代你的执行功能，而是用它来训练你的执行功能。

## 争议与局限

必须诚实指出：目前 Routinery 针对 ADHD 的效果缺乏大规模临床研究，多数证据来自用户报告（来源：矛盾与存疑）。个体差异也很大——对某些人来说，结构化时间反而增加焦虑。此外，Brain.fm 声称通过神经锁相改善专注，但缺乏 ADHD 独立研究（来源：Brain.fm）。这些工具的“AI”标签有时只是营销。

## 今天就能试的行动

1. **重锚定声明**：每次启动 Routinery 番茄钟前，大声说“这个番茄钟我要完成[具体任务]”。结束后，用 10 秒复盘是否做到。这模拟了 LLM 的重锚定 system prompt。
2. **结合外部问责**：如果 Routinery 不够，尝试 Focusmate 的虚拟身体在场——预约一个时段，让伙伴看到你工作。
3. **事后分析漂移**：用 RescueTime 记录一周，找出你最容易跑偏的时间段和场景，然后在这些时段设置 Routinery 的“强制重锚定”提醒。
4. **限制使用时长**：每周至少有一天不用任何工具，训练内在执行功能，避免依赖（来源：矛盾与存疑）。
