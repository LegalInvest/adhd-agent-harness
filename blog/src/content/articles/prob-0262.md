# Motion 之于 ADHD，就像 上下文工程 之于 LLM——但有人用错了

> 从同构视角实测 Motion：它到底补上了哪一层执行功能？

你刚打开 Motion，准备把今天的任务扔给它自动排程。三分钟后，它告诉你：14:00-15:30 写报告，16:00-17:00 回邮件。你松了口气，觉得终于有救了。但两周后，你可能发现自己的时间感知并没有变好——你只是把调度权交给了算法，而算法并不知道你此刻真正能做什么。

这个困境，和 LLM agent 工程师遇到的完全一样：当你给 agent 一个巨大的上下文窗口，它开始忽略关键指令、输出质量退化——你需要的不是更大的窗口，而是上下文工程：主动设计它‘当下该看什么’。

## 同构：高产核心 + 缺失的调度层

ADHD 大脑与 LLM 在结构上高度同构：两者都是“高产但缺乏可靠执行调度层的生成核心”（来源：AI 与 ADHD 的专注力）。ADHD 侧，创意发散能力惊人（创意产出率是普通人的 1.8 倍，来源：Lifted Ventures 2024），但执行功能——任务启动、时间感知、注意力维持——常常掉线。LLM 侧，文本生成能力强大，但上下文一膨胀就推理退化，控制逻辑一分散就行为失控。

这种同构意味着：ADHD 需要的不是更聪明的 AI，而是一个外部调度层——harness。曾国藩的“日课十二条”就是一套手工 harness：黎明即起、读书不二、写日记反省——用最笨最稳的结构抵消自己的冲动。李鸿章跟着曾国藩学“诚”，用“外须和戎，内须变法”的务实框架稳住自己的浮躁。这些系统与 LLM agent 中的“定时 re-grounding”和“外部调度器”完全对应：日课相当于每段时间的固定提示词，写日记相当于记录执行日志用于后续分析。

## Motion 补上了哪一层？

Motion 的核心功能是自动排程：你把任务丢进去，它根据优先级和截止时间自动分配到具体时段。对 ADHD 来说，这直接对应两大痛点：**任务启动困难**（不用再想“先做什么”）和**时间盲**（不用再估算“这个任务要多久”）。

但 Motion 真正独特的价值，是它把“时间盲”转化成了“可视化的上下文锚点”。ADHD 的时间盲表现为“无法感知时间的流逝”（来源：时间盲），而 Motion 用时间块在日历上画出任务边界，让时间变得可触摸。这就像 LLM agent 中的上下文窗口管理：当你把 agent 的上下文限制在最近 2000 token 时，它的推理质量明显提升——因为干扰信息被隔离了。Motion 做的，就是把你的注意力上下文限制在当前时间块内的任务上。

然而，Motion 的初始设置成本很高。你需要先输入所有任务、设定优先级和预估时长——这对执行功能弱的用户本身就是挑战。更关键的是，它默认“自动完成一切”，容易让人产生依赖：用户不再训练自己的时间感知能力，而是彻底交给算法。

## 脚手架 vs 拐杖：Motion 的边界

这里必须诚实指出争议：AI 工具是“外部执行功能层”还是“拐杖”？（来源：矛盾与存疑）Motion 如果只是帮你排好日程、你被动执行，那它就是拐杖——长期使用可能导致内在执行功能退化。但如果它同时提供“为什么这样排”的透明逻辑，让你逐渐学会自己估算时间、自己排优先级，那它就是脚手架。

类比 LLM agent：一个优秀的上下文工程方案不是把 agent 的所有决策都硬编码，而是设计一套规则让 agent 学会自己管理上下文。Focusmate 做得更好：它通过实时同伴问责提供外部结构，但用户必须自己承诺任务、自己面对伙伴——它补的是“启动”和“维持”，而不是“替代决策”。RescueTime 则通过事后记录帮你看见时间去向，属于“反馈层”而非“调度层”。

## 今天就能试的行动

1. **用 Motion 但只排 3 个任务**：不要把所有任务都丢进去，只选今天最重要的 3 个，手动设定时间块。观察自己是否真的在时间块内完成了——这是训练时间感知的第一步。
2. **每周一次 RescueTime 复盘**：用 Motion 排程一周后，打开 RescueTime 对比“计划 vs 实际”。看哪些时间块被高估或低估，调整你的预估能力。
3. **为 Motion 设置一个“不自动排程”模式**：如果 Motion 支持，关掉自动调度，只用手动拖拽。这能让你保持对时间分配的主控权。
4. **引入 Focusmate 作为启动器**：在 Motion 排好的时间块开始前，预约一个 Focusmate 时段。用同伴问责确保你准时坐到桌前。

Motion 不是坏工具，但用错它的人，只是把“被环境带偏”换成了“被算法带偏”。真正的脚手架，是让你最终能自己掌握上下文工程——无论你是 ADHD 大脑，还是 LLM agent。
