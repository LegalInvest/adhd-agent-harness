# Claude 之于 ADHD，就像 planner-executor 任务分解之于 LLM——但有人用错了

> agent 工程有个朴素到近乎无聊的发现:同一个模型,直接丢给它复杂任务会崩,先让它当 planner 拆解、再让它当 executor 逐步执行,成功率立刻上一个台阶。无聊,但这恰是 ADHD 用户拿 Claude 能兑现的最大一笔红利——前提是搞清楚 planner 和 executor 到底谁是谁。

先看这个架构为什么有效。LLM 一口气跑长任务的失败模式:中途忘记约束、跳过步骤、在某个子问题上钻进死胡同。planner-executor 把认知负荷切开:**planner 只管「分成哪几步」,不管做;executor 只管「做好当前这一步」,不管全局**——每个角色的工作记忆需求都被压到了它能承受的范围内。留意这句话:它同时也是 ADHD 任务瘫痪的精确诊断。「写年终总结」之所以启动不了,是因为你试图同时当 planner 和 executor:一边想整体结构,一边想第一句怎么写,一边焦虑来不来得及——**三个进程挤在一个本来就小的工作记忆里,系统死锁。**

所以「用 Claude 治任务瘫痪」的正确架构只有一句话:**让 Claude 当 planner,你当 executor。** 具体铺开:

## 正确的分工协议

**一、planner 阶段:把任务整个倒出去。** 连同你的焦虑一起:「要写年终总结,不知从哪下手,材料散在周报里,还有三天。」要求输出:步骤序列、每步 30-60 分钟、每步有产出物、按什么顺序。**你在这个阶段的唯一职责是审批**:哪步不对就说,让它改——审批比生成省力一个数量级,这正是你要占的生态位。

**二、executor 阶段:一次只看一步。** 计划批准后,进入执行:只做当前步,做完回来报告(「第 2 步完成」),再看下一步。**不要在执行中重新打开全局视图**——「还剩这么多」的全景恰是焦虑和瘫痪的来源;agent 工程里 executor 的上下文里也只放当前步,同一个道理。

**三、卡住时:降级调用。** 某一步做不动,不是意志问题,是这一步还是太大——丢回给 planner:「第 3 步启动不了,再拆细。」拆到能动为止。**「拆到能动为止」是这套架构的核心承诺:任何瘫痪都被翻译为「粒度不够细」,而不是「我不行」。**

## 用错的三种姿势

**自己当 planner,让 Claude 当 executor**:让它替你写完所有内容,你只点头——短期爽,长期两个代价:产出没有你的判断和材料(年终总结里全是正确的废话),以及你的执行肌肉持续萎缩。**分工的原则:拆解外包,因为那是你的系统性短板;执行保留,因为产出物的灵魂在执行里。**(需要外包的执行,见 function calling 那篇的清单——格式转换类可以,原创核心不行。)**规划循环成瘾**:计划改了五版,一步没执行——规划的多巴胺比执行便宜,ADHD 尤其容易在这里空转;防法:计划 v1 批准后立即做第一步,任何修订必须发生在「至少执行了一步」之后。**丢弃执行反馈**:做完不回报,下次重新开对话重新拆——执行中发现的真实信息(哪步比想的难、哪步其实不需要)是计划迭代的燃料,agent 系统里 executor 的反馈会回流给 planner,你也该让它回流。

## 边界

同构强度 A 级:任务分解对 LLM 复杂任务表现的增益有直接研究,ADHD 的规划与工作记忆负荷问题有扎实文献,「planner 外包+executor 保留」的映射直接且可操作。声明:任务瘫痪若伴随弥漫性的动力丧失、情绪低落,注意与抑郁鉴别,请专业评估;本文的架构对「结构缺失型瘫痪」有效,对「情绪淹没型瘫痪」只是半个答案,另半个在治疗室里。

## 今天就能试的 3 件事

1. **挑当前最瘫的任务,跑一遍完整协议**:倒出去→审批计划→只看第一步→做。
2. **练一次「降级调用」**:哪步卡住,丢回去拆细,体会「拆到能动」的下限在哪。
3. **立「先执行一步再改计划」规则**:防规划空转,从今天的任务开始。

planner-executor 不是什么高深架构,它只是承认了一件事:**拆解和执行是两种负荷,挤在一个脑子里会死锁。** Claude 替你扛走前一种,你专心做后一种——从此「不知从哪下手」不再是瘫痪的理由,它只是一次待发送的 planner 调用。
