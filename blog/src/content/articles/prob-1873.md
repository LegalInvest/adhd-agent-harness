# ADHD 创业者视角：为什么「治好 ADHD 的任务启动困难、不会拆解」和「让 LLM 不跑飞」其实是同一道工程题？

## 问题：为什么你列了清单，却迟迟不动？为什么 LLM 写了计划，却中途跑偏？

如果你是一个 ADHD 创业者，你大概很熟悉这个场景：脑子里有一堆好点子，但就是没法把它们变成行动；列了详细的任务清单，却连第一项都启动不了。如果你是一个在做 Agentic Harness 的工程师，你大概也很熟悉另一个场景：LLM 生成了完美的计划，但执行到一半就开始胡编乱造，或者陷入死循环。这两件事，其实是同一道工程题。

## 同构：ADHD 大脑与 LLM——高产但缺调度层的生成核心

ADHD 大脑的智力与创造力（生成核心）与 LLM 的生成能力类似，但缺乏执行功能（调度层）来组织行动（来源：AI 与 ADHD 的时间管理—核心论点）。LLM 本身也是生成核心，缺乏可靠的执行调度层（来源：拐杖与脚手架—LLM/harness 一侧）。两者都面临“无状态”困境：ADHD 的工作记忆容量有限且不稳定，导致跨会话任务难以持续（来源：无状态与外部记忆—ADHD 一侧）；标准的 LLM 调用是无状态的，每次对话都是独立的（来源：无状态与外部记忆—LLM/harness 一侧）。因此，ADHD 个体需要“第二大脑”工具来存储上下文（来源：无状态与外部记忆—ADHD 一侧），而 agent 系统需要外部记忆（如向量数据库）来解决溢出（来源：无状态与外部记忆—LLM/harness 一侧）。

## 解法：planner-executor 循环——同一套 harness 思路

### 对于 ADHD 侧：外部规划循环降低启动门槛

ADHD 的任务启动困难，核心在于“下一步该做什么”的决策负担过重。工具如 Motion 和 Reclaim.ai 通过自动排程与动态调整，消除这一决策负担，直接帮助用户克服时间盲和任务启动困难（来源：Motion—与 ADHD 的关系；Reclaim.ai—与 ADHD 的关系）。Motion 持续评估任务优先级、截止日期和可用时间，实时重建日程（来源：Motion—AI 如何介入）；Reclaim.ai 则通过创建深度工作块，自动防御会议侵占（来源：Reclaim.ai—AI 如何介入）。这些工具本质上提供了外部 planner-executor 循环：planner 自动排程，executor 通过提醒触发行动。

Tiimo 则通过视觉化时间线，将时间转化为可见元素，直接应对时间盲问题（来源：Tiimo—与 ADHD 的关系）。这相当于将时间感知外化为可视化的“上下文窗口”，帮助 ADHD 个体维持任务连贯性。

### 对于 LLM/agent 侧：harness 工程提供规划与验证循环

在 agent 系统中，harness 工程正是设计围绕 LLM 的脚手架——包括上下文传递、工具接口、规划工件、验证循环、记忆系统和沙箱（来源：拐杖与脚手架—LLM/harness 一侧）。没有 harness 的 LLM 只是一个无状态的 API 调用（来源：无状态与外部记忆—LLM/harness 一侧）。生产级 agent 通过 planner-executor 循环分解长任务：planner 生成子任务，executor 执行并返回结果，然后验证循环检查是否偏离，必要时重新规划。这与 Motion 的动态调整逻辑完全一致：规划、执行、反馈、再规划。

## 核心观点：脚手架，不是拐杖

无论是 ADHD 个体还是 LLM agent，外部规划循环都是必要的脚手架，但必须警惕过度依赖。真正的脚手架应帮助使用者“建造”，但使用者仍需自己“攀登”（来源：拐杖与脚手架—ADHD 一侧）。对于 ADHD 个体，若工具替代了治疗或自我管理，可能造成依赖（来源：拐杖与脚手架—ADHD 一侧）。对于 agent 系统，若 harness 过于僵化，代理将失去灵活性。目前多数 ADHD 工具（如 Goblin Tools、Saner.AI）设计为长期使用，未提及撤除机制（来源：矛盾与存疑—过度依赖 vs 脚手架可撤除性）。这提示我们：好的 harness 应设计为可逐步撤除的临时支撑，而非永久拐杖。
