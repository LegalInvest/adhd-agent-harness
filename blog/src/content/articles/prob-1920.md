# ADHD 程序员如何用「给生成核心套 harness」的思路应对在 IDE 与 agent 之间切换里的执行功能（计划/组织/启动）先天偏弱？

> 把 LLM 工程里的「agent 的编排/记忆/工具层就是「外部执行功能」」迁移到ADHD 程序员的在 IDE 与 agent 之间切换场景，双域证据作答。

## 问题：IDE 与 agent 之间的执行功能断崖

作为 ADHD 程序员，你可能熟悉这样的场景：你正在 IDE 中写代码，突然想起要测试某个 agent 行为，于是切换到 agent 面板。但就在切换的瞬间，你忘了原本要测什么；或者你被 agent 的输出吸引，开始调 prompt，半小时后才发现自己完全偏离了原计划。这不是意志力问题——这是执行功能（计划、组织、启动）先天偏弱在作祟。

ADHD 大脑的典型特征是“高产但缺乏可靠执行调度层的生成核心”（来源：ADHD × AI 的科学与研究前沿）。你的大脑能产生无数创意，但缺乏内建的机制来排序、启动、跟踪任务。在 IDE 与 agent 之间频繁切换时，这种缺陷被放大：上下文丢失、任务碎片化、启动困难。

## 同构：ADHD 大脑与 LLM agent 共享同一架构缺陷

有趣的是，LLM agent 面临完全相同的结构性问题。一个裸 LLM 能生成流畅文本，但缺乏编排层来管理多步任务、缺乏记忆系统来保持上下文、缺乏工具层来执行实际动作。这就是为什么工程师要为 agent 套上 harness——即 agent 的编排/记忆/工具层。Harness 提供上下文传递、工具接口、规划工件、验证循环和记忆系统（来源：幻觉与验证循环）。

ADHD 大脑与 LLM agent 是同构的：两者都是强大的生成核心，但缺少内建的执行调度层。因此，解法也同构：为生成核心套上外部执行功能层——也就是 harness。

## 证据：ADHD 侧与 LLM/agent 侧的对应

### ADHD 侧：外部执行功能工具的真实效果

现有 AI 工具已开始充当这种外部 harness。例如，Goblin Tools 的 Magic ToDo 功能自动将复杂任务分解为小步骤，降低启动门槛（来源：Goblin Tools）。用户反馈称它“将压倒性的事情变成一系列不压倒性的事情”（来源：Goblin Tools）。Saner.AI 通过本地记忆和知识回忆减少搜索循环和标签切换（来源：Saner.AI），直接补偿工作记忆缺陷。Motion 则自动排程并动态调整每日计划，消除“下一步该做什么”的决策负担（来源：Motion）。这些工具本质上提供了 ADHD 大脑缺失的计划、组织和启动功能。

### LLM/agent 侧：Harness 工程如何解决类似问题

在 agent 工程中，harness 通过结构化工作流和规划将复杂任务分解为子任务，并在每一步进行验证（来源：幻觉与验证循环）。上下文管理是关键：有效的 harness 需防止上下文膨胀和推理退化（来源：上下文工程）。验证循环（如对模型编写的代码运行测试用例）则校正 LLM 的“幻觉”倾向（来源：幻觉与验证循环）。这些机制与 ADHD 工具的任务分解、记忆补偿、时间管理形成一一对应。

## 核心判断：Harness 是脚手架，不是拐杖

本文的核心观点是：ADHD 程序员应主动设计自己的“个人 harness”，将 IDE 与 agent 之间的切换场景结构化。但必须警惕依赖风险。Wiki 资料中明确指出，脚手架应设计为可逐步撤除的临时支撑，而多数现有工具（如 Goblin Tools、Saner.AI）设计为长期使用，未提及撤除机制（来源：矛盾与存疑）。长期依赖是否削弱内在执行功能？目前缺乏纵向研究。因此，你的 harness 应包含撤除计划：例如，先使用工具建立习惯，然后逐渐减少使用频率，让内在能力成长。

## 今天就能试的 3 条行动

1. **用 Goblin Tools 分解一次切换任务**：下次从 IDE 切换到 agent 前，在 Goblin Tools 中输入“测试 agent 的 X 功能”，让 AI 分解为具体步骤（如“打开 agent 面板”“输入测试 prompt”“记录输出”），贴在显示器边缘。这降低了启动难度，并防止上下文丢失。

2. **用 Saner.AI 建立切换记忆库**：在 Saner.AI 中创建“切换笔记”项目，每次切换前快速记录当前 IDE 中的代码状态和意图，切换后记录 agent 中的发现。这模拟了 agent harness 的上下文传递功能。

3. **用 Motion 规划切换时段**：在 Motion 中设置每日“IDE 专注块”和“agent 实验块”，让 AI 自动排程并提醒切换。这消除了“何时切换”的决策负担，并利用外部时间感知克服时间盲。

## 局限与争议

必须诚实指出，ADHD 大脑与 LLM 同构的命题主要基于概念类比和工具案例，缺乏大规模实证（来源：矛盾与存疑）。工具的有效性证据多来自用户反馈，而非独立临床研究。长期使用外部 harness 是否削弱内在执行功能？目前无定论。此外，多巴胺干预的有效性存在争议（来源：矛盾与存疑），因此本文未推荐基于多巴胺的工具。你的个人 harness 应基于自身反馈不断调整，而非盲目依赖。

总之，ADHD 程序员在 IDE 与 agent 之间切换的困境，不是个人缺陷，而是生成核心缺少外部调度层的必然结果。通过套用 harness 架构——即外部执行功能层——你可以将混乱的切换转化为可控的工作流。记住：脚手架是为了让你最终能自己站立。
