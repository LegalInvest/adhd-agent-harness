# Tiimo 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Tiimo：它到底补上了哪一层执行功能？

你有没有过这样的体验：明明设了闹钟、写了待办清单，甚至用了最花哨的日程 App，但时间还是像流沙一样从指缝溜走，你依然在截止日前最后一刻才启动任务？

如果你是 ADHD 人群，你大概率会点头。如果你是做 Agentic Harness 工程的工程师，你可能会想到另一种场景：你给 LLM 配了计算器工具，但它还是算错数——因为它没有在正确时机调用，或者调用了却不知道如何利用结果。

这两个问题共享同一个结构缺陷：**生成核心（ADHD 大脑 / LLM）有强大的产出能力，但缺乏可靠的执行调度层**。而 Tiimo 这样的工具，被设计来补上这个缺口——但很多人用错了。

## 同构：ADHD 的时间盲与 LLM 的编排盲

ADHD 的核心缺陷之一是时间盲——无法准确感知时间流逝，导致计划与执行脱节（来源：ADHD 的 AI 工具全景）。LLM 的对应缺陷是编排盲——它能理解任务描述，但无法自主决定何时、以什么顺序调用工具（来源：ADHD 的 AI 工具全景）。

两者都需要一个外部调度层：对 ADHD 来说是视觉化日程与定时提醒，对 LLM 来说是 function calling 的编排逻辑。Tiimo 正是为前者而生——它用色块、图标和倒计时将抽象时间转化为可感知的视觉序列。

## Tiimo 到底补上了哪一层？

Tiimo 的核心功能是**将时间结构化**。它不是简单的日历，而是把一天切成可管理的“时间块”，每个块有明确的任务、时长和视觉提示。这直接对治 ADHD 的时间盲：当你看到“15 分钟写邮件”的色块，你不再需要自己判断“15 分钟有多长”，而是依赖外部提示来启动和切换（来源：ADHD 的 AI 工具全景）。

这就像 LLM 的 function calling：当你给模型一个工具，比如 `calculate()`, 你不仅需要告诉它工具存在，还需要在编排层定义“当用户问数学问题时，先调用 `calculate()`，再根据结果回答”。没有编排的 function calling 只是摆设——同样，没有主动使用的 Tiimo 也只是另一个被遗忘的 App。

## 用对了：脚手架 vs 拐杖

问题在于，很多人把 Tiimo 当作拐杖——只是被动地看日程，而不是主动用它来驱动行为。脚手架与拐杖的关键区别在于：**脚手架支持你完成原本做不到的事，并可以逐步撤除；拐杖则让你依赖它，撤除后能力反而退化**（来源：ADHD 的 AI 工具全景）。

Tiimo 的正确用法是：
- **主动交互**：在每个时间块开始时，手动标记“开始”，结束时标记“完成”，形成闭环反馈。
- **动态调整**：如果某个任务超时，立即调整后续时间块，而不是让整个计划崩盘。
- **事后复盘**：回顾一天的时间块完成情况，识别自己的时间感知偏差。

这种用法对应 LLM 的智能编排：不是简单地把工具列表丢给模型，而是设计一个循环——调用工具、获取结果、评估结果、决定下一步。如果只是让 LLM 自己决定是否调用工具，结果往往是不调用或乱调用（来源：ADHD 的 AI 工具全景）。

## 证据与局限

Tiimo 的设计理念与 ADHD 干预策略一致：结构化日程、视觉提示、时间感知训练。但需诚实指出，目前缺乏针对 Tiimo 的独立随机对照试验，其效果主要基于用户反馈和概念对齐（来源：ADHD 的 AI 工具全景）。

更大的争议在于：过度依赖视觉日程是否会削弱内在时间感知能力？一些批评者认为，如果用户永远依赖外部提示，他们可能永远学不会内部计时（来源：矛盾与存疑）。这就像 LLM 如果永远依赖外部工具来计算，它自己的数学推理能力可能永远不会提升。

## 今天就能试的行动

1. **把 Tiimo 当遥控器，不当电视**：每次时间块开始，手动点“开始”，结束后点“完成”。坚持一周，你会开始感知时间块的真实长度。
2. **设置“缓冲块”**：在每天安排 2-3 个 15 分钟的空白时间块，用于应对超时或意外中断。这对应 LLM 编排中的 fallback 机制。
3. **每周复盘一次**：查看 Tiimo 的完成记录，找出哪些任务经常超时，然后调整时间块长度。这是对调度层的迭代优化。

Tiimo 的价值不在于它有多漂亮，而在于你是否把它当作一个主动的调度层来使用。同样，function calling 的价值不在于有多少工具，而在于你是否设计了智能的编排逻辑。两者共享同一个教训：**工具只是脚手架，真正的能力在于如何使用它**。
