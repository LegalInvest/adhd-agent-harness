# Tiimo 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Tiimo：它到底补上了哪一层执行功能？

你有没有过这样的瞬间：明明知道该做什么，大脑却像死机一样无法启动；或者刚打开浏览器，就忘了自己为什么要打开它——然后陷入一连串无意义的标签切换？对于 ADHD 人群，这种「知道该做，但做不出来」的体验是日常。而对于 LLM/Agent 工程师，类似的场景是：模型知道该调用哪个 API，却因为上下文膨胀或指令模糊，给出了一个格式错误的 JSON，或者干脆在幻觉中自说自话。

这两类问题，本质上是同一个底层结构缺陷的两种表现：一个高产但缺执行调度层的生成核心，遇到了需要精确、重复、状态维护的任务。ADHD 大脑与 LLM 同构——两者都是「高产但缺执行调度层的生成核心」（来源：ADHD 的 AI 工具全景）。而 Tiimo 这类视觉时间管理工具，之于 ADHD，就像 function calling 工具调用之于 LLM：它们都是为生成核心套上 harness（脚手架），通过外部精确层来补偿内在的调度缺陷。但许多人用错了——他们把它当成了拐杖，而不是脚手架。

## 问题：精确、重复、状态维护——生成核心的致命弱点

ADHD 人群在哪些任务上最容易崩溃？不是创意发散，而是那些需要「保持精确状态」的事：比如按时吃药、记录每日支出、在固定时间回复邮件。这些任务要求你维持一个稳定的内部状态，而 ADHD 大脑的执行功能恰恰在此失灵——工作记忆容量保留但控制弱，易被干扰（来源：ADHD 的 AI 工具全景）；时间感知缺陷导致「时间盲」（来源：时间盲概念页）；任务启动困难则让「开始」本身成为一道坎（来源：执行功能障碍概念页）。

LLM/Agent 也有类似的「死穴」。一个没有 function calling 的 LLM，你让它「帮我查一下明天的天气然后设置提醒」，它可能会生成一段漂亮的描述，但不会真的去调用天气 API 和日历 API。它擅长生成内容，但不擅长「精确执行状态维护类任务」——这恰恰是 function calling 要解决的问题。Agent 通过 function calling 把精确计算外包给工具，就像 ADHD 大脑通过外部脚手架来补偿调度缺陷。

## Tiimo 与 function calling：同构的解决方案

Tiimo 是一款视觉时间管理应用，专为 ADHD 和自闭症人群设计。它用时间轴+图标的方式展示日程，而不是传统的列表。这种设计直接针对「时间盲」和「任务启动困难」：视觉化的时间线提供了外部时间线索，减少了对内部时间感知的依赖；图标化的任务降低了文字阅读的认知负荷，让「下一步该干什么」一目了然。

从同构视角看，Tiimo 之于 ADHD，就像 function calling 之于 LLM：
- **Tiimo 提供外部时间线索**，补偿 ADHD 的时间感知缺陷（来源：时间盲概念页）；function calling 提供外部 API 调用能力，补偿 LLM 的时序推理弱点（来源：ADHD 的 AI 工具全景）。
- **Tiimo 将模糊的「上午做点事」转化为可视化的时间块**，降低任务启动门槛；function calling 将模糊的「处理数据」转化为具体的 API 调用链，降低执行错误率。
- **Tiimo 通过持续视觉提醒维持状态**，减少 ADHD 大脑的「上下文漂移」；function calling 通过工具返回的结构化数据维持上下文，减少 LLM 的「上下文膨胀」。

但这里有一个关键区别：Tiimo 是「被动脚手架」，需要用户主动去看、去遵循；而 function calling 是「主动执行层」，Agent 自动调用工具并返回结果。这恰恰是许多人用错 Tiimo 的地方——他们期望它像 function calling 一样自动推进任务，但 Tiimo 本质上只是一个外部记忆和提示系统，它不会替你执行。

## 核心判断：脚手架 vs 拐杖的边界

同构视角揭示了「脚手架 vs 拐杖」的辩证关系：AI 工具应作为脚手架促进能力发展，而非永久拐杖（来源：ADHD 的 AI 工具全景）。Tiimo 和 function calling 都是脚手架——它们补偿缺陷，但不替代能力。

然而，许多 ADHD 用户把 Tiimo 当作「自动执行器」：设置了日程就以为任务会自动完成，结果到了时间点还是不想做。同样，许多 LLM 工程师把 function calling 当作「万能胶」：只要加上工具调用，就以为 Agent 能自动解决所有问题，结果忽略了 prompt 设计、错误处理和状态管理。

证据也支持这一判断：现有 AI 工具的有效性证据主要来自用户报告，缺乏大规模对照实验（来源：矛盾与存疑）。过度依赖 AI 工具可能削弱内在能力——比如过度依赖 Tiimo 可能削弱内在时间感知能力，过度依赖 function calling 可能让工程师忘记如何手动调试 API（来源：矛盾与存疑）。

真正的用法是：把 Tiimo 当作「外部执行功能层」来补偿调度缺陷，但同时有意识地训练自己的时间感知和任务启动能力。就像 function calling 应该被用作「精确执行层」，但工程师仍需理解底层 API 的逻辑和边界。

## 局限与争议

必须诚实指出：Tiimo 针对 ADHD 的效果缺乏独立大规模临床研究，证据主要来自用户报告和设计逻辑（来源：矛盾与存疑）。个体差异很大——有人觉得视觉时间线是救星，有人觉得它只是另一个需要维护的列表。同样，function calling 在复杂任务中的可靠性也受限于模型能力和工具设计。同构视角是一个有用的框架，但它不是临床证据。

最终，Tiimo 和 function calling 都是工具，不是魔法。它们最有效的用法，是作为「脚手架」来支撑你搭建自己的能力，而不是作为「拐杖」让你永远依赖。下次当你打开 Tiimo 或者设计一个 function calling 流程时，问自己：我在用脚手架，还是在找拐杖？
