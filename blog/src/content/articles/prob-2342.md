# ADHD 创业者视角：为什么「治好 ADHD 的不擅长精确、重复、状态维护类任务」和「让 LLM 不跑飞」其实是同一道工程题？

> agent 通过 function calling 把精确计算外包给工具——同一套 harness 思路，两边都成立。

你有没有过这样的体验：脑子里同时涌出十几个想法，每一个都精彩绝伦，但当你试图把它们落地成具体行动时，大脑却像一台没有操作系统的超级计算机——算力爆棚，但调度全崩？

如果你是一名 ADHD 创业者，这种体验可能是日常。如果你是一名 Agentic Harness 工程师，你每天面对的问题几乎一模一样：LLM 能生成惊艳的文本，但一涉及精确计算、状态维护、多步执行，它就“跑飞”了。

这两个问题看似风马牛不相及，但当你把“不擅长精确、重复、状态维护类任务”这个痛点放在一起看，会发现它们共享同一个底层结构：**高产但缺执行调度层的生成核心**。而解法也高度同构——通过外部脚手架（harness）把精确计算外包给工具。

## 同构：ADHD 大脑与 LLM 都是“缺调度层的生成核心”

ADHD 大脑的核心特征是什么？不是“注意力不集中”这么简单。神经科学揭示，ADHD 的多巴胺系统失衡导致中脑皮质分支功能低下，引起执行功能差、行为计划不良（来源：a dynamic developmental theory of ADHD）。换句话说，ADHD 大脑能产生大量创意和冲动，但缺乏一个可靠的内置调度层来排序、启动、维护这些任务。

LLM 呢？它也是强大的生成核心，但无状态、缺乏持续的执行调度。让它写一首诗它可能惊艳全场，但让它按顺序执行“先查数据库，再计算平均值，然后发邮件”这种多步任务，它大概率会中途忘记上下文或输出幻觉。

这正是同构的核心：**两者都是“高产生成 + 低调度可靠性”的组合**。因此，为 ADHD 搭建执行功能支架，与为 LLM 搭建 agent 脚手架，本质上是同一类工程问题（来源：生成核心与调度层）。

## 历史中的 harness：孔子和释迦牟尼的“外部调度器”

这个思路并非新发明。两千多年前，两位“高产但缺调度”的创始人就已经在实践了。

**孔子**：身高近1米9，精力旺盛得周游列国14年也坐不住；冲动爱骂宰予“朽木不可雕”；思维跳跃，《论语》全是场景化语录，没有系统著作。但他的 harness 系统是“克己复礼”——用外在的“礼”作为行为边界，每日反省（“吾日三省吾身”），70岁才做到“从心所欲不逾矩”。这套系统本质上是一个外部调度器：用固定的仪式和反省机制，补偿内在执行功能的不足。

**释迦牟尼**：29岁说走就走放弃王位，先后跟两个仙人学、苦行6年，不断试错不断放弃；说法49年，全场景化比喻，自己一字未写。他的 harness 是“八正道”和“持戒”——用戒律管行为，用正念拴住乱跑的念头。这相当于一个定时 re-grounding 机制，防止生成核心跑飞。

这两位都用自己的 harness 系统，把“缺调度”的大脑变成了影响千年的思想引擎。

## 当代工具：ADHD 的 harness 与 LLM 的 function calling 是同一套模式

今天，AI 工具正在把这种 harness 数字化。

**ADHD 侧**：工具通过微任务分解、外部记忆、智能调度来补偿执行功能缺陷。
- **Goblin Tools** 的 Magic ToDo 能把“清理厨房”自动分解成“把碗放进洗碗机”“擦台面”“拖地”等小步骤，降低启动门槛（来源：Goblin Tools）。
- **Saner.AI** 用本地记忆存储减少搜索循环，相当于给 ADHD 大脑配了一个外部工作记忆（来源：Saner.AI）。
- **Lex** 允许“单一指令触发多步骤任务”，类似 agent 的规划机制（来源：Lex）。

**LLM/Agent 侧**：同样的思路——通过 function calling 把精确计算外包给工具。
- Agent 通过 function calling 调用数据库、计算器、日历 API，而不是让 LLM 自己算。这就像 ADHD 患者用 Goblin Tools 分解任务：**把生成核心不擅长的部分卸载给外部工具**。
- 上下文工程强调主动设计注意焦点，避免 LLM 跑飞（来源：上下文工程）。这对应 ADHD 的“定时 re-grounding”策略。

## 脚手架 vs 拐杖：一个必须警惕的边界

但这里有一个关键争议：**脚手架和拐杖的边界在哪里？**

如果 ADHD 患者过度依赖 AI 工具，可能削弱内在执行功能，变成永久拐杖（来源：拐杖与脚手架）。同样，如果 agent 过度依赖外部工具，一旦工具不可用，系统就瘫痪。

健康的使用方式应该是：**工具作为补偿，而非替代**。孔子70岁才“从心所欲不逾矩”，说明 harness 最终要内化。AI 工具应该帮助用户逐步建立内在能力，而不是永远依赖外部提示。

另一个局限是：目前多数工具缺乏严格临床验证（来源：矛盾与存疑）。同构性理论也仅是类比，并非经过验证的科学事实（来源：矛盾与存疑）。所以，请带着实验心态使用。

## 结语

ADHD 创业者和 Agentic Harness 工程师，你们面对的是同一个敌人：**一个强大的生成核心，配了一个糟糕的调度层**。而解法也相同：用外部脚手架把精确、重复、状态维护类任务外包出去。

这不是巧合。这是认知架构的深层同构。当你下次为自己的“不擅长精确”而沮丧时，不妨想想：你只是缺一个更好的 harness，而不是缺能力。
