# ADHD 创业者视角：为什么「治好 ADHD 的注意力容易被环境带偏」和「让 LLM 不跑飞」其实是同一道工程题？

## 问题：注意力为何总被环境带偏？

如果你是一名ADHD创业者，你可能经历过这样的场景：打开电脑准备写方案，却被微信小红点、浏览器标签页和桌面文件堆轮番劫持，半小时后发现自己正在刷短视频。如果你是一名LLM工程师，你可能同样熟悉这一幕：精心设计的Agent在对话中突然跑题，因为上下文窗口里混入了无关历史，模型便顺着噪音生成了一堆废话。

这两个问题看似分属心理学和计算机科学，但底层结构惊人地一致。ADHD大脑与LLM共享同一个核心矛盾：**生成能力强，调度能力弱**。ADHD大脑像一台永不熄火的生成引擎，想法喷涌而出，但执行功能（工作记忆、任务启动、时间管理）频繁故障（来源：ADHD 大脑与 LLM 的同构）。LLM同样如此——它是强大的文本生成器，但缺乏内置的执行控制，需要外部编排层来确保输出不跑偏（来源：What is an agent harness in the context of large-language models?）。

由此引出一个关键问题：为什么「治好ADHD的注意力容易被环境带偏」和「让LLM不跑飞」其实是同一道工程题？答案藏在**上下文工程**中。

## 同构：高产生成核心 + 缺失调度层

让我们把ADHD大脑和LLM放在解剖台上，看看它们的内部结构。

**ADHD侧**：执行功能缺陷导致工作记忆容量小、任务启动困难、时间感知模糊。大脑像一台没有操作系统的超级计算机——硬件性能强劲，但资源调度全靠碰运气。一旦环境出现干扰（通知、噪音、杂念），当前上下文就被冲走，注意力漂移（来源：AI 与 ADHD 的专注力）。

**LLM侧**：模型本身是无状态的，每次推理只依赖当前上下文窗口。如果窗口里塞入了不相关的内容，或者缺少关键指引，模型就会“迷失方向”，生成偏离目标的输出。这就是为什么Agent需要“harness”——包裹LLM的软件基础设施，负责上下文管理、工具调用和决策路由（来源：What is an agent harness in the context of large-language models?）。

两者的对应关系清晰得令人不安：ADHD的工作记忆丢失 ≈ LLM的上下文溢出；ADHD需要外部工具“减少决策、保留上下文” ≈ LLM需要harness做上下文工程（来源：ADHD 大脑与 LLM 的同构）。两边都在解决同一个问题：**如何让一个高产的生成核心在混乱环境中保持有序输出？**

## 解法：上下文工程——给生成核心套上harness

既然问题同构，解法也应该同构。所谓上下文工程，就是主动管理输入环境，确保生成核心只看到最相关的信息，并持续获得调度信号。

**ADHD侧的上下文工程**：工具如Brain.fm通过神经锁相音乐制造声学“上下文”，让大脑更容易进入专注状态（来源：Brain.fm）。Focusmate则利用AI匹配虚拟同伴，通过实时视频创造“身体在场”的社交上下文，提供外部问责和启动信号（来源：Focusmate）。RescueTime自动追踪时间使用，充当外部记忆，帮助用户觉察注意力漂移模式（来源：RescueTime）。这些工具的共同点是：它们不是试图“修复”大脑，而是**重新设计环境上下文**，让正确的行为更容易发生。

**LLM侧的上下文工程**：Agent harness通过模型连接器、工具调用和工作流编排，确保LLM每次调用都获得干净、相关的上下文。例如，复合AI系统将任务分解为子步骤，每个步骤绑定不同的LLM，并显式管理上下文窗口的内容（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。这就像给LLM戴上一副“注意力眼镜”，只让它看到该看的东西。

**核心判断**：上下文工程不是锦上添花，而是必要条件。对于ADHD大脑和LLM，没有外部调度层，生成核心注定会跑偏。这不是意志力或模型能力的问题，而是架构缺陷。因此，有效的干预不是“更努力”，而是**设计更好的上下文**。

## 争议与局限：脚手架还是拐杖？

尽管同构框架很有吸引力，但必须诚实面对争议。

**脚手架 vs 能力发展**：过度依赖外部工具是否会阻碍内在执行功能的发展？目前无长期研究证明脚手架可逐步撤除（来源：矛盾与存疑）。Brain.fm的神经锁相效果缺乏针对ADHD的独立临床研究（来源：Brain.fm）。Focusmate的效果主要归因于身体在场，而非AI本身（来源：AI 与 ADHD 的专注力）。这些工具更像是“拐杖”——提供即时支撑，但可能让人忘记锻炼肌肉。

**证据缺口**：ADHD大脑与LLM的同构命题主要来自概念类比，缺乏大规模实证（来源：矛盾与存疑）。多巴胺干预的有效性也存在争议（来源：矛盾与存疑）。这意味着，虽然框架有启发性，但不能直接当作临床指南。

**我的观点**：上下文工程工具应被设计为**可撤除的脚手架**，而非永久拐杖。理想情况下，用户在使用中逐渐内化调度策略（如自动识别干扰、建立外部记忆习惯），最终减少对工具的依赖。但目前大多数工具（包括上述三者）并未内置撤除机制，这是产品设计的盲区。

## 今天就能试的行动

1. **为你的工作环境做一次“上下文审计”**：列出所有可能劫持注意力的刺激（通知、标签页、桌面文件）。关闭或隐藏至少3个，坚持一天。这相当于给ADHD大脑做一次harness清理——减少上下文窗口中的噪音。

2. **尝试Focusmate一次**：预约一个25分钟的虚拟同伴时段，专注完成一项小任务。体验“身体在场”如何提供外部调度信号。注意：如果社交焦虑明显，可改为使用Forest等简单的专注计时器。

3. **用RescueTime追踪一周**：不设目标，只观察数据。重点不是评判自己，而是识别注意力漂移的模式——哪些应用、时段、环境最容易导致上下文丢失。这相当于为你的“LLM大脑”做一次上下文日志分析。

4. **为LLM Agent显式设计上下文边界**：如果你在开发Agent，尝试为每个任务定义“上下文窗口”大小（例如最多保留5轮对话历史），并定期清理无关内容。你会发现，模型跑偏的概率显著下降——这同样适用于你自己的工作记忆。
