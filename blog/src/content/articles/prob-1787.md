# ADHD 创业者视角：为什么「治好 ADHD 的工作记忆容量小、边做边忘」和「让 LLM 不跑飞」其实是同一道工程题？

> LLM 无跨会话状态，靠外部记忆/向量库续命——同一套 harness 思路，两边都成立。

## 问题：为什么你一边做事一边忘，LLM 也一边跑一边丢？

如果你是一名 ADHD 创业者，你一定熟悉这样的场景：刚想好要回复客户邮件，打开电脑却开始刷朋友圈；走到厨房拿东西，却忘了自己要拿什么。这是工作记忆容量小的典型表现——大脑像一块黑板，写上去的字很快就被擦掉。

如果你是一名做 Agentic Harness 的工程师，你也一定熟悉这样的场景：LLM 在对话中表现得像个天才，但一旦关闭会话，它就像失忆了一样，下次对话从头开始。LLM 本质上没有跨会话状态，所有上下文都依赖输入窗口，一旦超出限制或会话结束，信息就丢失了。

这两个问题看起来风马牛不相及，但本质上是同一道工程题：**一个强大的生成核心，缺少一个可靠的执行调度层来维持上下文、管理记忆、控制执行流程。** ADHD 大脑和 LLM 都是这样的生成核心——前者有丰富的联想和创造力，后者有强大的语言生成能力，但两者都缺乏内置的“工作记忆管理器”和“任务调度器”。（来源：ADHD 的 AI 工具全景）

## 同构解法：外部记忆 + 脚手架

解决思路也同构：把记忆和调度功能外包到外部系统。

### ADHD 侧：给大脑装一个外部硬盘

ADHD 患者的工作记忆容量有限，所以需要外部工具来“记住”任务上下文。例如 **Goblin Tools** 的 Magic ToDo 功能，能把“整理房间”这种模糊指令自动分解成“捡起地板上的衣服”“擦桌子”等具体步骤（来源：Goblin Tools）。这相当于把大脑的“执行功能”部分卸载到 AI 上——AI 负责分解、排序、提醒，用户只负责执行每一步。

**Saner.AI** 则直接针对“知识回忆”痛点，通过本地记忆存储和快速检索，减少 ADHD 用户频繁的搜索循环和标签切换（来源：Saner.AI）。这就像给大脑配了一个永不丢失的“第二记忆库”。

**Lex** 更进一步：用户只需一个指令，就能触发一系列复杂操作，比如“准备下周客户汇报”自动拆成“收集数据→做图表→写讲稿→排练”（来源：Lex）。这相当于给大脑装了一个“宏命令”，把多步骤任务压缩成一次启动，降低启动门槛。

### LLM/Agent 侧：给模型装一个外部记忆体

LLM 本身没有跨会话状态，所以 Agent 架构必须引入外部记忆——向量数据库、会话历史存储、知识图谱等。例如，一个客服 Agent 需要记住用户之前提过的问题、订单状态、偏好设置，否则每次对话都像第一次见面。这跟 ADHD 患者用 Saner.AI 记住自己写过的东西、用 Mem 存储零散笔记，本质是一样的——**把“上下文”从脆弱的内部缓存转移到可靠的外部存储**。

工程师们做的“Harness”或“编排层”，其实就是给 LLM 提供执行功能：任务分解（类似 Goblin Tools 的 Magic ToDo）、状态管理（类似 Mem 的记忆持久化）、多步骤流程控制（类似 Lex 的单指令触发）。**ADHD 患者需要的“脚手架”工具，和 LLM Agent 需要的“编排框架”，在架构上是同构的。**（来源：ADHD 的 AI 工具全景）

## 脚手架 vs 拐杖：边界在哪里？

这里有一个必须直面的争议：**这些工具到底是帮助用户成长的“脚手架”，还是让用户能力退化的“拐杖”？**

脚手架的设计理念是：提供临时支撑，待用户能力提升后可以逐步撤除。但当前多数 ADHD 工具（如 Goblin Tools、Saner.AI）设计为长期使用，没有提供撤除机制（来源：矛盾与存疑）。同样，LLM Agent 的编排层一旦依赖外部记忆，模型本身的能力是否会被“架空”？如果 Agent 永远靠外部数据库才记得住上下文，那模型自身的上下文理解能力是否会退化？

我的判断是：**脚手架与拐杖的关键区别在于，用户是否有意识地使用工具来训练内在能力，还是无意识地依赖工具替代内在能力。** 例如，用 Goblin Tools 分解任务时，如果用户同时学习分解方法，最终能自己分解，那就是脚手架；如果永远依赖工具分解，那就是拐杖。同样，工程师设计 Agent 时，如果外部记忆只是缓存，模型仍需自己处理上下文，那就是脚手架；如果外部记忆完全替代了模型的理解，那就是拐杖。

## 争议与局限

需要诚实指出，ADHD 大脑与 LLM 同构的命题目前主要基于概念类比和工具案例，缺乏大规模实证研究（来源：矛盾与存疑）。此外，ADHD 的异质性很高，单一工具难以覆盖所有亚型（来源：ADHD 的 AI 工具全景）。长期效果也缺乏随机对照试验验证（来源：矛盾与存疑）。

但即便如此，**同构视角提供了一个非常实用的工程化思维：用设计 Agent 的方法来设计自己的执行功能系统，或者反过来，用 ADHD 患者踩过的坑来优化 Agent 架构。** 两者互相启发，比单独研究更有价值。
