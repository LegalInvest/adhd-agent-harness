# Session Lifecycle Policy

更新时间：2026-07-03

## 核心定义

每一条 session 都是一次性、可报废的 Agent 工具人。它的生命周期上限按上下文窗口约 50% 处理；超过这个阶段，必须优先沉淀资产、压缩状态、交接或归档，而不是继续把复杂上下文堆在当前会话里。

## 判断标准

一个 session 是否完成，不看它“聊了多久”，只看它是否给项目留下了可恢复资产：

- 是否推进了一个明确任务。
- 是否留下了本地文件增量。
- 是否更新了项目状态或交接说明。
- 是否提交并推送 GitHub，或留下失败审计。
- 是否让下一个 Agent 可以无损继续。

满足以上条件后，该 session 可以被删除、归档或停止续用。

## 生命周期阶段

| 阶段 | 上下文占用 | 行为 |
|---|---:|---|
| 启动 | 0-20% | 读取项目入口、确认当前目标、避免重复劳动 |
| 执行 | 20-50% | 做出资产增量：文件、数据、报告、脚本、映射、状态 |
| 压缩 | 50% 左右 | 停止扩散，写交接、更新状态、提交 Git |
| 退出 | 50% 以后 | 只做收尾；不再开启大新任务 |

## 必须留下的记录

每个有效 session 结束前至少留下其中三项：

- `PROJECT_STATE.json` 或项目状态文件更新。
- `NEXT_ACTIONS.md` 更新。
- `handoff/` 交接文件更新。
- 新增或更新的研究、内容、脚本、数据文件。
- Git commit。
- GitHub push。
- 如果 push 失败，留下失败原因和恢复路径。

## 不要做

- 不要把 session 本身当记忆库。
- 不要依赖聊天窗口保存关键上下文。
- 不要在 50% 后继续大规模发散。
- 不要只口头总结，不落盘。
- 不要把没有文件增量的长对话当项目推进。

## 对后续 Agent 的要求

后续 Agent 接手时，应默认前一个 session 已经报废，只信项目文件和 Git 历史。先读：

1. `HANDOFF_FOR_NEXT_AGENT.md`
2. `PROJECT_STATE.json`
3. `NEXT_ACTIONS.md`
4. `SESSION_LIFECYCLE_POLICY.md`
5. `handoff/`

然后选择一个能产生文件增量或真实回流的动作执行。
