# Content Pipeline

本项目的工作流不是“想标题，然后写文章”，而是一个反应堆到现金流的变电系统。

## 1. 倾倒区

输入可以非常粗糙：

- 聊天片段
- flomo 碎片
- 社交媒体评论
- 读书或听书触发
- 情绪爆点
- 工作流失败现场

Agent 不要求作者先整理，先接住。

补充规则：作者每天都会有新的 ADHD / AI / Agent 思考，这是项目燃料，不是异常。Agent 默认按 `docs/workflows/每日灵感接入与周末发布协议.md` 接入：平日接住、去重、编号、补正文或模板；周末统一协助准备微信公众号草稿箱排期材料。

## 2. 发酵池

把倾倒内容标成：

- 金句
- 个人故事
- 机制解释
- 可验证例子
- 可产品化需求
- 可发布标题

旧文件 `发酵池.md` 可作为历史素材，但当前状态以 `PROJECT_STATUS.md` 和 `indexes/` 为准。

## 3. 草稿池

正文文件命名：

```text
问题NNN_主标题_副标题.md
```

正文结构默认使用五步：

1. 收敛：一句话打穿读者痛点。
2. 穿透：解释背后的机制。
3. 铁证：给例子、经验、研究或反事实。
4. 决策：告诉读者怎么重新看待自己。
5. 落地：给一个可执行动作。

## 4. 发布池

每篇正文优先拆成多个接口：

- 公众号长文
- 小红书短文
- 知乎回答
- 短视频口播
- 社群讨论引导

发布包输出到 `delivery/`。

发布节律：作者计划每个周末陆续上传到微信公众号草稿箱，并设置下一周定时发布。平日新增灵感不等于当天必须发布；平日负责蓄水和加工，周末负责排期和上架。

## 5. 回流池

每次发布后，必须把真实世界的反应写入 `feedback/feedback_log.md`。

没有反馈记录，就不能把“我觉得这个选题好”当成市场验证。

## 6. 产品化池

当某个选题连续触发评论、私信或强共鸣时，沉淀为：

- 一个可执行清单
- 一个 Agent 提示词
- 一个轻量工具
- 一个服务报价
- 一个社群主题
