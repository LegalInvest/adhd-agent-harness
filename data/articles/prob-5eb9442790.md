# 番茄钟对 ADHD 成功率有限，prompt 重锚对 LLM 几乎总有效——这个类比让工程师低估了人的难度吗？

> 一位 agent 工程师在读完本站的重锚定系列后,提交了一个反向的担忧——不是质疑类比对 ADHD 者不公,而是质疑它对**工程师**有误导:「我给 agent 的上下文里重注一次目标,它立刻回到任务上,重锚成功率接近 100%;而文献和真实经验都说,番茄钟这类重锚策略对 ADHD 的长期依从率很低(多少人的番茄钟 app 活不过三周)。如果同构成立,成功率为什么差这么远?我担心这个类比会让我们这些工程师产生幻觉:『重锚不就是解决了吗?人怎么会做不到?』——**类比是否系统性地隐藏了人类意志力的脆弱性,让做工具的人低估真实世界的难度,然后做出一堆『理论上有效』的产品?**」这个问题问到了同构叙事的一个真实软肋——而答案要从「重锚在两个系统里的成本结构」里找。

收敛:本文只回答——**成功率差距的机制分解(重锚这个动作在两个系统里的成本结构差在哪)?「意志力脆弱性」这个表述哪里对哪里误导?工程师(以及工具设计者)应该从这个差距里学到的三条设计军规?**

## 穿透:给 agent 重锚,执行者是工程师;给自己重锚,执行者是那个正需要被锚的系统——这是全部差距的根源

机制分解。先精确化「重锚」:把目标重新注入系统的注意焦点。对 LLM,这个动作的**执行者在系统外部**(工程师或编排代码把目标字符串塞进上下文)——注入是无条件的、零成本的、必然发生的;对人,重锚的执行者是**系统自己**——「到点了,该回到任务上」这个动作需要:听见提醒(注意)、中断当前吸引物(抑制——恰是弱项)、忍受切换的不适(情绪调节)、重新点火(启动——又是弱项)。**同一个「重锚」,在机器侧是外部写入,在人侧是一整套内部执行功能的调用——而后者正是这个系统本来就短缺的东西。**所以成功率差距根本不是「意志力脆弱 vs 机器可靠」,而是**同一功能的两种实现拓扑:外部强制 vs 自我引导**——不可微篇的公式再次命中:agent 靠架构,人靠意愿;意愿贵,架构便宜。

「意志力脆弱性」哪里对哪里误导:说「人的重锚成本高」——对;但「意志力脆弱」这个框架**把成本记错了科目**——它暗示问题出在「执行重锚的力量不足」(练意志就好),而机制分解显示问题出在**重锚被设计成需要意志力**(拓扑错误)。证据恰好支持这个重新记账:同样是人类重锚,**当执行者移到外部时,成功率立刻上升**——body double 在旁边说一句「回来啦」(外部注入)远比自己听番茄钟铃声(自我引导)有效;紧急 deadline(外部强制)几乎人人锚得回来。**「人做不到重锚」是假命题,真命题是「人做不到自己给自己当锚」——差一个字,产品设计差一个世界。**

三条设计军规(给工程师/工具设计者):**①永远问「这个环节的执行者是谁」**——评估任何「理论上有效」的功能时,把每一步的执行者标出来:执行者是「用户在低状态下的自我引导」的环节,预期依从率打两折起——**你的 agent 系统里没有一个环节靠模型『自觉』,凭什么你的人类产品里到处都是?**②**把自我引导环节尽量改写为外部注入**——铃声(要求用户响应)不如强制介入(全屏接管/物理震动+屏幕变灰);「用户设定提醒」不如「系统观察到超时主动问」——每改写一个环节,依从率涨一截(低摩擦篇的捕捉侧原则,同一军规的另一面);③**用「三周后还在用吗」验收,不用「第一周有效吗」**——人对新工具的头两周有新鲜感供电(此时自我引导环节被兴奋补贴),第三周补贴退场,真实成本结构才显形——**所有「理论上有效」的尸体,都死在第三周;你的 A/B 测试跑多久?**

对 ADHD 读者的顺手收益:这篇分析再次确认了你早就体感的事——**别买「需要你自觉配合」的工具,别练「自己锚自己」的功法;把锚交给外部(人、强制介入、环境结构),不是软弱,是拓扑正确。**

## 验证

机制可测:同一人群、同一任务,对比「自我引导重锚」(铃声)与「外部注入重锚」(真人/强介入)的依从率——预测后者显著更高(现有 body double 与环境干预证据方向一致,GRADE 中)。可证伪:若两种拓扑的依从率无差,则「意志力脆弱」的传统框架反而成立——现有证据不支持这个方向。

## 决策

做什么(工程师):执行者审计;自我引导改外部注入;三周验收。做什么(用户):选强介入型工具,把锚外置。

不做什么:不要再造(或再买)「铃声响了请自觉回来」式产品;不要用 agent 的重锚成功率类推人类产品的预期效果(拓扑不同,数字不可迁)。

先做什么(设计者):把你产品的用户旅程摊开,数一数「依赖用户自我引导」的环节数——那就是你的流失预算。

## 边界

番茄钟类干预对 ADHD 的效果证据混杂(短期可用、长期依从率低的模式有支持,GRADE 中);「拓扑决定依从率」为机制推导+方向性证据(无专门对照研究,如实标注)。同构在此的正确用法恰是本文示范:**差异点比相似点信息量更大**——这正是 B 级同构的诚实使用方式。

## 今天就能试的 3 件事

1. (用户)把一个「自觉型」提醒改成强介入型。
2. (设计者)跑一次执行者审计。
3. (双方)检查你正在依赖的某个方法:第三周了吗?还活着吗?

本文服务于人生 Harness 金字塔的**执行层的设计哲学**:工程师给 agent 重锚时,从没指望过 agent 自觉——他们直接写进架构里;而人类的工具,却一直在要求用户「配合一下」——这个双标才是真正该被类比纠正的东西:**像尊重 agent 的机制一样尊重人的机制:不指望自觉,把可靠性做进结构里。**
