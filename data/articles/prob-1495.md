# Otter.ai 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> function calling 那篇讲了 Otter 当「任务注册器」:把会议里的承诺自动转成文本。这一篇讲会议本身。ADHD 的会议有一种独特的失败形态,不是没在听,是**听着听着,这场会在你脑内变成了另一场会**:讨论从议题 A 滑到闲聊再滑到议题 C,你的注意力全程在线,但「这场会要解决什么」早就丢了——散会时热热闹闹,该定的事没定。这不只是 ADHD 的问题(会议漂移是全人类的病),但 ADHD 对它几乎零抵抗力。而这恰好是 system prompt 重锚定最经典的应用场景:**多轮长对话,必须反复对齐初始目标,否则每一轮都在合谋着漂移。** Otter 在这里的角色,比记录员深得多——用对的话。

先看工程侧的对应。多轮对话是 LLM 漂移的重灾区:每一轮的回应都基于最近几轮的内容,初始目标的权重逐轮衰减——**对策是显式的目标重注入**:关键轮次重申「本次对话的目标是 X,已确认 Y,待定 Z」。会议就是人类的多轮对话,同样的衰减、同样的对策:**开场声明目标(注入)、中途拉回议题(重锚定)、结束确认产出(校对)**——这是好会议的三板斧,也是 ADHD 参会者最难徒手完成的三件事(工作记忆里同时持有议题和当下讨论,太贵了)。Otter 的转录+AI 摘要,恰好能把三板斧外包一半。误用三种:

**误用一:只当录音笔,不当锚(记录了漂移,没有阻止漂移)。** 最普遍的用法:开着 Otter,安心漂——反正有记录。**但转录是事后的,漂移是实时的:记录一场跑偏的会,得到的是一份跑偏的完美文档**。工程对应:日志记全了,执行还是失败的——observability 不替代控制。修法:把锚定做进流程——**开会前把「本会目标+必须定的三件事」写进日历邀请或开场说出来**(你主持的会),或至少写在自己面前的便签上(你参加的会);Otter 管记录,那张便签管方向,两个组件,别指望一个顶两个。

**误用二:实时转录屏当第二块屏幕刷(锚变成了新的分心源)。** Otter 的实时转录很好看——好看到有人整场盯着文字流,校对它的错别字、回看十分钟前的段落——**注意力从会议本身迁移到了会议的镜像上**(工具篡位的会议版)。修法:**实时屏只有一个合法用途:走神后的快速回锚**——发现自己漂了,瞟一眼最近三行转录,十秒内知道讨论到哪了,重新接入;其余时间,眼睛离开屏幕,看人。

**误用三:会后摘要不消费,锚链断在最后一环。** 散会即结束,AI 摘要没人看,行动项没人认领——**下一场会用二十分钟争论上一场会「到底定了什么」**(这是组织级的上下文丢失:每场会都从模糊的记忆重启)。工程对应:多轮对话不做阶段性总结固化,已确认的事项反复被重新讨论。修法(注册器篇的延伸):**会后五分钟,摘要过目、行动项入库、有争议的标注出来发给参会人确认**——把这五分钟做成会议的固定尾巴,一场会的产出才算真正落袋。你的 ADHD 让你比别人更需要这套流程,也让你有理由成为团队里推动它的人——这是把弱项倒逼成的制度红利。

## 边界

同构强度 B 级:多轮对话的目标衰减与重注入是真实的 LLM 工程议题,会议漂移与 ADHD 的工作记忆负担有文献基础,Otter.ai 无对照研究。声明:录音转录务必合规并征得同意(注册器篇讲过);另外锚定技术管得住会议结构,管不住会议过载——如果你的日历被会议塞满且多数与你无关,那是边界问题,该修的是接受邀请的手,不是开会的姿势。

## 今天就能试的 3 件事

1. **给明天的会写一张锚便签**:本会目标一行+必须定的事两三件,放在视线里。
2. **练一次「三行回锚」**:走神后瞟最近三行转录,十秒接回,别恋战文字流。
3. **给会议装尾巴**:散会后五分钟,摘要过目、行动项入库——今天起当成会议的一部分。

会议是最贵的多轮对话,而没有锚的多轮对话,结局早已写好——**开场把目标说出来,中途把话题拉回来,结束把结论钉下来**。机器负责记住一切,你只需要负责别让这一切白记。
