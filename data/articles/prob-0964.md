# Tiimo 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 Tiimo：它到底补上了哪一层执行功能？

你打开 Tiimo，设好“10:00-10:45 写报告”，然后45分钟过去了，你发现自己在刷手机。这不是 Tiimo 的错——它完美地告诉了你该做什么。问题在于：**你需要的是一个能不断把你拉回当前目标的系统，而不是一个只告诉你“现在该做什么”的时间表。**

这正是 ADHD 大脑与 LLM agent 面临的核心同构难题：两者都是**高产但缺乏可靠执行调度层的生成核心**（来源：AI 与 ADHD 的专注力）。ADHD 大脑创意发散、LLM 文本生成，但都容易在长程任务中发生**目标漂移**——ADHD 侧是分心走神或超聚焦跑偏，LLM 侧是上下文膨胀导致推理退化（来源：上下文工程）。

Tiimo 被设计为 ADHD 的时间管理工具，它提供结构化日程和视觉倒计时，试图补偿**时间盲**和**任务启动**困难（来源：AI 与 ADHD 的专注力）。但它的核心机制是“预先规划”，而非“实时重锚定”。这就像给 LLM 一个固定的 system prompt 却从不更新——初始指令再完美，随着上下文膨胀，agent 仍会偏离原始目标。生产级 agent 系统通过**外部记忆**（如向量数据库）和**检索增强**来将决策锚定在持久状态上（来源：无状态与外部记忆），而不是仅靠一次提示。

**Tiimo 之于 ADHD，就像固定 system prompt 之于 LLM——它提供了初始结构，但缺少动态重锚定机制。** 真正有效的工具应该像 agent harness 中的“重锚定 system prompt”：在任务执行过程中，持续将注意力拉回当前目标，并更新上下文。

## 重锚定：ADHD 与 LLM 的共同需求

ADHD 大脑的工作记忆容量有限且不稳定，导致跨会话任务难以持续（来源：无状态与外部记忆）。典型表现为：每天需要重新回忆项目上下文，否则容易偏离轨道。这正好对应 LLM 的无状态性——每次对话独立，模型不保留跨会话记忆（来源：无状态与外部记忆）。

解决方案在两边是同构的：ADHD 需要“永远在线”的外部记忆体来补偿内部工作记忆的波动（来源：无状态与外部记忆）；LLM agent 需要外部记忆（如向量数据库）和工具调用来将决策锚定在外部证据上（来源：无状态与外部记忆）。

Tiimo 提供了**时间锚点**（视觉倒计时），但缺少**目标锚点**——它没有机制在任务中途检查“你还在做正确的事吗？”相比之下，Focusmate 通过实时同伴问责提供外部结构和社会压力，弥补内在多巴胺调节不足（来源：Focusmate）。它的“AI-Matched Body Doubling”本质上是一种动态重锚定：每25分钟，另一个人的存在就迫使你重新确认当前目标。

## 脚手架 vs 拐杖：Tiimo 的边界

Tiimo 容易沦为拐杖——用户依赖它的时间块来启动，但一旦启动后，如果缺乏动态重锚定，仍可能漂移。真正的脚手架应该是**可逐步撤除的结构**。例如，RescueTime 通过自动记录时间使用，帮助用户识别注意力漂移模式（来源：RescueTime），从而逐渐培养内在的时间感知能力。而 Tiimo 的固定日程可能让用户永远依赖外部时间提示，无法内化时间管理能力。

**关键在于：工具是否提供了“元认知反馈”，而不仅仅是“执行指令”。** Tiimo 只给指令，不反馈执行结果；Focusmate 通过同伴的实时监督提供反馈；RescueTime 通过事后报告提供模式识别。后两者更接近脚手架。

## 争议与局限

必须诚实指出：AI 工具作为外部执行功能层的有效性证据不足，多数来自用户报告和概念类比，缺乏大规模对照实验（来源：矛盾与存疑）。过度依赖 AI 工具可能削弱内在时间感知能力（来源：矛盾与存疑）。个体差异也很大——同一工具对不同 ADHD 亚型效果迥异（来源：矛盾与存疑）。

此外，Brain.fm 等神经锁相工具虽宣称影响大脑状态，但针对 ADHD 的效果缺乏独立临床研究支持（来源：Brain.fm）。这些工具的“AI”标签有时掩盖了证据的薄弱。

## 今天就能试的行动

1. **将 Tiimo 与“重锚定提示”结合**：在每个时间块中途（如20分钟时），设置一个振动提醒，问自己：“我现在在做的，是我计划做的事吗？”这模拟了 agent 的上下文检查。
2. **尝试 Focusmate 的实时问责**：每天预约一个时段，体验外部结构如何动态重锚定注意力。注意观察是否比单独使用 Tiimo 更少漂移。
3. **使用 RescueTime 进行事后分析**：每周查看一次时间报告，识别你的目标漂移模式（比如总是在某类任务后分心）。这相当于 agent 的日志分析。
4. **自定义 Tiimo 的任务块**：在每个任务块内加入“检查点”子任务，例如“写报告”块拆分为“10:00-10:15 列出大纲，10:15-10:30 检查大纲是否对准目标”。这相当于给 system prompt 添加了阶段性约束。

Tiimo 不是坏工具——它是一个优秀的**时间结构提供者**。但如果你只把它当时间表用，你就错过了它的真正潜力：成为你大脑的**重锚定 system prompt**。关键在于，你要主动为它添加动态检查机制，让它在任务执行中不断问你：“我们还在正轨上吗？”
