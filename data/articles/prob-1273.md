# Perplexity 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> Perplexity 对 ADHD 用户是个甜蜜陷阱式的礼物:问一句,直接给带引用的答案——省掉了搜索引擎那十个蓝链接和随之而来的四十分钟标签页漫游。作为「检索调用」的封装,它把 ADHD 最危险的操作之一(打开一个充满钩子的搜索结果页)变成了相对无菌的一问一答。但检索类调用恰恰是 function calling 家族里误用花样最多的一支——**工程师踩过的每个坑,这里都有人类版本。**

先立正确基线。工程上,检索调用的健康形态:**带着明确的信息需求发起→拿到结果→提取所需→回到主任务**——检索是子程序,主任务是调用方,控制权必须归还。Perplexity 用对的样子(上下文工程篇讲过):一句话问题进,答案落地,立即离场。三种误用:

**误用一:无终止条件的递归调用。** 工程版:检索结果里出现新概念,模型对新概念再发起检索,再出现新概念,再检索——没有深度限制的递归,栈溢出。人类版:查「怎么申请签证」→答案里提到「申根区」→顺手问「申根区历史」→提到某条约→问条约——**四十分钟后你在读欧洲一体化史,签证表格还空着**。Perplexity 的相关问题推荐让这个递归几乎无摩擦。修法:**发起检索前写下那个「原始问题」(一张便签),每次想点下一问,先看便签——原始问题答了吗?答了就退栈。** 深挖的冲动进「问题停车场」,给它第二时间盒。

**误用二:用检索替代决策。** 工程版:模型面对该自己判断的问题,反复调用检索工具「再多找点信息」——本质是用调用推迟承诺。人类版:买哪个型号、选哪个方案、要不要接这个活——**第九次搜索「X vs Y 对比」的时候,你已经不是在收集信息,是在逃避决策**。ADHD 的决策困难(每个选项都在闪烁)让这个误用格外高发,而 Perplexity 的流畅让「再查一下」永远比「现在定」更容易。修法:**给决策型检索设调用上限——三次查询后必须给出当前最佳选择**;写下「如果查到什么才会改变决定?」,答不出来就说明信息已经够了,缺的是承诺。

**误用三:引用不核验,答案直接入账。** 工程版:检索返回的内容不校验来源就拼进输出——RAG 系统幻觉的经典来源。人类版:Perplexity 给的答案自带权威感(有引用!),但**它的引用有时对不上原文、有时来源本身就是内容农场**——你直接转述给同事/写进报告,错误就过户到你名下。对 ADHD 的额外风险:核验是典型的「无聊但重要」任务,正是最容易被跳过的那种。修法:**分级核验——闲聊级的答案随它去;要写进产出、要做决策依据的,点开至少一条引用看原文**。十秒钟,过滤掉大部分事故。

一个反向提醒收尾:这些误用的解法都不是「少用它」。**回到十个蓝链接的旧世界,对 ADHD 是更差的方案**——封装过的检索仍然是净收益,前提是给它装上终止条件、调用上限和抽查机制。工具没错,错的是不带协议的调用。

## 边界

同构强度 B 级:递归失控与检索依赖是真实的 agent 工程问题,ADHD 的决策困难与刺激驱动的信息漫游有文献基础,Perplexity 本身无对照研究。声明:若「查资料」已经成为一天的主要活动、且伴随明显的功能损害(工作停摆、睡眠侵蚀),那不只是工具用法问题,值得专业评估;信息渴求有时是焦虑的马甲,那一层需要被单独看见。

## 今天就能试的 3 件事

1. **下次检索前写便签**:原始问题一句话,答了就退,想深挖的进停车场。
2. **给手头那个悬而未决的决定设上限**:再查三次,然后按当前信息定,写下「什么信息才会改变决定」。
3. **抽查一次引用**:把你今天准备转述的一个答案,点开原文核对——感受一下过户前验货的必要性。

检索的封装省下的时间,别再还给检索本身——**答案是拿来用的,不是拿来续杯的**。带着问题进去,带着答案出来,门在身后关上,这才是调用的完整姿势。
