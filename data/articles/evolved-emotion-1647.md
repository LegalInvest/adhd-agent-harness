# 为什么用 Routinery 治 ADHD 的情绪失调，和给 agent 套 会褪去的脚手架 是一回事？

> Routinery 实测：同一套 harness 思路，ADHD 与 LLM 两边都成立。

## 问题：为什么 ADHD 的情绪失调和 LLM agent 的工程困难是同一回事？

想象你有一个超强的大脑，能瞬间产生无数创意，但就是没法按计划执行。或者你有一个强大的 LLM，能写出惊艳的文本，但一旦让它自主完成多步任务，它就迷失方向、重复循环、情绪崩溃（在 agent 语境下是输出混乱或死循环）。这就是 ADHD 大脑与 LLM/agent 的核心同构：**两者都是高产但缺执行调度层的生成核心**。

ADHD 的情绪失调，根源在于执行功能缺陷——工作记忆过载、任务启动困难、时间盲——这些缺陷让情绪像脱缰的野马。而 LLM agent 的工程困难，同样源于缺乏可靠的编排层：上下文窗口有限、任务分解不稳定、状态管理混乱。两者都需要一个外部脚手架来补偿调度层的缺失。

## 答案：会褪去的脚手架——从 Routinery 到 agent harness

Routinery 是一款针对 ADHD 的日常流程建立工具，它通过固定步骤、定时提醒和可视化进度，帮助用户建立稳定的执行模式。但它的高明之处在于：**它不是永久的拐杖，而是可褪去的脚手架**。用户逐渐内化流程后，可以减少对工具的依赖。这种设计思路，与 agent 工程中的「会褪去的脚手架」完全同构。

在 agent 工程中，我们给 LLM 套上 harness——即外部编排层，包括任务分解、状态追踪、错误恢复等。这个 harness 不是永久植入的，而是随着 agent 能力提升（或任务固化）逐渐褪去。例如，初期可能需要详细的任务清单和每一步的验证，后期可以简化为高层目标。

## 证据：两边都成立的同构

**ADHD 侧**：Routinery 通过外部化执行功能，直接补偿工作记忆缺陷。工作记忆缺陷导致情绪信息难以维持，AI 工具（如 Inflow）通过提醒、笔记和进度追踪，充当外部记忆载体（来源：AI 与 ADHD 的情绪调节）。Goblin Tools 的 Magic ToDo 功能将复杂任务分解为小步骤，降低启动门槛，从而减少因任务启动困难引发的焦虑和自责（来源：Goblin Tools）。Saner.AI 通过知识回忆减少搜索循环和标签切换，降低认知负荷（来源：Saner.AI）。这些工具的共同点：它们都是外部执行功能层，即脚手架。

**LLM/agent 侧**：给 agent 套 harness 的典型做法包括：
- **任务分解**：将复杂指令拆解为子任务，类似 Goblin Tools 的分解。
- **上下文管理**：通过外部记忆（如向量数据库）补偿有限上下文窗口，类似 Saner.AI 的本地记忆。
- **状态追踪**：记录执行进度，避免重复或遗漏，类似 Routinery 的进度条。
- **错误恢复**：当 agent 输出异常时，回退或重试，类似 ADHD 工具中的「崩溃容纳」功能。

这些 harness 组件与 ADHD 工具的功能一一对应。当 harness 设计为可逐步撤除时，agent 就能逐渐内化这些能力，就像 ADHD 用户逐渐内化 Routinery 的流程。

## 核心判断：脚手架 vs 拐杖的边界

关键区别在于：脚手架是**可褪去**的，拐杖是**永久依赖**的。Routinery 的设计允许用户逐渐减少使用频率，而很多 ADHD 工具（如 Inflow）声称通过神经科学原理改善执行功能，但缺乏撤除机制（来源：矛盾与存疑）。同样，agent harness 如果设计为永久固定，就会限制 agent 的自主性。

我的核心观点是：**真正的脚手架必须包含褪去计划**。没有褪去计划的脚手架就是拐杖，它可能削弱内在能力。证据显示，ADHD 工具可能削弱内在情绪调节能力，如何设计「可撤除的脚手架」尚无定论（来源：AI 与 ADHD 的情绪调节）。同样，agent harness 如果过于 rigid，会阻碍 agent 的泛化能力。

## 局限与争议

本文的同构命题——ADHD 大脑与 LLM 同构——主要来自概念类比和工具案例，缺乏大规模实证（来源：矛盾与存疑）。多巴胺干预的有效性也存在争议：基于多巴胺的 AI 干预可能强化依赖而非内在动机（来源：AI 与 ADHD 的情绪调节）。此外，现有工具（如 Routinery）的长期效果证据缺乏，多数研究为短期观察（来源：AI 与 ADHD 的情绪调节）。

## 今天就能试的行动

1. **用 Routinery 建立一条「褪去计划」**：在设置每日流程时，明确标记哪些步骤可以在一周后移除，哪些可以简化。例如，第一周使用详细步骤，第二周只保留关键节点。
2. **给 agent 套一个「最小 harness」**：在编写 agent 指令时，只添加必要的任务分解和状态追踪，并注明「如果连续成功 3 次，移除这条约束」。
