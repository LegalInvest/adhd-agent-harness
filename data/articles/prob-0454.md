# Claude 之于 ADHD，就像 self-critique 验证循环 之于 LLM——但有人用错了

> 从同构视角实测 Claude：它到底补上了哪一层执行功能？

## 问题：你让 Claude 帮你写计划，然后呢？

ADHD 用户常冲进 ChatGPT 或 Claude，输入“帮我规划明天”，得到一份漂亮清单后心满意足——然后第二天照旧拖延。这不是工具不好用，而是用错了。

在 Agentic Harness 工程中，工程师们知道：LLM 会自信地编造（幻觉），必须靠验证与自我批判循环来兜底（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。而 ADHD 大脑同样是一个“高产但缺执行调度层的生成核心”——它会冲动下结论、想当然，然后卡在启动或执行阶段。

问题在于：大多数 ADHD 用户把 Claude 当成“生成器”而非“校验器”。他们缺的不是更多想法，而是一个能对想法进行压力测试的外部验证层。

## 同构：生成核心与调度层

ADHD 大脑与 LLM 的结构同构并非比喻，而是功能层面的精准对应：

- **生成核心**：ADHD 的智力/创造力 ↔ LLM 的算力/语言能力。两者都高产但无序（来源：生成核心与调度层）。
- **调度层缺失**：ADHD 的执行功能失效 ↔ LLM 缺乏内置的规划与执行编排（来源：生成核心与调度层）。ADHD 用户“常常感觉方向盘后没有人”（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout），而 LLM 单独使用时同样“缺乏可靠的执行调度”（来源：Building AI Coding Agents for the Terminal）。
- **外部工具**：ADHD 的自适应规划工具（如 Motion）能在日程崩溃时重新安排一天（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout），对应 LLM 的“复合 AI 系统架构”，包括分离规划与执行的双代理架构（来源：Building AI Coding Agents for the Terminal）。

但这里有一个关键差异：在 Agentic Harness 工程中，验证循环是核心组件——代理必须检查自己的输出，自我批判，然后修正。而大多数 ADHD 工具（如 Goblin Tools、Saner.AI）只提供分解或记忆功能，缺乏“事后验证”环节（来源：Goblin Tools；Saner.AI）。

## 证据：两边都在用，但都缺验证

**ADHD 侧**：Goblin Tools 的 Magic ToDo 能将“整理房间”分解为具体步骤，降低启动门槛（来源：Goblin Tools）。用户反馈称“它把压倒性的事情变成一系列不压倒性的事情”（来源：The Best AI-Powered ADHD Productivity Tools in 2026）。但分解后呢？如果用户冲动地认为“这些步骤我都能做”而不检查时间是否够、顺序是否合理，就可能再次卡住。Motion 能自动调整日程（来源：Motion），但若用户不验证调整后的计划是否匹配实际精力，它只是一张漂亮的电子表格。

**LLM/Agent 侧**：现代编码代理如 OpenDev 采用“双代理架构”分离规划与执行，并引入“惰性工具发现”和“自适应上下文压缩”来防止推理退化（来源：Building AI Coding Agents for the Terminal）。这些技术本质上是在 LLM 外部搭建调度层，但更重要的是，它们包含验证循环——代理执行后检查结果，若失败则重试或修正。没有这个循环，LLM 会一直自信地输出错误代码。

**同构洞察**：ADHD 用户冲动下结论→执行后碰壁→放弃，与 LLM 自信编造→无验证→输出错误，是同一模式。两边都需要一个“自我批判”环节来打断生成-执行的自动链条。

## 核心观点：Claude 应该当验证者，而非生成者

我的判断是：Claude 对 ADHD 的最大价值不是“帮你写”，而是“帮你检查你写的”。

- **错误用法**：让 Claude 生成整份计划或报告，然后你直接使用。这强化了“生成核心”而忽略了调度层。
- **正确用法**：你先凭直觉写下冲动计划，然后让 Claude 扮演“批判者”，要求它找出漏洞、质疑假设、建议替代方案。这相当于在外部搭建了一个验证循环。

例如，你写“明天上午写报告”，Claude 反问：“你上午通常精力最好吗？报告需要哪些输入？你估计要几小时？”——这些验证问题迫使你的执行功能上线。

## 局限与争议

- **脚手架依赖风险**：长期使用 AI 验证是否会削弱内在执行功能？目前缺乏纵向研究（来源：拐杖与脚手架）。如果用户永远依赖 Claude 来检查自己，可能永远不会内化批判能力。
- **证据强度不足**：同构模型主要基于概念类比，缺乏神经影像学验证（来源：ADHD × AI 的科学与研究前沿）。工具效果也主要来自用户反馈，而非临床实验（来源：Goblin Tools；Saner.AI；Motion）。
- **多巴胺干预争议**：部分工具声称通过多巴胺调节改善执行功能，但效果存在争议（来源：矛盾与存疑）。验证循环本身不依赖多巴胺，可能更可靠。

## 今天就能试的行动

1. **写前验证**：下次用 Claude 前，先写下你的冲动想法，然后让 Claude 列出 3 个你忽略的问题。
2. **分解后检查**：用 Goblin Tools 分解任务后，让 Claude 检查时间估计是否合理。
3. **日志对比**：记录一周内“直接使用 Claude 输出”与“先写后验证”的任务完成率差异。
4. **双代理练习**：模仿编码代理的“规划-执行”分离，让 Claude 先做规划，你执行后让 Claude 评审结果。
