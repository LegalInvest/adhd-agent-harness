# Goblin Tools 之于 ADHD，就像 会褪去的脚手架 之于 LLM——但有人用错了

> 从同构视角实测 Goblin Tools：它到底补上了哪一层执行功能？

## 问题：怕依赖，也怕崩溃

你打开 Goblin Tools 的 Magic ToDo，输入“整理房间”，AI 立刻吐出十个小步骤。那一刻你松了口气——终于不用自己扛了。但紧接着，一个声音冒出来：我是不是越来越离不开它了？没有它，我还能启动吗？

这种两难，ADHD 人群太熟悉了。怕过度依赖工具，又怕没有支撑就崩。同样的焦虑，也出现在 LLM agent 工程师的脑海中：我们给模型套上的 harness，到底是会褪去的脚手架，还是拔不掉的拐杖？

## 同构：两个“生成核心”共享的脆弱

ADHD 大脑与 LLM 共享一个核心结构：强大的生成能力（智力/算力）加上不可靠的调度层（执行功能/编排）。ADHD 的执行功能缺陷——工作记忆不足、任务启动困难、时间盲——导致大脑的输出波动剧烈，就像 LLM 的采样温度（temperature）过高时输出随机、不可控（来源：采样温度与表现波动）。

LLM agent 的工程解决方案是 harness：一个外部调度层，负责工具调用、状态管理、确定性控制流（来源：AI Agent Systems: Architectures, Applications, and Evaluation）。工程师们强调“确定性状态转换”和“计划-执行分离”，来驯服 LLM 的随机性（来源：Planner-Executor Agentic Framework）。

ADHD 的解决方案，在结构上完全同构：AI 工具充当“外部执行功能层”，补偿工作记忆、任务启动和时间盲的缺陷（来源：AI 与 ADHD 的情绪调节）。Goblin Tools 的任务分解，本质上就是一个 harness——它把模糊意图转化为确定性的步骤序列，降低启动门槛。

## 证据：两边的 harness 都在工作

ADHD 侧：Goblin Tools 被用户评价为“将压倒性的事情变成一系列不压倒性的事情”（来源：The Best AI-Powered ADHD Productivity Tools in 2026），其 Magic ToDo 功能通过 NLP 将模糊任务分解为可管理步骤（来源：Goblin Tools）。Inflow 则通过个性化训练强化背外侧前额叶皮层的工作记忆功能（来源：Inflow）。Saner.AI 通过本地记忆和快速检索减少搜索循环（来源：Saner.AI）。这些工具都降低了认知开销，让用户能启动、维持、完成任务。

LLM/agent 侧：Harness 通过 function calling 将复杂任务拆解为工具调用，强调“agent UX”——命名、模式、错误处理等接口质量（来源：工具使用与认知卸载）。现代 agent 系统“组合多个模型、检索器和工具”来达成 SOTA 结果（来源：Building AI Coding Agents for the Terminal）。确定性控制流和计划-执行分离，确保了 agent 行为的可靠。

两边都在做同一件事：为不稳定的生成核心加装一个外部调度层。

## 判断：脚手架 vs 拐杖的边界

但这里有一个关键分歧。工程上的 harness 被设计为可替换、可降级的：你可以换一个编排器，可以降级到纯规则。而 ADHD 侧的 AI 工具，几乎全部设计为长期使用，没有撤除机制（来源：矛盾与存疑）。

这就是“用错了”的核心：Goblin Tools 应该是一个会褪去的脚手架，但被当成了永久拐杖。

脚手架与拐杖的边界在于：脚手架训练你下次自己搭，拐杖让你永远依赖。当前工具缺乏“撤除路径”——没有哪个工具会告诉你“用四周后试着减少使用频率”。Inflow 声称通过神经可塑性改善工作记忆（来源：Inflow），但它的训练本身就需要工具介入，撤除后效果能维持多久？没有证据。

更根本的问题：多巴胺干预的争议。一些工具通过奖励系统调节多巴胺水平，但“基于多巴胺的干预可能强化依赖而非内在动机”（来源：AI 与 ADHD 的情绪调节）。如果工具让你感觉良好，你更难离开它。

## 局限与诚实

必须承认，ADHD 大脑与 LLM 的同构命题“证据主要来自概念类比和工具案例，缺乏大规模实证”（来源：矛盾与存疑）。Goblin Tools 的效果也“缺乏独立随机对照试验”（来源：Goblin Tools）。我们基于结构相似性提出的脚手架-拐杖判断，是一个理论框架，而非定论。

但框架的价值在于指导设计。如果工具开发者开始考虑“撤除机制”，如果用户开始问“我什么时候可以不用它”，那么脚手架就正在发挥作用。

## 今天就能试的行动

1. **用 Goblin Tools 分解一个你拖延已久的任务**，然后尝试在不打开工具的情况下，自己回忆并写出步骤。对比两个列表的差异——这是你的“撤除练习”。
2. **设置一个“工具使用到期日”**：比如用 Inflow 训练两周后，停用三天，记录你的执行功能变化。这能帮你判断工具是脚手架还是拐杖。
3. **给 agent 工程师的提议**：在你设计的 harness 中，加入一个“降级模式”——当 LLM 输出足够稳定时，自动减少 harness 的干预。这不仅是工程优化，更是对用户自主性的尊重。
