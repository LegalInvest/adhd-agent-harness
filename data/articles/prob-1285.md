# Goblin Tools 之于 ADHD，就像 planner-executor 任务分解 之于 LLM——但有人用错了

> agent 架构里,planner-executor 是最经典的分工:一个组件负责把大目标拆成步骤序列(planner),另一个负责逐步执行(executor)——分开的理由很硬:规划和执行需要的「认知模式」不同,混在一起,两个都做不好。Goblin Tools 的 Magic ToDo 恰好就是一个外置 planner:你当 executor,它管拆解。这个架构对 ADHD 是对症的——但 planner-executor 在工程史上有一整章「失败模式」,而每一种,都有人正在用 Magic ToDo 复刻。

先说架构为什么对症。ADHD 的困难高度不对称:**执行单个明确的小步骤,通常没问题;把模糊大任务拆成小步骤,是重灾区**(这需要工作记忆里同时持有全局和局部,恰恰是最贵的操作)。所以「拆解外包+执行自留」的分工方向完全正确。问题出在接口和时序上——四种误用:

**误用一:一次性全量规划(plan-then-execute 的脆性)。** 工程教训:planner 一口气生成五十步的完整计划,executor 执行到第七步发现现实和计划不符——后面四十三步全部作废,重规划的成本让系统直接放弃。人类版:把一个月的项目一次拆成八十个子任务,**看着那张吓人的清单,启动能量当场清零**——过度规划的产出不是路线图,是一堵新的墙。修法是工程界的共识:**滚动规划(rolling planning)——只详细拆「下一步可执行的部分」,远处的保持粗粒度**;Magic ToDo 只拆今天要动的那一块,其余留待临近再拆。

**误用二:粒度错配。** planner 拆得太粗(步骤仍需现场展开=没拆)或太细(每步琐碎到侮辱智商,清单长到超载)。人类版两个都常见:拆出「完成调研」这种伪步骤,或者把回封邮件拆成九步。**正确粒度的判据只有一个:每一步是否小到「看见就能直接动手」,同时大到「做完有真实进展感」**——Magic ToDo 有辣度(拆解深度)可调,这个参数应该按任务和当天状态调,不是设一次用终身。

**误用三:executor 越权重规划。** 工程版:执行组件不信任计划,每步都自行改道——规划形同虚设,系统退化回无 planner 的混乱。人类版:拆好了步骤,执行第一步时觉得「其实应该先做那个」,当场重排——**每次执行都变成新一轮规划,而规划恰恰是你要外包出去的高耗能操作**。修法(重排窗口的思路通用):执行时段内,计划只读;对计划的不满写下来,批量提交给下一次规划时段。**规划与执行的时间边界,就是这个架构的全部纪律。**

**误用四:规划成为表演(planning theater)。** 最隐蔽的一种:拆解做得漂亮、清单赏心悦目、颜色标签齐全——**但系统的产出是清单本身,不是完成的任务**。工程对应:planner 疯狂产出计划文档,executor 空转。判据冷酷而简单:**看 executor 的吞吐量——这周完成了几步?** 规划的一切精致,只有被执行消费了才算数。发现自己在第三次美化同一张清单时,强制切换:立刻执行清单上最小的一步,用行动打断表演。

## 边界

同构强度 B 级:planner-executor 及其失败模式是真实的 agent 架构议题,ADHD 的规划-执行不对称有执行功能文献支撑,Magic ToDo 无对照研究,四种误用是功能映射。声明:拆解工具优化的是流程层;若拆到最小步骤依然普遍启动不了,瓶颈在启动的神经与情绪层(见本系列相应篇章),或需要专业评估——planner 再好,也替代不了给 executor 供能的那套系统。

## 今天就能试的 3 件事

1. **改用滚动规划**:手头项目只拆「明天要动的部分」,远处保持三行粗纲。
2. **校准一次粒度**:把现有清单过一遍,合并侮辱智商的、再拆看见就头皮发麻的。
3. **量一次吞吐**:本周计划了多少步、完成了多少步?比值低于三成,先停止规划一天,只执行。

planner 和 executor 分开,是为了让两边各干最擅长的事——**不是为了让规划变成新的逃避所**。计划的价值以「被执行的步数」计价,其余都是装订精美的拖延。
