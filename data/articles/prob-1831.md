# ADHD 创业者视角：为什么「治好 ADHD 的冲动下结论、想当然」和「让 LLM 不跑飞」其实是同一道工程题？

> LLM 自信地编造（幻觉），靠验证与自我批判兜底——同一套 harness 思路，两边都成立。

## 问题：冲动下结论与 LLM 幻觉——同一类故障

如果你有 ADHD，你一定经历过这个场景：别人刚说半句话，你已经在心里把结论画好，甚至开始反驳了。事后发现，你根本没听完，也理解错了。这就是冲动下结论——大脑在信息不足时，凭借模式匹配“脑补”出一个完整故事，然后自信地输出。

与此同时，工程师们正在为另一个问题头疼：LLM 在回答时，会自信地编造事实、引用不存在的论文、生成看似合理但完全错误的代码。这就是幻觉。两者共享同一底层机制：一个高产的生成核心，在缺乏外部验证循环时，会优先选择“流畅的编造”而非“费力的真实”。

这让我想起一个 ADHD 创业者朋友的话：“我的大脑就像一台 24 小时运转的发电机，但开关和变压器都坏了。” 这句话，同样可以描述一个裸奔的 LLM。

## 同构：高产但缺调度层的生成核心

**ADHD 一侧**：执行功能被描述为“大脑的驾驶座”，但对 ADHD 来说，“常常感觉方向盘后没有人”（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout）。工作记忆是另一个执行功能瓶颈（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout），导致上下文丢失、计划难以执行。传统工具如计划本和便签“用了一周就崩溃了”（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout），因为它们无法解决时间盲和任务启动困难。

**LLM/agent 一侧**：LLM 本身是无状态、仅生成文本的核心，需要外部“脚手架”和“harness”来提供工具接口、上下文管理、安全执行等编排能力（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。现代 AI 编码代理通过“复合 AI 系统架构”来弥补这一缺陷，包括“工作负载专用模型路由、分离规划与执行的双代理架构、惰性工具发现、自适应上下文压缩”（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。这些技术本质上是在 LLM 外部搭建调度层，防止“上下文膨胀和推理退化”（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。

**同构对应**：ADHD 大脑的智力/创造力 ↔ LLM 的算力/语言能力——两者都高产但无序；ADHD 的执行功能失效 ↔ LLM 缺乏内置的规划与执行编排——两者都需要外部脚手架；ADHD 的“自适应计划工具如 Motion 和 Tiimo，能在日程崩溃时重新安排一天”（来源：AI Tools for ADHD: Boosting Productivity and Reducing Burnout）↔ LLM 的上下文压缩与惰性工具发现——都是外部调度机制。

## 解法：外部验证与自我批判循环

**ADHD 侧**：工具如 Goblin Tools 的 Magic ToDo 功能能自动将任何任务分解为更小、更易管理的步骤（来源：AI Tools for Kids with ADHD: A Complete Guide for Parents...），从而降低启动门槛。它“将压倒性的事情变成一系列不压倒性的事情”（来源：The Best AI-Powered ADHD Productivity Tools in 2026 (That ...）。Saner.AI 通过增强知识回忆，帮助用户快速找回信息，减少认知负荷（来源：Best AI Tools for ADHD Productivity in 2026 (Honest Review) - Iwo Szapar）。Motion 通过自动排程与动态调整，消除“下一步该做什么”的决策负担（来源：11 Best ADHD Productivity Apps for Fluctuating Energy - rivva blog）。这些工具本质上是外部执行功能层——为 ADHD 大脑搭建验证循环：分解任务（验证可行性）、回忆上下文（验证连贯性）、自动排程（验证优先级）。

**LLM/agent 侧**：Harness 工程被定义为“设计围绕 AI 代理的脚手架——上下文交付、工具接口、规划工件、验证循环、记忆系统和沙箱”（来源：GitHub - ai-boost/awesome-harness-engineering: Awesome list for AI agent harness engineering: tools, patterns, evals, memory, MCP, permissions, observability, and orchestration. · GitHub）。具体实现包括：用 Git 仓库存储项目上下文（类似 ADHD 侧的 Second Brain），通过 MCP 连接器访问外部数据（来源：Worker Agents | Harness Developer Hub），以及将控制逻辑外化为可移植的自然语言工件。验证循环是关键：agent 生成计划后，需通过外部工具（如代码执行器、搜索 API）验证结果，若发现错误则自我批判并修正。

**核心判断**：ADHD 工具与 LLM harness 是同一类工程——为高产生成核心搭建外部调度层。这不是替代大脑或模型，而是提供“第二大脑”或“外部驾驶座”。正如 ADHD 用户需要“AI 第二大脑”来承载项目上下文（来源：Best AI Tools for ADHD Productivity in 2026 (Honest Review) - Iwo Szapar），LLM 也需要 Git 仓库和 MCP 来承载上下文。

## 争议：脚手架还是拐杖？

然而，我们必须诚实面对争议。现有工具的设计多假设用户能维持长时间专注、记住昨天的问题并执行计划，但实际用户往往做不到（来源：Best AI Tools for ADHD Productivity in 2026 (Honest Review) - Iwo Szapar）。更关键的是，[[拐杖与脚手架]]概念强调“脚手架应设计为可逐步撤除的临时支撑”，但多数工具（如 Goblin Tools、Saner.AI）设计为长期使用，未提及撤除机制（来源：矛盾与存疑）。长期依赖是否阻碍内在能力发展？目前缺乏纵向研究。

同样，ADHD 大脑与 LLM 同构命题的证据主要来自概念类比和工具案例，缺乏大规模实证（来源：ADHD × AI 的科学与研究前沿）。多巴胺干预的有效性也存在争议（来源：矛盾与存疑）。我们需要警惕：将理论框架当作事实，可能过度简化问题。

## 结语

ADHD 创业者与 LLM 工程师面对的，是同一道工程题：如何为一个高产生成核心，搭建可靠的外部调度与验证层。这不是巧合，而是认知架构的深层同构。当你下次感到冲动下结论时，不妨想想 LLM 的幻觉——然后给自己加一个验证循环。这，就是 harness 的力量。
