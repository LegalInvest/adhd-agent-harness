# 如果把 ADHD 大脑当成生成核心，Routinery 能不能当它的外部执行功能层？

> 用 agent 工程的思路重新理解 Routinery 对 ADHD 的真正价值。

想象一下：你有一个强大的生成核心——无论是 GPT-4 还是 ADHD 大脑——它能在瞬间冒出无数创意、计划和解决方案。但当你试图让它执行一个具体任务时，它要么忘记上下文，要么被新输入冲散注意力，要么干脆拒绝启动。这正是 ADHD 与大型语言模型（LLM）的结构同构：两者都是高产但缺乏内建执行调度层的生成核心（来源：ADHD × AI 的科学与研究前沿）。

Routinery 是一款专为 ADHD 设计的例程应用，它通过固定时间表强制用户按顺序完成任务。但它的价值远不止“提醒”或“计时”——在 agent 工程的视角下，Routinery 扮演的是外部执行功能层的角色：它提供了编排（task orchestration）、记忆（routine persistence）和工具调用（启动提示），正好对应 LLM/agent 系统中缺失的 harness 层。

## 问题：ADHD 大脑的“无状态”与“上下文溢出”

ADHD 的核心执行功能缺陷包括工作记忆差、时间盲和任务启动困难（来源：执行功能）。工作记忆差对应 LLM 的无状态性：模型无法记住对话历史，除非显式保存；ADHD 大脑也无法在脑中同时保持多个任务状态。时间盲则对应 LLM 的上下文窗口溢出：当任务链过长，大脑会丢失时间线索，就像模型忘记早期输入。任务启动困难则源于缺乏外部触发——LLM 需要 prompt 才能生成，ADHD 大脑也需要外部“启动信号”。

传统计划工具（如纸质清单）无法解决这些缺陷，因为它们要求用户自己管理上下文和时间感知（来源：时间盲）。而 AI 工具通过适应性和实时反馈，能更贴合 ADHD 大脑对新颖性和即时奖励的需求（来源：时间盲）。

## 答案：Routinery 作为外部编排层

Routinery 的核心机制是“固定例程”：用户预设一系列任务（如“起床→刷牙→吃药→早餐”），应用按顺序自动推进，每个任务有固定时长，完成后自动进入下一步。这本质上是一个硬编码的 agent harness：它接管了编排（决定下一步做什么）、记忆（记住当前进度）和工具调用（通过通知触发行动）。

这种设计直接针对 ADHD 的两大痛点：
- **消除决策负担**：Motion 等自适应规划器通过 AI 自动排程来减少“下一步做什么”的决策（来源：Motion），但 Routinery 更进一步，将决策降低到零——你只需跟随。
- **提供外部时间锚点**：Tiimo 专为时间盲设计，用视觉时间表帮助大脑从“无限延迟”转向“开始行动”（来源：时间盲）。Routinery 的倒计时与进度条起到同样作用，将抽象时间转化为具体信号。
- **降低启动门槛**：Goblin Tools 通过任务分解将压倒性任务变成可管理步骤（来源：Goblin Tools）。Routinery 的固定顺序也起到了分解作用：你不需要思考“先做什么”，因为顺序已定。

从 agent 工程角度看，这类似于为 LLM 设计一个“工作流引擎”：输入（起床闹钟）触发一个预定义 pipeline，每一步的输出（刷牙完成）自动激发下一步。这正是外部执行功能层的定义——AI 工具作为外部脚手架，提供记忆、任务启动和时间管理，弥补内在调度缺陷（来源：外部执行功能层）。

## 同构证据：ADHD 与 LLM 共享的“外部脚手架”需求

ADHD 侧的证据：多项来源指出，身体在场效应（如 Focusmate 的虚拟伙伴）能显著提升专注度，因为社会存在感提供隐性问责（来源：身体在场效应）。Routinery 的固定例程相当于一个“数字身体在场”——应用本身成为那个“在场的人”，通过结构化的推进提供问责。

LLM/agent 侧的证据：当前最成功的 agent 框架（如 LangChain、AutoGPT）都依赖外部编排层：记忆模块存储上下文，工具层调用 API，规划器决定下一步。没有这些，LLM 会陷入循环或遗忘。这正是 ADHD 大脑在没有 Routinery 时的状态——反复检查手机、忘记要做什么、被新通知打断。

## 脚手架 vs 拐杖：Routinery 的边界

但这里有一个关键争议：Routinery 是脚手架（可逐步撤除的临时支撑）还是拐杖（长期依赖的替代品）？（来源：拐杖与脚手架）。目前 Routinery 的设计并未提供撤除机制——它假设用户需要永久的结构化支持。这与 ADHD 工具领域的普遍问题一致：多数工具（如 Goblin Tools、Saner.AI）设计为长期使用，未考虑如何培养内在能力（来源：矛盾与存疑）。

我的判断是：Routinery 更适合作为“初始脚手架”，尤其对于执行功能严重受损的用户。但理想的工具应包含“渐进式撤除”功能——例如，随着用户成功完成例程的次数增加，自动延长任务间隔或减少提醒频率。目前缺乏这样的设计，也缺乏纵向研究验证长期使用是否会削弱内在执行功能（来源：矛盾与存疑）。
