# Goblin Tools 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> Goblin Tools 大概是 ADHD 社区口碑最好的免费工具:Magic ToDo 把任务拆成小步、Formalizer 把暴躁的话改成得体邮件、Judge 帮你判断语气——像一排随叫随到的小函数。这个类比本身成立:它就是给人用的 function calling,把「现场要动脑的操作」变成「一键调用的封装」。但正因为它太好用、太顺手,它的错误用法也格外隐蔽——**function calling 的工程史上,每一种著名的误用,都能在 Goblin Tools 用户身上找到对应版本。**

先立正确基线:function calling 的设计意图是**把高频、可标准化的操作封装起来,让核心把稀缺算力花在真正需要判断的地方**。对应到 Goblin Tools:把「拆解」「措辞润色」这类执行功能税外包出去,把你有限的启动能量留给做事本身。用对的样子:卡在一个大任务前→丢进 Magic ToDo→拿到步骤→**立刻开始第一步**。调用是过路的桥,不是逗留的地方。三种误用,按工程史对号:

**误用一:过度调用(over-calling)。** agent 开发的经典病:模型明明自己能答,却每一步都调用工具——延迟翻倍、成本翻倍、还引入了不必要的失败点。人类版:**「刷个牙」也丢进 Magic ToDo 拆成五步**。拆解工具的适用域是「卡住的、看不清结构的任务」;对本来就能直接做的事,调用本身变成了新的拖延仪式——你在用「准备做」的动作,推迟「开始做」的时刻,而且做得心安理得,因为它看起来像生产力。**判据:拆完之后你动了吗?拆了不动,连续三次,说明你在过度调用。**

**误用二:调用结果不验证(blind trust)。** 工程侧的教训:工具返回什么就用什么,不校验——脏数据一路污染下游。人类版:Magic ToDo 给的拆解顺序其实不适合你(它不知道你下午才有状态、不知道第三步依赖别人的回复),你照单全收,卡在第二步,然后得出「拆了也没用」的结论。**工具给的是通用拆解,你才是那个了解现场的人**——调用结果过目三秒:顺序对吗?哪步其实不用做?这三秒是验证层,省不得。

**误用三:把工具当核心(tool as crutch for the wrong layer)。** 最深的误用。Formalizer 把你的话改得体面,于是渐渐地,**每一条消息都要过一遍它才敢发**——工具从「补短板」蔓延成「替主干」(脚手架还是拐杖篇讲过这个滑坡)。措辞润色是可教技能区:用它的正确姿势是顺便看它怎么改的(「哦,把指责句换成请求句」),几个月后简单消息自己就能写得体——**让调用递减的用法是脚手架,让调用递增的用法是拐杖**,同一个按钮,两种命运。

还有一个反向的错:**该调用时不调用。** 有人觉得「拆任务这种事都要靠工具,太丢人」——这相当于 agent 开发者出于骄傲拒绝给模型挂工具,让它徒手算三十七位乘法。执行功能的外部化不是作弊,是架构(本系列的全部主题);**羞耻感造成的调用不足,和依赖造成的调用过度,浪费的是同一种东西:你本可以用在正事上的能量。**

## 边界

同构强度 B 级:function calling 的误用模式是真实的工程经验,ADHD 的拆解辅助有行为干预基础,Goblin Tools 本身无对照研究,三种误用的映射是功能类比。声明:工具用法的优化,前提是任务系统底层大致健康——如果所有任务都拆了也动不了、伴随持续的情绪耗竭,那不是用法问题,请专业评估;工具的正确用法救不了放错层的问题。

## 今天就能试的 3 件事

1. **给 Magic ToDo 立准入门槛**:只有「盯了两分钟还不知道从哪下手」的任务才有资格被拆。
2. **建立「拆完即启动」的绑定**:拆解的最后一步永远是「现在做第一步的前五分钟」。
3. **审计你的 Formalizer 曲线**:这个月的调用次数比上个月多还是少?递增的话,今天起简单消息先自己写再对照。

好工具的危险恰恰在于它的好——**顺手到你忘了问:这次调用,是在推进任务,还是在体面地逃避它?** 桥就是桥,别在桥上安家。
