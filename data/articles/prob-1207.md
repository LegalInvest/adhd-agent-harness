# ChatGPT 之于 ADHD，就像 function calling 之于 LLM——但有人用错了

> function calling 的设计哲学是：模型不该亲自干所有事，但必须由模型决定何时调用什么。用 ChatGPT 的正确姿势藏在这个「但」字里。

LLM 工程史上有个转折点：工程师终于承认，让模型亲自算 38×47 是愚蠢的——它是概率生成器，不是计算器。于是有了 function calling：模型遇到超出自己架构强项的子任务时，生成一个工具调用请求，让计算器、数据库、搜索引擎去干，拿回结果继续推理。

注意这个设计里最精妙的分工：**执行外包了，但「何时调用、调用什么、结果怎么用」的判断权留在模型手里**。模型没有变懒，它升级成了调度者。

## ADHD 版 function calling：外包的应该是执行，不是判断

ADHD 大脑面对自己的架构短板（状态维护、精确重复、多步骤流程），最优策略和 LLM 一模一样：识别出「这不是我该亲自跑的负载」，然后调用外部工具。ChatGPT 是史上最通用的那个可调用函数：拆任务、起草邮件、整理混乱的思路、把 40 分钟的文书恐惧压缩成 5 分钟的修改工作。

用对的人的共同特征是：**调用前有一个明确的判断动作**。「这封邮件的难点是开头的语气，我调用它生成三种开头」——任务的定义权、选择权、验收权全程在自己手里。ChatGPT 是被调用方，你是那个带 function calling 能力的主模型。

## 用错的两种模式：判断权外流

**模式一：反向调用**。不是你调用它，是它调用你——打开对话框漫无目的地聊，让它的回复决定你下一步想什么。主客颠倒的标志：一小时后你完成的事，没有一件在打开它之前的计划里。工程对应物是失控的 agent 循环：工具调用产生的中间结果不断触发新调用，原始任务早已丢失。

**模式二：全量外包**。所有认知任务无差别丢给它——包括那些本该自己做判断的：这个方案好不好、我该选哪个 offer、这段关系怎么办。工程界从不让 function calling 替模型做推理决策，工具处理的是**良定义子任务**（输入输出清晰），不是主推理链本身。把主推理链也外包出去的 ADHD 用户，短期爽，长期两个恶果：自己的判断回路失训练，以及对一个平均化文本生成器的判断产生了不该有的信任。

## 一张自查表：你的调用是健康的吗

健康调用的四个特征，每次用完可以对一遍：

1. **调用前有任务定义**：我知道我要它干什么（哪怕只有一句话）。
2. **输出有验收**：我读了、改了、或者拒绝了它的结果，而不是直接照搬。
3. **调用有边界**：它处理的是子任务，主线的方向和责任在我。
4. **有些事明确不调用**：我保留了一份「必须亲自做」的清单——重要判断、需要练的技能、以及一切我希望六个月后自己变强的领域。

四条全中：你在用 function calling。中两条以下：你正在变成自己工具的工具。

## 为什么 ADHD 者两个方向的风险都更高

值得说破：ADHD 的兴趣驱动系统让「反向调用」格外危险（它是完美的新奇刺激泵），而长期的自我怀疑让「全量外包」格外诱人（终于有个东西比我靠谱）。同一个工具在 ADHD 用户手里的分化远大于常人——它恰好同时按着你最强的瘾点和最深的伤口。这不是不用的理由，是必须**带协议使用**的理由。

## 边界

同构强度 B 级：function calling 侧是成熟工程实践，ADHD 侧的认知卸载研究（cognitive offloading）真实存在且结论审慎——卸载低价值负载提升表现，卸载训练性负载损害长期能力。分界线的位置因人因阶段而异，本文的自查表是启发式，不是定理。

## 今天就能试的 3 件事

1. **下次打开 ChatGPT 前，先说完一句话**：「我调用它来做____」。说不完整就先别开。
2. **写你的「不外包清单」**：三件你决定永远亲自做的认知任务。贴在屏幕边上。
3. **给一次历史对话做验收复盘**：翻一段最近的长对话，数一数——里面有几次是你在调用它，几次是它在调用你。

工具调用是智能的标志——前提是调用方还是你。握着判断权去外包一切执行，你是长了一百只手的调度者；交出判断权，你只是一个会呼吸的前端。
