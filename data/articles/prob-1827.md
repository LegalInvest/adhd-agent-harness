# ADHD 职场人视角：为什么「治好 ADHD 的冲动下结论、想当然」和「让 LLM 不跑飞」其实是同一道工程题？

> LLM 自信地编造（幻觉），靠验证与自我批判兜底——同一套 harness 思路，两边都成立。

## 问题：为什么你总是“想当然”？

你有没有过这样的体验：会议上别人刚说半句话，你已经在心里把结论写好了；或者写代码时自信满满地提交，结果被 review 打回——因为漏了一个边界条件。这不是粗心，而是 ADHD 大脑的默认模式：**冲动下结论、想当然**。你的大脑像一台高速生成器，但缺少一个“验证与自我批判”的内置模块。

与此同时，LLM 工程师们正为同一个问题头疼：大模型自信地编造事实（幻觉），然后你需要用 agent harness 给它套上“验证循环”——比如让模型先输出计划，再执行，再检查结果，错了就重试。这套思路，和“治好”ADHD 的冲动下结论，其实是同一道工程题。

## 同构：高产生成核心 + 缺失的执行调度层

### ADHD 一侧

ADHD 大脑的核心特征是高产但无序的生成能力，同时缺乏可靠的执行调度层（来源：ADHD 大脑与 LLM 的同构）。执行功能（工作记忆、任务启动、时间管理）常出现故障，导致“想法很多，行动困难”。因此，ADHD 个体需要外部工具来“减少决策、保留上下文、让下一步行动显而易见”（来源：Best AI Tools for ADHD Productivity in 2026 (Honest Review) - Iwo Szapar）。

### LLM/Agent 一侧

LLM 本身是强大的生成核心，但缺乏可靠的内置调度与执行控制。因此，构建 AI agent 时需要“agent harness”——即“包裹 LLM 或 AI agent 的软件基础设施，处理模型本身之外的一切”（来源：What is an agent harness in the context of large-language models?）。这种架构常采用“复合 AI 系统”，通过“每个工作流可绑定不同 LLM”的架构来分配任务，并依赖“模型连接器”等组件管理外部工具和上下文（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。

### 同构对应

| ADHD 大脑 | LLM + Harness |
|----------|---------------|
| 高产但缺乏执行调度层的生成核心 | 高生成能力但需外部调度的 LLM |
| 工作记忆易丢失上下文 | LLM 无状态，需外部记忆管理 |
| 需要 AI 助手减少决策、保留上下文 | Harness 负责上下文工程和决策路由 |
| 外部工具（如 Goblin Tools）作为执行功能支架 | Harness 中的模型连接器、工具调用等 |
| 多巴胺驱动任务启动困难 | 缺乏内在奖励机制，需外部提示 |

## 解法同构：验证与自我批判循环

### ADHD 侧：用外部工具建立“验证步骤”

ADHD 的冲动下结论，本质是生成后缺少一个“检查”步骤。外部工具可以充当这个检查者。例如，**Goblin Tools** 的 Magic ToDo 功能自动将任务分解为小步骤（来源：Goblin Tools），这相当于将“完成一个大任务”这个模糊结论，拆解成一系列可验证的子步骤。每个小步骤完成后，你得到即时反馈，从而避免“想当然”地跳过细节。**Saner.AI** 通过本地记忆和快速检索，减少搜索循环和标签切换（来源：Saner.AI），相当于为你的工作记忆提供了一个外部缓存，防止你因为丢失上下文而做出错误假设。**Motion** 自动根据任务优先级和截止时间动态调整日程（来源：Motion），消除了“下一步该做什么”的决策负担，让你不必在冲动下随意选择任务。

这些工具的共同逻辑是：**在生成（想法、计划）之后，插入一个外部验证或引导步骤**，迫使你暂停、检查、再行动。

### LLM/Agent 侧：用 harness 实现验证循环

在 agent harness 工程中，核心模式是“计划-执行-验证”循环。例如，一个 coding agent 会先让 LLM 生成代码，然后自动运行测试、检查语法、对比预期输出，如果失败则返回错误信息让 LLM 自我修正。Harness 中的“模型连接器”和“工具调用”负责管理外部验证工具（如 linter、测试框架）。这种设计直接对应 ADHD 侧的需求：**将验证外化，而不是依赖内在的自我批判能力**。

### 核心观点：脚手架，不是拐杖

关键区别在于：**脚手架是可撤除的临时支撑，拐杖是长期依赖**。ADHD 工具和 agent harness 都应该设计为“逐步撤除”的脚手架，而不是永远替代内在能力。例如，Goblin Tools 的任务分解可以逐渐内化为用户的习惯，最终不再需要工具；同样，agent harness 中的验证循环应该随着模型能力提升而减少干预。目前，多数工具设计为长期使用，未提及撤除机制（来源：矛盾与存疑），这是需要警惕的。

## 争议与局限

- **同构命题的证据强度**：ADHD 大脑与 LLM 的同构主要基于概念类比和工具案例，缺乏大规模实证（来源：矛盾与存疑）。神经影像学和行为实验尚需更多验证。
- **过度依赖风险**：长期使用 AI 工具是否会削弱内在执行功能？目前缺乏纵向研究（来源：矛盾与存疑）。工具设计者应考虑撤除机制。
- **多巴胺干预争议**：多巴胺相关的 AI 检测技术尚在早期，且药物对认知改善的个体差异大（来源：矛盾与存疑）。

## 今天就能试的行动

1. **用任务分解对抗冲动**：下次接到复杂任务时，先用 Goblin Tools 或手写拆成 3-5 个可验证的小步骤，每完成一步打勾。这相当于给你的“生成核心”加了一个外部验证循环。
2. **建立“输出前检查清单”**：在工作流中插入一个 30 秒的验证步骤——比如写邮件前检查收件人、附件、语气。用 Saner.AI 或笔记应用保存这个清单，避免依赖记忆。
3. **用动态日程减少决策**：尝试 Motion 或类似工具，让它自动安排你的任务，而不是凭直觉决定“现在做什么”。这相当于把调度权交给外部 harness。
4. **反思工具依赖**：每周问自己一次：“没有这个工具，我能否完成同样的任务？”如果答案是否定的，尝试逐步减少使用，练习内化能力。

## 结语

ADHD 的冲动下结论和 LLM 的幻觉，根源相同：一个强大的生成核心，配了一个不靠谱的调度器。解法也相同：套上外部验证循环，让生成之后必有检查。这不是否定生成能力，而是承认“生成”只是半程，验证才是闭环。对 ADHD 用户来说，这意味着学会用工具建立你的“agent harness”；对工程师来说，这意味着理解你正在做的 harness 工程，本质上是在为人类认知搭建脚手架。两边的人，其实在解决同一个问题。
