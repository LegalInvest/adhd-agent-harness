# ADHD 职场人视角：为什么「治好 ADHD 的不擅长精确、重复、状态维护类任务」和「让 LLM 不跑飞」其实是同一道工程题？

> agent 通过 function calling 把精确计算外包给工具——同一套 harness 思路，两边都成立。

如果你是个ADHD职场人，你一定熟悉这种体验：脑子里有绝妙的方案，但一落到写邮件、填表格、跟进度这些“精确、重复、状态维护”的活儿，就像被抽走了CPU。如果你是个做Agentic Harness的工程师，你也一定熟悉这种体验：LLM能写出惊艳的文案，但一涉及数学计算、状态追踪、多步流程，就频频跑飞、幻觉、崩溃。

这两类人其实在解决同一个问题。ADHD大脑和LLM/agent共享一个核心缺陷：**它们都是高产但缺执行调度层的生成核心**。ADHD大脑的创造力不受限，但执行功能（工作记忆、任务启动、时间感知）像失控的采样温度——输出波动大、不可靠（来源：采样温度与表现波动）。LLM同样：它生成文本的能力惊人，但缺乏稳定、确定性的执行层来管理工具调用、状态维护和错误恢复（来源：AI Agent Systems: Architectures, Applications, and Evaluation）。

所以，当ADHD患者用Goblin Tools把“整理房间”分解成“捡起地板上的衣服”“擦桌子”等小步骤时，这和agent通过function calling把“计算订单总额”外包给计算器工具，是同一套工程思路。两者都在做同一件事：**把生成核心不擅长的精确、重复、状态维护类任务，卸载给外部系统**。

## 同构的工程：认知卸载与harness

ADHD侧的证据很具体。Goblin Tools的Magic ToDo功能“将压倒性的事情变成一系列不压倒性的事情”（来源：Goblin Tools），直接降低了任务启动门槛。Saner.AI通过本地记忆和快速检索，减少ADHD用户频繁的“搜索循环”和“标签切换”（来源：Saner.AI）。Lex允许用户通过单一指令触发多步骤任务序列，适应ADHD大脑对即时反馈和低启动门槛的需求（来源：Lex）。这些工具的核心逻辑是：**把需要精确分解、记忆维护、顺序执行的“脏活”外包给AI，让大脑只负责生成和决策**。

LLM/agent侧的证据同样清晰。现代agent系统通过“harness”调用工具、管理文件、处理长对话（来源：工具使用与认知卸载）。工程实践强调“确定性状态转换”和“计划-执行分离”模式，强制单向、确定性的控制流（来源：采样温度与表现波动）。最简单的agent工作流本身就是确定性的，遵循预定义步骤（来源：采样温度与表现波动）。这些做法的本质是：**把LLM不擅长的精确计算、状态追踪、多步执行外包给工具和确定性流程，让LLM只负责生成意图和理解上下文**。

两边的解法结构完全一致：生成核心（ADHD大脑/LLM）+ 外部执行层（AI工具/harness）。ADHD的“工具即用户界面”原则——工具应“插入现有堆栈”，不增加认知开销（来源：工具使用与认知卸载）——与agent工具设计的“agent UX”理念（命名、模式、错误处理等接口质量）如出一辙（来源：工具使用与认知卸载）。

## 脚手架，不是拐杖

但这里有一个必须划清的边界：**脚手架 vs 拐杖**。AI工具应该像脚手架，帮助大脑发展能力；而不是像永久拐杖，让大脑萎缩。wiki资料中多处警告“过度依赖可能削弱内在能力”（来源：矛盾与存疑）。例如，Otter.ai减轻笔记负担，但过度依赖可能削弱主动记笔记的能力（来源：ADHD的AI工具全景）。同样，在agent工程中，如果harness设计得过于“智能”，掩盖了底层LLM的不可靠性，那么当harness失效时，系统会崩溃得更惨。

关键在于：**脚手架的目标是让生成核心学会在必要时自己搭建临时支架，而不是永远依赖外部支架**。对于ADHD，这意味着工具应该帮助用户逐步内化分解任务、管理上下文的技能，而不是永远替用户做。对于agent，这意味着harness应该提供可观察、可调试的接口，让开发者理解并改进LLM的行为，而不是用黑箱掩盖问题。

## 局限与争议

必须诚实面对：目前AI工具对ADHD的有效性证据主要来自用户报告和概念类比，缺乏大规模对照实验（来源：矛盾与存疑）。Goblin Tools、Saner.AI、Lex等工具的设计逻辑合理，但个体差异很大——对某些人有效的工具对另一些人可能适得其反（来源：矛盾与存疑）。同样，agent harness工程也在快速发展中，但“小错误会累积，非确定性使可重复性复杂化”仍是核心挑战（来源：采样温度与表现波动）。

此外，多巴胺干预效果存在争议（来源：矛盾与存疑），而AI工具若过度依赖多巴胺激励（如完成任务后的正反馈），可能反而加剧注意力波动。

## 今天就能试的行动

1. **ADHD侧**：用Goblin Tools的Magic ToDo分解一个你拖延了很久的任务（比如“写周报”）。观察分解后的步骤是否降低了启动焦虑。如果有效，尝试将这种分解模式内化：下次面对复杂任务时，先自己列出3-5个子步骤，再对比工具的输出。

2. **Agent侧**：在你开发的agent中，识别一个容易“跑飞”的环节（比如多步数学计算）。将其重构为一次function call，调用一个确定性的计算工具。对比重构前后的成功率和响应时间。

3. **通用**：无论是ADHD还是agent，都试试“计划-执行分离”。ADHD：先花5分钟用纸笔列出任务步骤（计划），再逐一执行（执行），中间不切换。Agent：先让LLM生成一个结构化计划（JSON格式），再让确定性流程逐条执行计划，每一步调用工具验证结果。

4. **反思依赖**：每周检查一次：我/我的agent是否在某个环节过度依赖外部工具？如果是，尝试在没有工具的情况下完成一次，评估能力是否退化。

同构视角的价值不在于制造一个漂亮的类比，而在于让两类人看到：**你面对的困难不是孤立的，解法也不是孤立的**。ADHD患者可以从agent工程中学习“确定性状态转换”的纪律；agent工程师可以从ADHD工具中学习“降低认知开销”的用户体验。最终，两者都在做同一件事：为强大的生成核心，配上一个可靠的执行层。
