# RescueTime 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 RescueTime：它到底补上了哪一层执行功能？

如果你是个 ADHD 患者，你可能有过这样的体验：打开电脑想写一份报告，结果两小时后发现自己刷了 20 个标签页，文档里只有标题。如果你是个 AI agent 工程师，你可能见过这样的日志：agent 收到了一个任务，然后开始漫无目的地调用工具，最终输出了一堆幻觉——因为它的上下文已经膨胀到丢失了原始目标。

这两个场景共享同一个底层问题：**强大的生成核心缺乏一个可靠的执行调度层来维护状态和上下文**。ADHD 大脑和 LLM 都是高产但容易“跑偏”的生成器，而 RescueTime——这个老牌的注意力追踪工具——恰好提供了一种解决方案，但大多数人用错了它。

## 问题：状态维护的缺失

ADHD 患者的核心困难之一是“状态维护”——在执行一个任务的过程中，保持对当前目标和进度的感知。Wiki 资料中反复提到，ADHD 大脑“易被环境带偏”，时间感知扭曲，任务碎片化（来源：上下文工程）。同样，LLM agent 在复杂任务中会面临“上下文膨胀与推理退化”（来源：上下文工程），控制逻辑分散，导致 agent 忘记自己最初要做什么。

RescueTime 通常被当作“时间追踪工具”使用——它默默记录你花在哪些应用和网站上的时间，然后给你一份报告。但如果你只把它当“事后监控”，你就错过了它真正的价值：**实时维护你的注意力状态**。

## 同构：RescueTime 作为“状态维护层”

对于 ADHD 大脑，RescueTime 的核心功能不是记录，而是**提供实时的注意力反馈**。它的“FocusTime”功能可以让你设定一个专注时段，期间自动屏蔽干扰网站。它的“Get Focused”提示可以在你分心时弹出提醒。这些功能本质上是在做**上下文工程**——主动管理你当前“注意什么”的范围，防止上下文被无关刺激污染。

对于 LLM agent，对应的概念是 function calling 中的“工具调用”与“上下文维护”。一个设计良好的 harness 不会让 agent 随意调用工具，而是通过“规划工件”和“验证循环”来保持对原始目标的追踪（来源：幻觉与验证循环）。RescueTime 的 FocusTime 就像 agent 的“沙盒”——限制工具集，防止上下文膨胀。

但问题在于：**多数人把 RescueTime 当成了“拐杖”而非“脚手架”**。他们打开 RescueTime 的监控，然后继续放任自己分心，只是事后看报告感到内疚。这就像给 agent 加了一个 function calling 接口，但不给它规划器和验证循环——结果 agent 依然会跑偏，只是现在有了更多工具可以滥用。

## 证据：RescueTime 到底补上了哪一层？

从执行功能的角度看，RescueTime 主要补上的是**状态维护**（working memory 的持续更新）和**时间盲**（时间感知）。Wiki 资料中提到的 Goblin Tools 通过任务分解降低启动焦虑（来源：Goblin Tools），Saner.AI 通过知识回忆减少搜索循环（来源：Saner.AI），Lex 通过单一指令触发多步骤（来源：Lex）。这些工具分别补上了任务启动、工作记忆和任务执行中的某些环节。

而 RescueTime 独特的地方在于：它**实时**告诉你“你现在在做什么”。这对 ADHD 患者来说至关重要，因为“时间盲”让他们难以感知自己花了多久在无关事情上。一个 2026 年的评测提到，RescueTime 的“实时提醒”功能被用户评价为“像有一个教练在旁边说‘嘿，你跑题了’”（来源：Best AI Tools for ADHD Productivity in 2026）。

对于 LLM agent，这对应的是“验证循环”中的**状态监控**。Harness 工程强调“复杂的 harness 不会盲目执行工具”，而是实现验证步骤，例如检查输出格式或对模型编写的代码运行测试用例（来源：幻觉与验证循环）。RescueTime 的实时提醒就是 ADHD 大脑的“验证步骤”。

## 核心观点：脚手架 vs 拐杖的边界

我的判断是：**RescueTime 的正确用法是作为“注意力状态维护的脚手架”，而非“事后监控的拐杖”**。

- **脚手架用法**：主动设置 FocusTime 时段，打开实时提醒，在分心时立刻回拉。这相当于 agent 的“上下文窗口管理”——主动限制输入范围，防止膨胀。
- **拐杖用法**：只开着监控，从不主动设置限制，事后看报告自责。这相当于 agent 的“无限制 function calling”——工具越多，跑得越偏。

Wiki 资料中的“矛盾与存疑”指出，多数工具设计为长期使用，未提及撤除机制（来源：矛盾与存疑）。RescueTime 的“脚手架”性质在于：你可以逐渐减少对实时提醒的依赖，从“外部执行功能”内化为“注意力自我监控”。但如果你永远只依赖事后报告，你永远不会发展出内在的注意力调节能力。

## 争议与局限

需要诚实指出的是：RescueTime 对 ADHD 的效果缺乏独立随机对照试验，现有证据主要来自用户评价和综述（来源：矛盾与存疑）。此外，它的“实时提醒”可能对某些 ADHD 亚型造成干扰——比如对注意力过度集中的患者，频繁提醒反而打断心流。

从 LLM agent 的角度看，RescueTime 的“状态维护”功能对应的是 harness 中的“上下文管理”，但 agent 还有“规划”和“验证”等更复杂的组件。RescueTime 只解决了其中一层，不能替代完整的执行功能脚手架。

## 今天就能试的行动

1. **设置一个 FocusTime 时段**：每天选一个 25 分钟的任务，打开 RescueTime 的 FocusTime，屏蔽所有干扰网站。完成后记录感受。
2. **打开实时提醒**：在 RescueTime 设置中启用“Get Focused”提醒，当你在非工作站点停留超过 5 分钟时弹出。
3. **对比实验**：一周只用事后报告（拐杖用法），一周用实时提醒+FocusTime（脚手架用法），比较任务完成率和主观感受。
4. **结合其他工具**：用 RescueTime 维护状态，用 Goblin Tools 分解任务，用 Saner.AI 存储中间结果——形成完整的执行功能 harness。

RescueTime 不是灵丹妙药，但它提供了一个关键洞察：**状态维护是执行功能的基石**。无论是 ADHD 大脑还是 LLM agent，没有这一层，再强大的生成核心也只是脱缰的野马。
