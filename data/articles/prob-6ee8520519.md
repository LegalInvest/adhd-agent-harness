# ADHD 职场人视角：Agent 工程师每天在解决的问题，ADHD 大脑已经面对了一辈子

> 公司引入 AI agent 做流程自动化的第三个月,一场跨部门复盘会上出现了戏剧性一幕。工程团队汇报 agent 的落地难题:「它单步能力很强,但长流程会中途忘记最初目标」「它会在两个工具之间反复横跳」「它需要我们把任务拆到很细才稳定」「监控它比想象中费人力」。会议室后排,一位从不发言的行政岗女员工在笔记本上越记越快,散会后她对同事说了一句让对方愣住的话:「**他们花三个月发现的每一条,我三十岁之前就都知道了——那就是我每天上班的感受。区别是,他们的系统有一整个团队在给它搭脚手架,而我这些年被要求『自己克服』。**」两周后,她带着一份对照表去找了主管。

收敛:「职场人视角」这个题,本文只回答一问——**ADHD 职场人怎么把「我一辈子的困境经验」转化为职场里的实际筹码(三个可操作的兑换动作)?**(同构本身的论证见站内各专文,这里直接使用结论。)

## 穿透:公司愿意为 agent 付出的耐心,就是你谈判桌上的锚点

先看清那个不对称,它是全部筹码的来源。公司对 agent 的态度是纯工程的:发现它长流程会漂移→加 checkpoint 和 verifier;发现它需要细粒度任务→改任务分发的格式;发现它需要监控→配监控面板和人力——**没有人说「这 agent 怎么这么不自觉」,没有人建议它「用心一点」**;预算、流程改造、专人支持,全部顺理成章。而同一家公司对表现出同构困难的员工,给出的历来是道德处方(「加强责任心」)。这个双标不是恶意,是**框架差**:agent 的困难被默认为架构问题,人的困难被默认为品格问题。她那份对照表的杀伤力就在这里:**当公司已经用真金白银承认了「这类困难是架构问题、值得配架构解」,它就失去了对同构困难说「你自己克服」的逻辑立足点。**

三个兑换动作:①**语言兑换——用 agent 词汇重新申报你的困难。**「我容易分心」是品格自白,「我和我们的 agent 一样,长流程需要 checkpoint——我申请把季度目标拆成双周对账」是架构提案;「我总忘事」换成「我需要显式的任务队列而不是口头分派——就像我们不会口头给 agent 派活」。**同一个请求,前一种语言乞求宽容,后一种语言引用公司自己已经批准过的工程逻辑**(岗位接口篇的谈判策略,在 agent 时代获得了现成的公司内部判例);②**专家兑换——把你的经验变成 agent 落地的咨询价值。**那场复盘会暴露的每个难题(长程漂移、工具横跳、任务粒度、监控成本),你都有几十年的一线数据:什么样的拆解粒度真的可执行、什么样的提醒会被忽略、什么样的监控令人窒息什么样的赋能——**公司正在花钱试错的问题,你身上就有答案**;主动参与 agent 流程设计的讨论,你的「病历」在那个房间里是稀缺的专业知识(创业者篇会把这个逻辑推到底,职场版的用法是:换一个被看见的位置);③**基建兑换——搭 agent 的顺风车提需求。**公司为 agent 建的基础设施(任务看板、结构化交接、流程显式化、进度自动追踪),恰好是 ADHD 员工需要了多年而不好意思提的外部执行功能——**现在提,不再是特殊照顾,是「让人和 agent 用同一套基建」的合理化建议**;agent 时代给 ADHD 职场人最实际的礼物,可能就是让全公司的流程被迫显式化——那正是你一直需要的世界。

## 验证

可测:用语言兑换重新提一次你搁置已久的工作调整请求,对比旧语言版本的结果。可证伪:若你的公司对 agent 的困难同样只有道德语言(「这 AI 怎么这么笨」然后弃用),说明组织本身没有工程框架——三个兑换在此失效,你的筹码换不出东西,该考虑的是接口更兼容的环境(接口成本篇的迁移决策)。

## 决策

做什么:列你的「agent 对照表」(它的每个已被接受的困难↔你的同构困难↔它获得的架构解);挑一条做语言兑换;关注公司 agent 落地会议并找机会入场。

不做什么:不要把对照表用成控诉(「你们对机器都比对我好」的怨气版本会立刻输掉——它的正确用法是逻辑引用,不是情绪呈堂);不要等公司主动发现这个同构(不会的,框架差是默认设置,兑换要自己发起)。

先做什么:写下你最想被架构化解决的一个困难,用 agent 词汇翻译一遍——下次一对一,带着它去。

## 边界

同构的证据边界见各专文(功能层 B 级,道德层不适用——问责篇已辨析);「兑换」策略的效果依赖组织文化,无对照研究(本文未做正式 A/B/C 分级)。职场披露诊断与否是独立的风险决策(披露篇),语言兑换的优点恰在于不要求披露——谈架构,不谈诊断。

1. 列三行版 agent 对照表。
2. 翻译一个困难成架构提案,预演一遍。
3. 打听公司 agent/自动化项目的例会——那是你的经验最值钱的房间。
