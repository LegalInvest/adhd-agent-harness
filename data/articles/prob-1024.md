# Routinery 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 Routinery：它到底补上了哪一层执行功能？

你是否有过这样的经历：计划好一天的任务，打开手机想查个资料，结果半小时后发现自己刷完了整个社交媒体首页，而原本要写的那份报告还纹丝不动。或者，你正沉浸在一项工作中，突然想起要回一条消息，结果一小时后你还在聊天界面里，完全忘了刚才在做什么。这种分心走神或超聚焦跑偏、丢失目标的状态，是 ADHD 人群的日常。而另一边，LLM/Agent 工程师们也在为类似的问题头疼：一个设计良好的 agent，在长程任务中往往会逐渐偏离初始目标，产生“目标漂移”。他们需要一种“重锚定”机制——定期将 system prompt 重新注入，让 agent 回到正轨。

这两类问题看似无关，实则共享同一个结构：一个高产的生成核心（ADHD 大脑或 LLM），却缺乏可靠的执行调度层。ADHD 大脑像一台没有操作系统的超级计算机，能并行处理海量想法，但无法有序调度；LLM 同样能生成流畅文本，但若无编排框架，就会在上下文中迷失。而工具 Routinery，恰好扮演了那个“重锚定 system prompt”的角色。

## 同构视角：ADHD 大脑与 LLM 的“目标漂移”

ADHD 的执行功能缺陷——包括工作记忆不足、任务启动困难、时间盲和注意力分散——可以类比为 LLM 的上下文溢出和调度失败（来源：AI 与 ADHD 的专注力）。当 ADHD 大脑被一个新刺激捕获，原有的目标就从工作记忆中“溢出”，就像 LLM 的上下文窗口被无关信息填满，导致模型忘记初始指令。同样，一个 agent 在长程任务中，如果没有定期重锚定，其输出会逐渐偏离用户意图。

这种同构意味着，解法也应是同构的：给生成核心套上一个“harness”（脚手架），通过外部系统补偿执行功能。Routinery 正是这样的脚手架——它通过结构化的步骤引导和定时提醒，帮助用户将注意力锚定在当前任务上，就像 LLM 工程师每隔 N 步就向 agent 注入一次 system prompt。

## Routinery 的实际机制：重锚定而非自动化

Routinery 的核心功能是创建“例行程序”（routines），将复杂任务分解为小步骤，并设置每个步骤的预计时长。当你启动一个 routine，它会按顺序展示步骤，并倒计时。这种设计直接针对 ADHD 的两大痛点：
- **任务启动**：面对一个庞大任务时，ADHD 大脑常因“启动障碍”而瘫痪。Routinery 将任务拆解为“第一步做什么”，降低了启动门槛。
- **注意力维持**：倒计时和步骤切换提供了外部节奏，防止超聚焦跑偏。

这类似于 LLM 中使用的“思维链”（Chain-of-Thought）提示——将复杂问题分解为中间步骤，让模型一步步推理，而不是一次性输出。Routinery 的步骤序列就是 ADHD 大脑的“思维链”。

更重要的是，Routinery 的“重锚定”机制：当用户完成一个步骤，应用自动切换到下一步，并显示剩余时间。这个切换动作本身就是一个“重锚定”信号——它把用户从可能的分心中拉回，重新聚焦于当前步骤。这与 LLM 工程师在 agent 的每个循环中注入 system prompt 的做法如出一辙。

## 为什么有人用错了：拐杖与脚手架的区别

然而，许多用户抱怨 Routinery“没用”——他们下载后设置了几个 routine，用了两天就放弃了，或者觉得太死板。问题出在哪里？在于他们把 Routinery 当成了“拐杖”，而不是“脚手架”。

拐杖与脚手架的区别在于：拐杖是永久替代，而脚手架是临时支撑，旨在逐步撤除（来源：矛盾与存疑）。Routinery 的设计初衷是脚手架——它帮助你建立内在的执行节奏，但如果你永远依赖它的每一步提醒，你就无法发展出内在的调度能力。更糟糕的是，如果你没有理解“重锚定”的机制，只是机械地跟着 app 走，那么一旦 app 不在，你就会彻底失控。

另一个常见错误是：把 Routinery 当作任务列表，而不是执行框架。许多人设置了“工作”routine，包含“打开电脑”“写报告”“检查邮件”等步骤，但写报告这一步本身仍然太大，导致卡住。Routinery 的有效使用要求将步骤细化到“可执行单元”——就像 LLM 的思维链需要足够细的步骤才能有效。

## 争议与局限

尽管 Routinery 在理论上有力，但必须诚实指出其局限：
- **缺乏临床证据**：目前没有针对 Routinery 在 ADHD 人群中的随机对照试验。它的效果主要基于用户主观报告和与执行功能理论的对齐（来源：AI 与 ADHD 的专注力）。
- **个体差异**：ADHD 亚型（注意力缺陷 vs 多动冲动）对结构化工具的反应不同。对于多动冲动型，过于僵化的步骤可能引发反抗（来源：矛盾与存疑）。
- **脚手架 vs 能力发展**：长期使用 Routinery 是否阻碍内在执行功能发展？目前无研究证明它可以逐步撤除（来源：矛盾与存疑）。

## 今天就能试的行动

1. **创建一个小 routine**：选一个你经常拖延的日常任务（如“早上准备出门”），分解成 5-8 个具体步骤，每个步骤不超过 5 分钟。设定每个步骤的倒计时。
2. **识别你的“漂移点”**：使用 RescueTime 或类似工具（来源：RescueTime）追踪你一天的时间使用，找出你最容易目标漂移的时刻。然后，在这些时刻设置 Routinery 的“重锚定”提醒。
3. **尝试“身体在场”替代**：如果 Routinery 的步骤让你感到孤独，可以结合 Focusmate（来源：Focusmate）——预约一个虚拟伙伴，在对方注视下执行 routine。身体在场效应能增强 accountability（来源：身体在场效应）。
4. **一周后评估**：使用一周后，问自己：我是否开始内化这些步骤？能否在不打开 app 的情况下自动执行？如果是，说明脚手架在起作用；如果完全依赖 app，则考虑调整步骤粒度或减少使用频率。

Routinery 不是魔法，它只是一个外部重锚定工具。真正的变化来自你理解了自己的“目标漂移”模式，并学会用脚手架来训练内在调度层。就像 LLM 工程师不会让 agent 永远依赖重复注入——他们最终希望 agent 学会保持目标一致性。你也是自己的工程师。
