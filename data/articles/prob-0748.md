# Routinery 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Routinery：它到底补上了哪一层执行功能？

## 问题：为什么 Routinery 对某些 ADHD 用户无效，对另一些却是救星？

你见过这样的场景吗？一位 ADHD 朋友兴奋地下载了 Routinery——一款号称“帮你建立晨间/夜间例行程序”的 AI 应用——然后一周后放弃，抱怨“它太死板了，我根本坚持不下来”。另一边，一位工程师在构建 AI agent 时，发现让 LLM 直接调用 Python 计算器比让它在内部推理准确得多，但前提是必须设计好 function calling 的上下文。

这两件事看似无关，实则共享同一个底层困境：**强大的生成核心缺乏可靠的执行调度层，需要外部工具来执行精确、重复、状态维护类任务，但工具本身也可能成为新的认知负担。**

Routinery 之于 ADHD，就像 function calling 工具调用之于 LLM——但有人用错了。本文从同构视角出发，结合 wiki 资料中的工具证据，帮你判断：你是在用脚手架，还是在用拐杖？

## 同构：ADHD 大脑与 LLM 共享同一类缺陷

ADHD 大脑的核心特征是高产但无序的生成能力，同时缺乏可靠的执行调度层（来源：ADHD 大脑与 LLM 的同构）。LLM 同样如此：它擅长生成文本，但无状态、缺乏内置调度，需要外部的“agent harness”来提供上下文管理、工具接口和验证循环（来源：外部执行功能层）。

具体对应关系如下：

| ADHD大脑 | LLM + Harness |
|----------|---------------|
| 高产但缺乏执行调度层 | 高生成能力但需外部调度的LLM |
| 工作记忆易丢失上下文 | LLM无状态，需外部记忆管理 |
| 需要AI助手减少决策、保留上下文 | Harness负责上下文工程和决策路由 |
| 外部工具（如Goblin Tools）作为执行功能支架 | Harness中的模型连接器、工具调用等 |

Routinery 的作用，就是充当 ADHD 大脑的“function calling 工具”——它接管了精确、重复、状态维护类任务（如按时刷牙、吃药），让生成核心可以专注在创意和决策上。但问题在于：**Routinery 的设计假设用户能维持长时间专注、记住昨天的问题并执行计划，但实际用户往往做不到**（来源：外部执行功能层）。这正是矛盾所在。

## 证据：Routinery 补上了哪一层执行功能？

从 wiki 资料中的工具证据来看，成功的 ADHD 工具都补上了特定的执行功能层：

- **任务启动**：Goblin Tools 的 Magic ToDo 功能通过自动分解任务降低启动焦虑，将“压倒性的事情变成一系列不压倒性的事情”（来源：Goblin Tools）。Routinery 的固化流程类似，但缺乏灵活性。
- **工作记忆**：Saner.AI 通过本地知识回忆减少搜索循环（来源：Saner.AI）。Routinery 不提供记忆功能，用户必须自己记住流程。
- **决策减负**：Lex 通过单一指令触发多步骤任务序列（来源：Lex），类似 function calling 的“一次调用，多次执行”。Routinery 也做同样的事，但它的步骤是预设而非动态生成的。

关键区别在于：**Goblin Tools、Saner.AI、Lex 都允许用户动态调整，而 Routinery 强加固定顺序**。这就像 LLM 的 function calling 如果写死参数而不允许模型选择，就会导致失败。

## 核心判断：脚手架 vs 拐杖——Routinery 的正确用法

我的核心观点是：**Routinery 只有在作为“外部执行功能层”使用时才有效——即它必须被设计为可逐步撤除的临时支撑，而非永久替代内在执行功能**（来源：矛盾与存疑）。当前多数工具（包括 Routinery）设计为长期使用，未提及撤除机制，这可能导致过度依赖，阻碍内在能力发展（来源：矛盾与存疑）。

正确用法类似 LLM 的 agent harness：将 Routinery 视为“模型连接器”（来源：外部执行功能层），只负责执行精确的、重复的子任务，而把决策和创意留给大脑。例如，用 Routinery 管理晨间洗漱流程（低认知需求），但不要用它规划当天的工作优先级（高认知需求）。

## 今天就能试的行动

1. **为 Routinery 设置“可跳过”选项**：在例行程序中加入“如果不想做，直接点跳过”的规则，避免僵化。
