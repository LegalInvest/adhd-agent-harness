# 为什么 ADHD 最危险的不是拖延，而是把搭系统误认为推进——你以为自己在铺路，其实一直在修停车场

> agent 工程有个笑话：有些团队花一年优化编排框架，还没让 agent 完成过一个真实任务。ADHD 生产力圈的版本是：第七次迁移笔记系统的人，笔记里还没有一篇写完的东西。

ADHD 者的自我改造史往往长这样：Notion 模板搭了三天，精美绝伦；GTD 读完了，工具买齐了；番茄钟、时间块、子弹笔记轮着来；上个月刚把全部笔记从 Notion 迁到 Obsidian，这个月在研究要不要迁回去。每一次都伴随着真实的兴奋和「这次终于要好起来了」的确信。

而真正要做的那件事——那份报告、那个项目、那次谈话——原地未动。

## 为什么搭系统对 ADHD 有毒品级的吸引力

这不是偶然踩坑，是精准踩中架构的坑。搭系统这个活动，恰好集齐了 ADHD 奖赏系统的所有兴奋剂：**新奇**（新工具新方法）、**即时反馈**（模板搭好立刻能看到）、**掌控幻觉**（混乱的人生在一个干净的界面里显得可控）、**零失败风险**（系统不会像真实任务那样评判你）。而它同时又顶着「生产力建设」的正当名义——这让它成为完美拖延：**所有拖延的爽感，加上零拖延的负罪感。**

真实任务恰恰相反：旧的、反馈延迟的、失控感强的、可能失败的。同一颗奖赏系统，两种负载，选择毫无悬念。

## agent 工程的同款陷阱：harness 过度工程

LLM 圈对这个陷阱有一线体感。过去两年无数团队掉进「框架坑」：编排框架越搭越复杂，抽象层越垒越高，评估流水线精美绝伦——但系统从来没有稳定完成过一个端到端的真实任务。工程界事后总结出的解毒剂朴素得扎心：**harness 的每一个组件，必须由一个真实任务的真实失败来证明其必要性。** 先跑任务，跑挂了，看挂在哪，只补那一块。顺序不能反：不是先建好完美系统再开始干活，是干活干出伤口，再给伤口配绷带。

这条纪律直接可用于人生：**你的生产力系统只允许响应式生长**。忘了三次会议→配日历提醒（只配这个）；项目总在 80% 烂尾→配收尾承诺装置（只配这个）。任何不对应真实失败的系统建设,一律定性为娱乐——娱乐可以，但要记在娱乐的账上。

## 三条判别式：铺路还是修停车场

分不清的时候，用这三个问题当场检验：

1. **这个系统服务的具体任务叫什么名字？** 说不出具体任务名（「以后所有项目都能用」），停车场。
2. **系统建设完成的定义是什么？** 没有明确终点、可以无限精修的，停车场。路有尽头，停车场可以永远扩建。
3. **过去 7 天，系统产出了多少真实任务的推进？** 系统维护时间 > 系统节省时间，停车场——而且是收费的。

## 已经修了很多停车场怎么办

不必羞愧，也别急着拆——有个温和的转化协议：**冻结 + 挪用**。冻结所有系统迁移和新工具尝试 90 天（写下来，告诉一个人）；现有系统哪怕不完美也照用——工程界的另一句老话：能跑的丑系统胜过完美的设计稿。然后把原本用于搭系统的那股兴奋，刻意挪给真实任务里最接近「搭建」性质的部分（新项目的框架期、方案的构思期）——你对建设的热情是真实资产，只是一直投错了标的。

## 边界

同构强度 B 级：harness 过度工程是 agent 社区的真实教训，ADHD 侧的「生产力拖延」是临床与社区反复观察到的模式（其机制——新奇寻求与失败回避——有文献支撑，但「搭系统成瘾」本身不是正式诊断）。另外注意：本文批判的是替代性系统建设，不是系统本身——本系列通篇都在教搭 harness，区别只在顺序：先有真实失败，再有对应组件。

## 今天就能试的 3 件事

1. **给现有系统做一次审计**：用三条判别式过一遍你的工具和流程，诚实标注：路 / 停车场。
2. **签一份 90 天冻结协议**：不换工具、不迁移笔记、不研究新方法。把这句话发给一个会取笑你的朋友。
3. **今天推进一件真实的事**：从你最重要的搁置任务里取出 25 分钟的推进——用你现有的、不完美的系统。

系统的唯一价值是让路上的车跑起来。如果车三个月没动过,再漂亮的停车场也只是它的坟墓——熄火的原因从来不在停车场里，在你不敢拧的那把钥匙上。
