# 为什么用 Otter.ai 治 ADHD 的时间盲，和给 agent 套 planner-executor 任务分解 是一回事？

> 会议结束,老板说「那就按刚才说的推进」。对时间盲的 ADHD 者,这句话是一颗延时炸弹:刚才说的——哪几件?谁负责?什么时候要?这些信息在会议中出现过,但它们**没有被转换成带时间坐标的任务**,散会即蒸发;两周后炸响:「上次不是说好你来跟进吗?」Otter.ai 这类会议转录工具,涣散篇讲过它卸载「记」的信道;对时间盲,它的位置更靠上游,对准的是 planner-executor 架构里最初的一环:**计划的输入捕获——承诺(commitments)从对话流里被提取出来,才有资格进入 planner**。

先讲 agent 侧这一环。planner 不凭空规划,它的输入是需求与约束;而真实系统里,需求很少以整洁的格式到达——**它们混在对话流里**(用户的一段话里嵌着三个要求、两个期限)。所以成熟 agent 的第一道工序是**结构化提取**:从非结构化输入里抽出任务、期限、负责人、依赖,转换成 planner 可消费的格式;这道工序失守,后面全链白搭——**没被提取的需求,对系统等于不存在**。请注意这个表述,它马上要在人类侧字面复现。

ADHD 侧:会议对时间盲者是最恶劣的需求输入环境。承诺在对话中以口头、分散、非正式的形态飞过(「那这个你看一下」「下周碰一下进度」),捕获它们需要同时做三件事:听懂当下讨论+识别「这句是给我的任务」+记录含期限的条目——**三重并发,而双任务已经是 ADHD 的塌方区**(Otter 涣散篇)。结果是系统性的:会议中飞过的承诺,相当比例从未落进任何系统——**没被提取的承诺,对你的 planner 等于不存在**;两周后它以指责的形态回来时,你甚至没有「忘了」的记忆,因为它从未被记住过。这就是时间盲最冤的形态:**不是你估错了时间,是那件事根本没进入你的时间系统**。

Otter 类工具在这条链上的价值,要按工序拆:**①全量捕获兜底**——转录保证「说过的都在」,提取的原材料不再依赖你的三重并发;**②AI 摘要与行动项识别**——自动拉出「谁、什么、何时」的候选列表:这就是结构化提取的自动化(准确率不完美,但候选列表+人工确认,远胜于纯靠现场脑力);**③但最关键的一步它做不了**——行动项从摘要进入你的任务系统(带上 deadline、绑定提醒),这一步必须由你执行。所以完整的协议是涣散篇「会后十分钟」的时间盲加强版:**散会十分钟内,过一遍 AI 行动项→每一条要么进任务系统(带日期!)要么显式丢弃→有模糊期限的(「下周碰一下」)当场翻译成具体日期**——这十分钟是整条链的提交点,跳过它,前面的转录只是留了个案发现场的录像。

一个更结构的建议,借 agent 工程的「接口规范」思维:**推动你的会议输出标准化**。工程里 planner 的输入格式是被定义的;人类会议同理——散会前最后两分钟公共确认「行动项:谁/什么/何时」,对全员都是增益,对 ADHD 是救命(把提取工序从你的私人负担变成会议的公共工序)。你甚至可以用「我来发会议纪要」主动接管这道工序——**掌握提取权的人,反而最不会被漏掉的承诺伏击**。

## 边界

同构定位(本文未做正式 A/B/C 分级,链条偏功能类比):双任务负载与前瞻记忆的证据扎实,结构化提取是 agent 工程真实工序,但「转录工具改善 ADHD 承诺管理」无直接研究,链条依赖使用协议而非工具本身。声明:录音需全员同意;若承诺失守已严重损害职业关系,除工具外值得做系统的临床与职业支持评估。

## 今天就能试的 3 件事

1. **给下次会议装提交点**:散会 10 分钟内处理行动项——每条进任务系统(带日期)或显式丢弃,不许悬空。
2. **翻译一次模糊期限**:「回头」「下周碰」「尽快」——今天把手头所有这类表述改写成具体日期。
3. **抢一次纪要权**:主动说「我来发行动项清单」——提取权在手,被伏击的概率立降。

两周后炸响的那颗雷,不是你「忘了」——**是它从未被提取,从未进入你的时间系统**。agent 工程的第一道工序叫结构化提取;人类版叫:散会十分钟,把飞过的话钉成带日期的字。
