# Inflow 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Inflow：它到底补上了哪一层执行功能？

你是否有过这样的经历：打开一个复杂项目，大脑瞬间空白，明明知道该做什么，却像被冻住一样无法启动？又或者，你刚想好下一步行动，转头就忘了，只能反复在标签页间搜索？ADHD 人群对这些场景再熟悉不过。而另一边，AI 工程师们也在为一个类似的问题苦恼：大语言模型（LLM）能生成惊艳的代码，但一旦任务链条变长，它就“迷失”在上下文里，要么跑偏，要么重复错误。

这两类困境，其实共享同一个底层结构：**一个高产但缺乏执行调度层的生成核心**。ADHD 大脑与 LLM 在认知机制上惊人地平行——两者都有强大的生成能力，却缺少一个可靠的“驾驶座”来指挥行动。最新研究揭示，ADHD 患者的工作记忆容量未必差，但自上而下的控制和调节能力不足，呈现“强记忆、弱控制”的认知剖面（来源：Strong Memory, Weak Control: An Empirical Study of Executive Functioning in LLMs）。而 LLM 的 Transformer 自注意力机制同样缺乏人类特有的执行控制，在长上下文下冲突解决能力崩溃至随机水平（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。

正因如此，两边的解法也结构同构：**为生成核心套上 harness（脚手架）**。在 LLM/agent 侧，现代 AI 编码代理通过“复合 AI 系统架构”来弥补调度缺陷，包括“工作负载专用模型路由、分离规划与执行的双代理架构、惰性工具发现、自适应上下文压缩”（来源：Building AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned）。这些技术本质上是在 LLM 外部搭建调度层，防止“上下文膨胀和推理退化”。在 ADHD 侧，AI 工具则充当外部执行功能层，通过分解任务、管理上下文、增强记忆和提供实时反馈，补偿内在的调度缺陷（来源：ADHD 的 AI 工具全景）。

## 工具调用：LLM 的“外部执行功能”

LLM 的 function calling 机制，本质上是把精确计算外包给外部工具。LLM 负责“生成意图”，工具负责“执行计算”。例如，当 agent 需要计算复杂数学或查询数据库时，它会调用专门的函数，而不是让 LLM 自己硬算。这种设计承认了 LLM 的弱点——它不擅长精确、重复、状态维护类任务，而这些恰好是传统软件的优势。

同样的逻辑适用于 ADHD。ADHD 大脑不擅长精确、重复、状态维护类任务——比如记住截止日期、分解复杂项目、维持长时间专注。因此，AI 工具作为“外部函数”，应该承担这些任务，让大脑专注于创意和决策。

## Inflow：为 ADHD 大脑搭建的“function calling”

Inflow 是一款专为 ADHD 设计的 AI 工具，它通过结构化的课程、日常微习惯和社区支持，帮助用户建立执行功能技能。从同构视角看，Inflow 相当于一个“调度层 harness”，它做的事情与 LLM 的 function calling 框架完全对应：

- **任务分解**：Inflow 的“魔法开关”将大任务拆为小步骤，类似 Goblin Tools 的 Magic ToDo，降低启动焦虑（来源：Goblin Tools）。用户反馈称“将压倒性的事情变成一系列不压倒性的事情”。
- **记忆与上下文管理**：Inflow 整合了知识回忆功能，类似 Saner.AI 的本地记忆，减少搜索循环和标签切换（来源：Saner.AI）。
- **实时反馈与反思**：Inflow 提供结构化提问和反思，类似 Reflect 的反思模式，提供即时外部反馈（来源：Reflect）。

这些功能直接针对 ADHD 的执行功能缺陷：工作记忆、任务启动、时间盲。研究表明，ADHD 的工作记忆易受干扰，而 LLM 同样表现出类似人类的干扰特征——随着记忆负荷增加表现下降，受近因效应影响（来源：Human-like Working Memory Interference in Large Language Models）。因此，Inflow 的外部记忆系统相当于为 ADHD 大脑提供持久化上下文，就像 Mem 的“AI 第二大脑”一样。

## 但有人用错了：脚手架 vs 拐杖

然而，同构视角也揭示了陷阱。在 LLM 侧，过度依赖 function calling 会导致 agent 丧失自主规划能力——一旦工具不可用，agent 就瘫痪。同样，ADHD 用户如果过度依赖 Inflow 这样的工具，可能削弱内在的执行功能能力。

这里的关键区别在于：**脚手架 vs 拐杖**。真正的脚手架应帮助使用者“建造”，但使用者仍需自己“攀登”（来源：拐杖与脚手架）。例如，Otter.ai 减轻笔记负担，但过度依赖可能削弱主动记笔记的能力（来源：ADHD 的 AI 工具全景）。Inflow 的设计初衷是“补偿”而非“替代”，但实际使用中很容易沦为“拐杖”——用户可能只依赖工具提醒，而不训练自己的时间感知。

矛盾与存疑也提醒我们：AI 工具作为外部执行功能层的有效性证据仍以定性为主，缺乏大规模随机对照试验（来源：矛盾与存疑）。部分工具如 Brain.fm 的 ADHD 效果缺乏独立临床研究（来源：Brain.fm），而 Structured 的 AI 功能细节尚不明确（来源：Structured）。个体差异也很大，效果因人而异（来源：矛盾与存疑）。

Inflow 之于 ADHD，就像 function calling 之于 LLM——它是强大的外部调度层，但前提是你用它来搭建脚手架，而不是永久拐杖。
