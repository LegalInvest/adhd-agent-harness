# 3 个 LLM 工程概念，彻底重写了我对 ADHD 不擅长精确、重复、状态维护类任务的理解

> 报销单、周报、把文件放回原处——毁掉 ADHD 职业声誉的从来不是难的事，而是这些「简单」的事。LLM 工程对此有一个冷酷而解脱的解释。

我认识的每一个 ADHD 成年人，都有一份高度雷同的「翻车清单」：忘交报销单、表格填错行、订阅忘了退、同一封邮件三次忘记回。这些事没有一件是难的，所以每次翻车都会被（自己和别人）归因为「不上心」。直到我看到 LLM 工程师怎么谈论他们的模型，才意识到这个归因错在哪里。

## 1. 生成器不是状态机：架构决定了短板的位置

LLM 是概率生成器：它擅长联想、发散、在模糊问题上给出惊艳答案；但让它精确复述一串发票号、维护一个多步骤流程的中间状态、把同一件事严格重复一百遍——它会出错，而且是系统性地出错。工程师对此的反应很有启发性：没有人骂模型「不上心」，他们直接得出结论——**生成器不该干状态机的活**，然后把精确性外包给代码解释器、把状态外包给数据库、把重复外包给循环脚本。

ADHD 大脑是同一种架构。多巴胺调节的奖赏系统决定了它只为新奇、紧迫、有趣的信号分配资源，而「精确、重复、状态维护」恰好三项全不占。这不是性格缺陷，是架构属性——就像 LLM 会幻觉不是它「品行不端」。

## 2. 工具调用：识别「该出让的任务」是一种核心能力

agent 工程最优雅的设计是 function calling：模型自己判断「这一步我不该亲自算」，转而调用计算器、数据库、日历。注意——**知道何时调用工具，本身就是智能的一部分**。

这重写了我对「依赖工具」的羞耻感。ADHD 版的 function calling 清单：数字类交给自动扣款和报销 App、流程类交给 Goblin Tools 拆出的检查清单、记忆类交给 Saner.AI 这类外部索引。孔子说自己「述而不作」，却整理出六经——他把创造性生成和结构性维护清楚地分开处理。李白从不管账，但他身后有整个时代的抄写、编集系统替他维护状态；没有那套 harness，斗酒诗百篇也只是散佚的口头即兴。

## 3. 幂等与检查清单：给不可靠的执行器兜底

工程师假设每一步都可能失败，于是设计幂等操作（重复执行不出错）和显式校验点。这个「不信任内核」的设计哲学，恰恰是对内核最大的善待——它把可靠性从「每一步都别出错」降低为「出错了也能被接住」。

ADHD 生活的幂等化改造完全可行：所有账单自动支付（执行零次和一次结果相同）、钥匙钱包只允许存在于同一个碗里（状态收敛到单点）、周报用固定模板只填空不创作（把生成任务降级为查表任务）。每一项都在减少「必须靠当天状态在线才能完成」的环节。

## 边界

同构强度说明：这是 B 级机制类比——双侧都有真实文献支撑（ADHD 侧的奖赏系统研究、LLM 侧的可靠性工程），但「大脑=LLM」不成立。另一个诚实的提醒：外包状态维护不等于可以不知道自己的状态在哪里维护——你至少要维护一张「东西都外包给了谁」的总表，那是不可再外包的最后一层。

## 今天就能试的 3 件事

1. **列出你的翻车清单**：把过去一年所有「简单却翻车」的事写下来，逐条问：这是生成任务还是状态维护任务？
2. **做一次 function calling 改造**：挑清单里出现频率最高的一件，今天就把它接到一个自动化工具上。
3. **幂等化一个日常**：选一件反复出错的事，重新设计成「忘了也不炸」的形态。

「简单的事都做不好」是对 ADHD 最深的误解。正确的表述是：那些事对这种架构来说从来不简单——而正确的回应不是更努力，是换执行器。
