# Reflect 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Reflect：它到底补上了哪一层执行功能？

## 问题：ADHD 与 LLM 共享同一个核心缺陷

ADHD 大脑和 LLM 有一个惊人的同构：两者都是“高产但缺执行调度层的生成核心”（来源：ADHD 大脑与 LLM 的同构）。ADHD 侧，想法喷涌但启动困难、工作记忆易丢、时间感知失灵；LLM 侧，能生成流畅文本，但一旦涉及精确计算、多步状态维护、重复格式输出，就频频出错。两边的解法也同构：给生成核心套上 harness（脚手架），把“不擅长”的任务外包给工具。

对 LLM 而言，这个 harness 是 function calling——通过工具调用把精确计算、数据库查询、状态维护交给外部函数，LLM 只负责生成意图和参数。对 ADHD 而言，这个 harness 是一类特定的 AI 工具——它们不是帮你“想”，而是帮你“做”：分解任务、记住上下文、提供下一步提示。

但问题来了：很多人用错了 Reflect。

## Reflect 被误用的方式：当笔记工具而非反思脚手架

Reflect 是一款 AI 驱动的笔记和反思工具，其核心是“结构化反思”——通过提问引导用户回顾、整理、连接想法。但许多用户（包括 ADHD 群体）把它当成又一个笔记 App：记录会议、保存灵感、整理资料。这本质上是在用 LLM 的生成能力做“记忆存储”，而非执行功能补偿。

ADHD 真正需要的不是“记下来”，而是“下一步做什么”。Reflect 的结构化反思，如果只停留在“记录”，就错过了它的核心价值——作为外部执行功能层，提供“当下注意什么”的主动设计（来源：ADHD 的 AI 工具全景）。

## 正确用法：Reflect 作为反思型 function calling

LLM 的 function calling 不是让 LLM 记住所有函数细节，而是让 LLM 在需要时调用外部工具完成精确任务。同样，Reflect 的正确用法不是让 ADHD 用户记住所有反思内容，而是让 Reflect 在关键时刻主动提问，引导用户完成“状态维护”和“下一步规划”。

例如：
- 每次完成小任务后，Reflect 自动弹出：“刚才完成了什么？下一步最需要做什么？”——这相当于为 ADHD 大脑提供了一个“状态更新函数”。
- 每天结束时，Reflect 引导用户回顾：“今天哪件事的启动最困难？明天如何降低那个门槛？”——这相当于一个“冷启动优化函数”。
- 当用户连续三天未记录某项目进展，Reflect 主动提醒：“这个项目上次更新是三天前，当前状态是什么？”——这相当于一个“上下文持久化函数”。

这些用法直接补偿 ADHD 的执行功能缺陷：工作记忆（Reflect 提供持久化上下文）、任务启动（Reflect 的提问降低启动门槛）、时间盲（Reflect 的定时回顾提供时间线索）（来源：ADHD 大脑与 LLM 的同构）。

## 证据：同构视角下的工具验证

ADHD 侧，已有工具验证了类似思路：
- Goblin Tools 的 Magic ToDo 自动将大任务分解为小步骤，用户报告称“启动焦虑降低”（来源：Goblin Tools）。这相当于为 ADHD 大脑提供了一个“任务分解函数”。
- Lex 的单一指令触发多步骤序列，减少了决策疲劳（来源：Lex）。这相当于一个“批量执行函数”。
- Saner.AI 的增强回忆功能，减少搜索循环和标签切换（来源：Saner.AI）。这相当于一个“上下文检索函数”。

LLM/agent 侧，agent harness 的典型架构是“每个工作流可绑定不同 LLM”，并通过模型连接器分配任务（来源：ADHD 大脑与 LLM 的同构）。这本质上就是通过外部工具（函数）来补偿 LLM 的调度缺陷。

Reflect 的结构化反思，如果设计成可配置的“反思函数”（用户自定义触发条件、提问模板、输出格式），就能成为 ADHD 大脑的通用 harness——不是记录一切，而是在需要时主动提供执行功能支持。

## 脚手架 vs 拐杖：争议与局限

必须诚实指出，这种用法存在争议。

首先，过度依赖 AI 工具可能削弱内在能力（来源：矛盾与存疑）。如果 Reflect 总是替用户完成“下一步规划”，用户可能永远不会训练自己的前瞻记忆。关键在于将 Reflect 视为脚手架——逐步减少外部提示，直到用户内化反思习惯。

其次，现有证据主要来自用户报告，缺乏大规模对照实验（来源：矛盾与存疑）。Reflect 的结构化反思是否比简单的待办清单更有效？目前没有严格验证。

第三，个体差异大。有些 ADHD 用户可能对定时提问感到烦躁，反而增加认知负荷。工具设计者声称是“脚手架”，但实际使用中可能沦为“拐杖”（来源：矛盾与存疑）。

## 今天就能试的行动

1. **设置一个“状态更新”反思模板**：在 Reflect 中创建模板，每次完成小任务后手动触发（或利用 Reflect 的定时提醒），回答：“刚做了什么？下一步最需要做什么？有什么障碍？”坚持三天，观察启动焦虑的变化。

2. **配置“冷启动”反思**：每天开始工作前，让 Reflect 弹出：“今天最重要的一个任务是什么？第一步是什么？”完成后记录感受。这相当于为 ADHD 大脑提供启动脚本。

3. **利用 Reflect 的链接功能**：将相关任务、笔记、项目通过链接关联，形成“上下文网络”。当 Reflect 的 AI 自动推荐相关笔记时，注意它是否帮你减少了“信息丢失”的焦虑（来源：Saner.AI 的知识回忆功能）。

4. **反思使用模式**：一周后，回顾 Reflect 的使用记录。你是更多在“记录”还是“反思”？如果记录占主导，尝试减少记录频率，增加反思提问。

## 结论

Reflect 之于 ADHD，就像 function calling 之于 LLM——关键不在于工具本身，而在于你用它来做什么。用对了，它是帮你补上执行功能层的脚手架；用错了，它只是又一个数据坟场。同构视角告诉我们：ADHD 大脑和 LLM 都不需要更多的“生成能力”，而是需要一个可靠的“调度层”。Reflect 的结构化反思，正是这个调度层的雏形——但前提是你把它当成函数来调用，而不是当成数据库来填充。
