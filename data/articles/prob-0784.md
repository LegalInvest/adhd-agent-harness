# Habitica 之于 ADHD，就像 function calling 工具调用 之于 LLM——但有人用错了

> 从同构视角实测 Habitica：它到底补上了哪一层执行功能？

## 问题：为什么 Habitica 对有些人有用，对更多人没用？

如果你是一个 ADHD 患者，你大概试过 Habitica——那个把生活变成 RPG 的 App。你建了角色，接了每日任务，打怪升级，然后……两周后彻底弃坑。你责怪自己缺乏毅力，但问题可能不在你，而在工具的设计逻辑。

如果你是一个在做 Agentic Harness 的工程师，你大概遇到过类似场景：给 LLM 接上 function calling，让它能调用计算器、数据库、API，结果它要么疯狂调用不必要的工具，要么在简单任务上反复出错。你调试半天，发现是 prompt 没写对，或者工具接口设计得太重。

这两个场景背后是同一个结构问题：**一个高产的生成核心（ADHD 大脑 / LLM）缺少一个可靠的执行调度层，而外部工具本应充当这个调度层，但人们常常误用了它。**

## 同构视角：Habitica 到底补上了哪一层执行功能？

ADHD 大脑与 LLM 在结构上同构：两者都是强大的生成核心，但缺乏稳定的执行调度层（来源：ADHD 的 AI 工具全景）。ADHD 患者不擅长精确、重复、状态维护类任务——比如按时吃药、整理桌面、记录时间。LLM 同样不擅长精确计算、多步验证、状态跟踪——比如算 3.14×2.71 会出错，或者在一个长对话中忘记上下文。

解决方案也是同构的：给生成核心套一个 harness（脚手架）。对 LLM 来说，这个 harness 就是 agent 框架，通过 function calling 把精确计算外包给工具（如计算器、数据库），同时提供验证循环来纠正幻觉（来源：幻觉与验证循环）。对 ADHD 来说，这个 harness 就是外部工具，通过任务分解、记忆存储、时间提醒来补偿执行功能缺陷（来源：Goblin Tools、Saner.AI）。

Habitica 本质上就是一个 harness。它把“保持健康”这种模糊目标拆解为“喝一杯水”“做 10 个俯卧撑”等具体动作（类似 Goblin Tools 的 Magic ToDo 功能），并通过经验值、等级等即时反馈维持动机。但它的核心价值不是“游戏化”本身，而是**提供了一个外部调度接口，把需要内在执行功能的任务（如启动、切换、跟踪）外包给了系统**。

这就像 LLM 通过 function calling 把“计算 3.14×2.71”外包给计算器——不是让 LLM 变聪明，而是让它不用自己算。同样，Habitica 不是让 ADHD 患者变自律，而是让他们不用自己调度。

## 为什么有人用错了：脚手架 vs 拐杖

问题在于，很多人把 Habitica 当作动机引擎，而不是调度接口。他们期待游戏化本身能产生多巴胺，驱动行为改变。但 ADHD 的多巴胺系统本身是失调的，外部奖励的效果因人而异，且容易产生耐受（来源：矛盾与存疑）。如果工具只提供奖励而不提供调度，它就成了拐杖——你依赖它获得动力，但内在执行功能并未改善，一旦工具失效（比如忘记打卡），整个系统崩溃。

真正的脚手架应该是可逐步撤除的临时支撑（来源：矛盾与存疑）。Habitica 的设计没有考虑撤除——它假设你需要永远使用它。而像 Goblin Tools 这样的工具，通过任务分解降低启动门槛，但用户可以在学会分解模式后逐渐减少依赖。同样，LLM harness 中的 function calling 应该是可选的，agent 应该在验证循环中学会自己校验输出，而不是永远依赖外部计算器。

另一个常见错误是：把 Habitica 当作万能解决方案。ADHD 的异质性很高，单一工具难以覆盖所有亚型（来源：矛盾与存疑）。有些人需要时间提醒（如 Goblin Tools 的魔法待办），有些人需要知识回忆（如 Saner.AI），有些人需要单一指令触发多步骤（如 Lex）。Habitica 的 RPG 模式可能只适合那些对游戏化敏感的人，而对其他人来说，它反而增加了认知负荷。

## 证据：两边都有真实案例

**ADHD 侧**：用户反馈显示，Goblin Tools 的 Magic ToDo 能“将压倒性的事情变成一系列不压倒性的事情”（来源：Goblin Tools），这与 Habitica 的任务分解功能类似。Saner.AI 通过本地记忆减少搜索循环，直接补偿工作记忆缺陷（来源：Saner.AI）。Lex 通过单一指令触发多步骤任务，降低启动门槛（来源：Lex）。这些工具的共同点是：它们不是提供动机，而是提供调度。

**LLM/agent 侧**：Harness 工程的核心是“验证与 CI 集成”和“结构化工作流与规划”（来源：幻觉与验证循环）。复杂的 harness 不会盲目执行工具，而是实现验证步骤，例如检查输出格式或对代码运行测试用例（来源：同上）。缺乏这些验证会导致“更多幻觉和重复工作”（来源：同上）。这正好对应 ADHD 侧：如果工具只提供奖励而不提供验证（如 Habitica 仅靠经验值反馈），用户容易陷入“为了打卡而打卡”的循环，而非真正完成任务。

## 局限与争议

必须诚实指出：ADHD 大脑与 LLM 同构命题目前主要基于概念类比和工具案例，缺乏大规模实证（来源：矛盾与存疑）。Habitica 针对 ADHD 的效果缺乏独立随机对照试验，证据主要来自用户评价。此外，过度依赖工具可能阻碍内在执行功能发展，而现有工具大多未设计撤除机制（来源：矛盾与存疑）。对于某些人，Habitica 可能只是另一种待办清单，反而增加了压力。

## 今天就能试的行动

1. **如果你是 ADHD 患者**：打开 Habitica，但只设 3 个最困扰你的重复性任务（如“吃早餐”“喝水”“睡前收拾桌面”），不设任何长期目标。观察一周：是完成任务的成就感驱动你，还是系统提醒本身帮你启动了任务？如果是后者，说明你需要的是调度接口，而非动机引擎。可尝试用 Goblin Tools 的 Magic ToDo 替代 Habitica 的任务分解部分。

2. **如果你是 Agent 工程师**：检查你的 function calling 设计是否提供了验证循环。例如，调用计算器后是否检查结果格式？是否在工具返回错误时重试或回退？如果只依赖 LLM 的自信输出而不验证，你就在犯和 Habitica 用户同样的错误——把工具当拐杖，而非脚手架。

3. **两类人都可以**：尝试用 ChatGPT 或 Claude 模拟一个“执行功能教练”。给它一个你拖延的任务，让它分解成 5 步以内，并指定每步的完成时间。这相当于临时搭建一个 harness，用完即撤，避免长期依赖。对比使用前后，你的启动时间是否缩短。

4. **反思“工具依赖”**：问自己：如果这个工具明天消失，我还能完成同样的任务吗？如果答案是否定的，说明你已形成依赖。尝试每周减少一次使用，观察内在执行功能是否有所改善。这是检验工具是脚手架还是拐杖的试金石。
