# 当我的执行功能崩溃时，比如无法启动一项任务，agent 的编排系统是如何模拟这种状态并尝试干预的？

> 这个问题问得极好,因为它反着问:不是「AI 怎么帮我」,而是「工程系统里,『启动不了』这个状态长什么样、系统怎么处理」——把自己的崩溃当成一个可以被第三方视角观察的系统事件。这一篇就老老实实做这个技术对照:**agent 编排系统里与「无法启动」同构的四种故障,以及每种故障的标准干预**——然后你会发现,这四种干预翻译回人类侧,恰好是 ADHD 启动困难的四种最有效的应对。

**故障一:任务过大导致的规划失败(planning failure)。** 给 agent 一个「重构整个代码库」级别的指令,常见结果不是干得慢,是**根本不开始**——生成一堆元讨论、反复重写计划、或直接停滞;编排系统的诊断是任务粒度超出了单步可执行范围。**干预:强制分解**——编排层把任务递归拆到「每步都有明确的第一个动作」为止,才允许进入执行队列。人类侧的同构精确:「写年度报告」在你的执行系统里就是那个无法进入队列的巨型任务——**盯着它产生的不是行动而是元活动(焦虑、整理桌面、重列计划)**;干预同款:拆到第一步小于两分钟(「打开文档,写下标题」),启动的不是报告,是那两分钟。

**故障二:上下文缺失导致的冷启动挂起。** Agent 被唤起执行「继续昨天的任务」,但会话是新的、上下文是空的——它挂起在「不知道从哪继续」;编排系统的干预:**状态持久化+启动时注入**——上次的进度、决策、下一步,作为结构化状态存盘,新会话第一件事是加载它。人类侧:昨天思路火热的项目,今天坐下来大脑一片白——「我做到哪了?」这个检索本身就贵到足以劝退启动。干预同款:**离场快照**——每次停下前写三行(做到哪/下一步/当时的思路),下次启动从读快照开始,冷启动变热启动(程序员篇的「失败的测试」是同一招)。

**故障三:预期成本过高导致的调度回避。** 精细的编排系统会做成本预估,预估过高的任务会被持续调度到队尾——**表面是「还没轮到」,实际是永远轮不到**。人类侧这叫任务厌恶(task aversion):那个「想起来就烦」的任务在心理账户上被标了天价(混合着评价焦虑、上次的失败记忆、纯粹的无聊),于是被无限延后——而 ADHD 的延迟折扣让「延后」的代价在体感上约等于零。干预:工程上是**重估机制**(强制试执行一小步,用真实成本覆盖预估)——人类版:「烂开头十分钟」协议:只承诺做十分钟,允许烂——**十分钟后的真实数据几乎总是低于预估**,而重估之后任务经常自己滑进执行队列(「其实还好」是启动困难最常见的结案陈词)。

**故障四:唤醒/资源不足,队列整体停摆。** 有时不是某个任务的问题,是整个执行引擎低电量——agent 侧对应资源耗尽(配额、算力),编排系统不会「鼓励」agent 努力,**它做资源管理**:降级模式、只跑关键任务、等资源恢复。人类侧:睡眠不足、低血糖、情绪透支时的全面启动瘫痪——此时一切任务技巧都失效,因为故障不在任务层。干预同款,而且这是最常被 ADHD 者搞错的一条:**先修资源,不修任务**(吃东西、小睡、散步),并把当天队列砍到只剩关键项——在低电量状态死磕启动技巧,等于对着断电的服务器改代码。

四种故障,四种干预,一个共同点:**编排系统从不把停滞归因为 agent 的「态度」,它诊断故障类型,然后按类型干预**。这就是这一篇真正想给你的东西——下次启动不了时,先问:今天是哪一种?

## 边界

同构强度 B 级:四种编排干预是真实的工程模式,任务分解、实施意图、行为激活在人类侧各有证据支撑;「四型对照」是组织框架,真实的启动困难常是混合型。声明:持续数周的全面启动瘫痪、伴随情绪低落与兴趣丧失,要优先排查抑郁(它和 ADHD 高共病且症状重叠)——那是临床问题,不是调度问题;药物对启动困难的作用请与医生讨论。

## 今天就能试的 3 件事

1. **给当前卡住的任务定型**:过大?冷启动?预期成本虚高?低电量?——四选一,按型干预。
2. **给一个项目做离场快照**:三行——做到哪/下一步/思路——贴在下次会看到的地方。
3. **签一次十分钟协议**:挑那个「想起来就烦」的,只做十分钟,允许烂——十分钟后重估。

编排系统对付停滞的全部智慧就一句:**先诊断故障类型,再选干预,永远不审判引擎**。你也值得被这样对待——尤其是被你自己。
