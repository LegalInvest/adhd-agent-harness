# ADHD 学生如何用「给生成核心套 harness」的思路应对考试与作业里的怕过度依赖工具、又怕没有支撑就崩？

## 问题：怕依赖，又怕没支撑就崩

你是一名 ADHD 学生，面对考试和作业时，内心反复拉扯：用 AI 工具吧，怕它变成拐杖，让自己越来越不会独立思考；不用吧，又怕自己崩盘——任务启动困难、时间盲、情绪波动，每一项都像悬在头顶的剑。这种「怕过度依赖工具、又怕没有支撑就崩」的焦虑，本质上是一个工程问题：如何设计一个会褪去的脚手架，而不是永久拐杖？

## 同构：ADHD 大脑与 LLM，同一类生成核心

ADHD 大脑与 LLM 在结构上惊人相似：两者都是高产但缺乏可靠执行调度层的生成核心（来源：ADHD 大脑与 LLM 的同构）。ADHD 个体的情绪失调类似于 LLM 的上下文失控——当工作记忆过载或注意力被干扰时，情绪反应如同 LLM 的幻觉，脱离理性轨道（来源：AI 与 ADHD 的情绪调节）。LLM 天然倾向于产生自信但可能错误的输出（幻觉），而 harness 工程正是为此设计：它提供上下文传递、工具接口、规划工件、验证循环、记忆系统和沙盒（来源：幻觉与验证循环）。

ADHD 一侧的证据同样清晰：工作记忆缺陷与情绪调节困难高度相关，AI 工具通过外部记忆（如情绪日志）缓解此问题；时间盲导致情绪持续时间被高估，AI 视觉化时间工具（如计时器）帮助客观化情绪体验（来源：AI 与 ADHD 的情绪调节）。这些工具本质上是在给 ADHD 大脑套上一个外部执行调度层——就像 LLM 的 harness。

## 核心判断：脚手架 vs 拐杖的边界在于「是否内化」

我的核心观点是：脚手架与拐杖的边界不在于工具本身，而在于使用方式。如果工具只是替代你完成所有执行调度，它就是拐杖；如果工具帮助你逐步内化这些调度能力，它就是脚手架。

LLM/agent 侧的证据表明，复杂的 harness 不会盲目执行工具，而是实现验证步骤，例如检查输出格式或对模型编写的代码运行测试用例（来源：幻觉与验证循环）。这种「验证循环」正是脚手架的关键——它不替模型思考，而是帮模型检查自己的思考。同理，ADHD 学生需要的不是替自己完成所有任务的工具，而是帮自己建立「验证循环」的工具。

例如，Goblin Tools 的 Magic ToDo 功能自动将任务分解为小步骤（来源：Goblin Tools），这可以是一个脚手架——前提是你在使用后尝试自己分解类似任务，而不是永远依赖它。Saner.AI 通过本地记忆减少搜索循环（来源：Saner.AI），但如果从不训练自己的记忆检索，它就成了拐杖。Inflow 通过训练工作记忆和认知控制来改善 ADHD 症状（来源：Inflow），其设计本身就包含内化目标——强化背外侧前额叶皮层作为因果流入中枢的功能（来源：Dynamic causal brain circuits during working memory and their functional controllability）。

## 争议与局限：警惕「伪脚手架」

必须诚实指出，现有证据主要来自用户报告和概念类比，缺乏大规模对照实验（来源：矛盾与存疑）。多个工具页面（如 Goblin Tools、Saner.AI）都提到缺乏独立随机对照试验。过度依赖 AI 工具可能削弱内在能力——例如，过度依赖外部时间管理可能削弱内在时间感知能力（来源：矛盾与存疑）。此外，个体差异导致效果不一：ADHD 亚型不同，对 AI 工具的反应差异大（来源：AI 与 ADHD 的情绪调节）。

因此，选择工具时需警惕「伪脚手架」——那些声称帮你但实际替代你思考的工具。真正的脚手架应该像 LLM harness 中的验证循环：提供结构，但要求你参与验证。

## 今天就能试的行动

1. **任务分解脚手架**：下次面对复杂作业时，先用 Goblin Tools 的 Magic ToDo 分解任务，然后尝试自己手写一份类似的步骤清单，对比差异。这能训练你的执行功能，同时避免依赖。（来源：Goblin Tools）

2. **情绪验证循环**：当情绪波动时，用计时器设定 5 分钟，记录自己的情绪触发点和持续时间。这类似于 LLM 的上下文工程——主动设计注意焦点，实现情绪的可预测性。（来源：AI 与 ADHD 的情绪调节）

3. **记忆检索训练**：使用 Saner.AI 记录关键信息，但每周设定一天「无工具日」，强迫自己凭记忆检索。这能锻炼工作记忆，避免本地记忆变成拐杖。（来源：Saner.AI）

4. **计划-执行分离**：考试前，用纸笔写下答题策略（计划），然后执行时不看计划。这模仿了 LLM 工程中的「计划-执行分离」模式，强制单向、确定性的控制流（来源：采样温度与表现波动）。

## 结语

ADHD 学生与 LLM 工程师面对的是同一个问题：如何让一个高产的生成核心稳定输出，而不被自身的随机性拖垮。答案不是抛弃工具，也不是永久依赖，而是设计会褪去的脚手架——在需要时提供支撑，在能力内化后悄然退场。下次当你害怕工具变成拐杖时，问问自己：这个工具是在帮我建立验证循环，还是在替我跳过思考？
