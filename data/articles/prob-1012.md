# Otter.ai 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> 从同构视角实测 Otter.ai：它到底补上了哪一层执行功能？

你有没有过这样的经历：开会时拼命用 Otter.ai 录音，结束后看着满屏文字却不想回看，下次开会依然忘光重点？或者，你是一个 Agent 工程师，发现 LLM agent 在长任务中逐渐偏离初始指令，就像 ADHD 大脑在任务中不知不觉滑向无关细节？

这两个场景共享同一个底层问题：**目标漂移**。ADHD 大脑和 LLM agent 都是“高产但缺乏可靠执行调度层的生成核心”（来源：AI 与 ADHD 的专注力）。它们能生成海量内容，却无法自动锚定目标。Otter.ai 常被当作“录音笔”来用——这就像给 LLM agent 一个巨大的上下文窗口却不设 system prompt 重锚定，结果注定是漂移。

## 问题：重锚定缺失，工具变拐杖

ADHD 一侧，分心走神或超聚焦跑偏是常态。外部工具通过“提醒和通知”提供重锚定，将注意力拉回原任务（来源：重锚定与目标漂移）。但 Otter.ai 的典型用法——录音、转写、存档——只完成了“外部记忆”这一步，没有触发重锚定。用户回看转写内容的概率极低，因为工作记忆缺陷和启动困难让“回顾”本身成为另一项任务。结果，Otter.ai 变成了一个昂贵的录音机，而非执行功能助手。

LLM/agent 一侧，模型在长程任务中“开始忘记系统提示”（来源：重锚定与目标漂移），行为偏离初始指令。工程师的做法是事件驱动的系统提醒——在关键节点重新注入 system prompt。如果只给 agent 一个超长上下文而不做任何重锚定，它同样会漂移。

## 同构答案：重锚定系统才是核心

Otter.ai 的真正潜力不在于记录，而在于**将转写内容转化为可检索、可触发的重锚定信号**。ADHD 用户需要的不只是“录下来”，而是“在需要时自动弹出关键信息”。这对应 LLM/agent 的 harness 工程：通过人在回路监督、护栏和检查点来保持目标一致（来源：人在回路与身体在场）。

证据显示，有效的 ADHD 工具都具备重锚定机制：Focusmate 通过实时同伴问责提供外部结构（来源：Focusmate）；RescueTime 通过自动时间追踪对抗时间盲（来源：RescueTime）；Brain.fm 通过神经锁相音乐帮助进入专注状态（来源：Brain.fm）。这些工具的共同点是**主动干预注意流**，而非被动记录。

Otter.ai 若想成为真正的重锚定工具，必须做到：
1. **结构化摘要**：自动提取会议关键决策和待办事项，而非全文转写。
2. **定时回顾提醒**：在任务执行中弹出相关会议片段。
3. **与任务管理集成**：将会议要点直接推送到待办列表。

## 脚手架 vs 拐杖：边界在哪里？

但这里有一个关键争议：过度依赖外部重锚定是否会削弱内在执行功能？资料指出，“长期使用 AI 工具可能导致内在执行功能进一步退化”（来源：矛盾与存疑）。Focusmate 依赖实时视频，可能对社交焦虑用户造成压力（来源：Focusmate）；RescueTime 的自动分类可能不够精准，需要用户调整（来源：RescueTime）。

脚手架与拐杖的边界在于：**工具是否在逐步培养用户的自主能力**。好的重锚定工具应该像 Focusmate 那样，让用户逐渐内化“对他人负责”的机制，而不是永远依赖提醒。Otter.ai 若只提供被动记录，就是拐杖；若能引导用户主动提取和复盘，就是脚手架。

## 今天就能试的行动

1. **用 Otter.ai 做“主动回顾”实验**：每次会议后，花 5 分钟用 Otter 的搜索功能找 3 个关键决策，并写进待办清单。不要只看全文。
2. **设置“重锚定提醒”**：在 Otter 转写中标记关键时间戳，用日历或任务工具设置 24 小时后的回顾提醒。
3. **尝试 Focusmate 替代方案**：如果 Otter 的被动记录让你失望，试试 Focusmate 的实时问责——用外部身体在场对抗漂移。
4. **对 Agent 做同样的事**：如果你在构建 LLM agent，在长任务中每 5 轮对话注入一次原始 system prompt 摘要，观察目标保持率的变化。

Otter.ai 之于 ADHD，就像重锚定 system prompt 之于 LLM——关键不在于记录本身，而在于**何时、如何将关键信息重新注入注意流**。用错了，它是拐杖；用对了，它是让你从漂移中归位的锚。
