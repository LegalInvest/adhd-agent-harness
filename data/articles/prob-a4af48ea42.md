# Perplexity 之于 ADHD，就像 上下文工程 之于 LLM——但有人用错了

> 一个 ADHD 的自由撰稿人接到一个陌生领域的稿约，打开 Perplexity 准备「快速建立认知」。两小时后他有了 47 条收藏、9 个追问线程和 0 个段落。他不是没找到信息——他被信息找到了：每个答案末尾的相关问题都像多巴胺的传送门，而他全都点了。

收敛：Perplexity 的价值主张是「替你压缩信息空间」，这恰好是上下文工程对 LLM 做的事。但有人用它越用信息越多。本文只回答——**同一个压缩工具，为什么在一些人手里是降噪器，在另一些人手里变成了噪声放大器？**

## 穿透：上下文工程的第一定律

LLM 侧的机制：上下文工程的核心不是「给模型更多信息」，而是**给模型恰好够用的信息**——无关内容会稀释注意力权重，上下文越塞越满，性能反而下降。好的 harness 在检索后有一道残酷的工序：过滤、排序、截断，只让最相关的片段进入窗口。压缩的本质是丢弃。

ADHD 侧的机制镜像：ADHD 的信息获取问题从来不是入口太少，而是**出口没有闸门**——兴趣驱动的注意力系统对「相关问题」这类新异性诱饵几乎零抵抗，于是检索行为本身变成了刺激来源。Perplexity 用对了是「一次检索、一个答案、关闭窗口」，用错了是把它当成了带引用的信息老虎机。区别不在工具，在**有没有预设的截断条件**——LLM 的上下文有硬性 token 上限强制截断，而人的检索没有上限，除非你自己造一个。

利益侧看一眼：「相关问题」推荐是产品的留存设计，它的 KPI 是你的停留时长，不是你的稿件完成度。

## 验证：证伪与反例

「截断条件决定成败」这个判断可自测（一周）：给每次检索预先写下两样东西——①这次要回答的唯一问题；②答案的容器（比如「三句话笔记」）。容器写满即强制关闭。对照组是平时的自由检索。一周后对比两种模式下的产出物数量。反例边界：真正的开放式探索期（选题阶段、灵感搜集）不适用截断协议——那时发散是目的不是故障；关键是诚实区分「我在探索」和「我在逃避写作」，判据是探索有没有截止时刻。

## 决策

做什么：把 Perplexity 用法固定为「问题进，容器出」——检索前写下唯一问题和答案容器，容器满即离场。

不做什么：不要点「相关问题」——那不是你的问题，是算法的问题；不要用「多了解一点没坏处」说服自己——对 ADHD，信息过量的坏处就是启动被无限推迟。

先做什么：下一次检索（可能就是今天）先写那一行唯一问题——写不出来，说明你还不需要检索，需要的是先想。

## 边界

「Perplexity↔上下文工程」是功能层类比（本文未做正式 A/B/C 分级）。Perplexity 不是 ADHD 治疗手段；若信息漫游已严重挤占工作与睡眠、自我控制反复失败，请考虑寻求专业评估，行为策略之外可能需要更系统的支持。

## 今天就能试的 3 件事

1. 建一个检索模板便签：「本次唯一问题：___；答案容器：三句话」——存在输入法快捷短语里。
2. 回头看今天的浏览记录，数一数有多少标签页与最初的问题无关——用数据看见自己的截断缺口。
3. 给 Perplexity 使用加一个物理计时器：15 分钟响铃即合盖——把 token 上限翻译成现实。

本文服务于人生 Harness 金字塔的**执行层**：检索工具的价值不在它能给你多少，而在你敢不敢在拿到够用的那一刻，关掉它。
