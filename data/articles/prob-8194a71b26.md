# 为什么用 RescueTime 治 ADHD 的注意力涣散，和给 agent 套 上下文工程 是一回事？

> 问一个 ADHD 者「你今天的时间去哪了」,得到的答案和实际记录对比,误差经常大到离谱——不是撒谎,是真不知道:涣散的本质就是注意的去向不经过自觉审批,所以事后也没有账本。RescueTime 这类自动时间追踪工具做的事,冷静得近乎无情:后台记录每个应用、每个网站的实际停留,周末给你一份「你以为 vs 实际」的对账单。它不拦截、不提醒、不激励——**它只做一件事:可观测性(observability)**。而这件事在 agent 工程里的地位,恰恰是所有上下文工程的前置条件:**没有 telemetry 的系统,一切优化都是盲调**。

先讲 agent 侧为什么把可观测性排在优化之前。Agent 的失败大多不在你盯着的地方:开发者以为是模型能力问题,日志一看,是检索喂了垃圾;以为 prompt 不够好,轨迹一放,是第三步之后上下文被历史残留挤爆——**所以成熟 harness 的第一投资是全链路追踪**(每次调用、每个 token 去向、每步耗时),先有数据,才谈得上改进;没有数据的改进是在改传说。这条工程铁律移到注意管理上,一字不用换:**你对自己注意去向的印象,是传说,不是数据**。

而 ADHD 的「印象」失真得更系统,有两个已被反复观察的机制:**①时间感知的偏差**——时间盲不只影响未来估计,也影响回溯(那个「刷了一下」的实际时长,主观感受可能只有五分之一);**②状态切换的不留痕**——涣散的每次跳转都不经过自觉决策(自下而上的捕获,定义如此),没有决策就没有记忆锚点,事后重建自然是空白。结果:ADHD 者对自己一天的叙述,经常是「莫名其妙就没了」——**不是形容,是字面事实:账本上真的没有记录**。在这样的地基上做时间管理,等于在没有日志的系统上调性能:你修的永远是想象中的瓶颈。

RescueTime 式追踪补的就是这个账本,它的三个输出各对应一类盲区:**①分类对账单**——各应用/网站的真实停留时长:第一次看的人几乎都会震惊(「YouTube 两小时四十分?」)——这份震惊本身就是干预:主观模型被数据校准了一次;**②模式发现**——几周数据会浮出你自己不知道的规律:几点开始崩、哪个 app 是你的头号漏点、会议后的半小时是不是必然涣散——**这些规律是所有上游工具(门禁/排程/信道)的选型依据**:Forest 该在几点上、深度工作块该躲开哪个时段,答案在数据里;**③无操作成本**——全自动后台记录,不需要你手动打卡(手动时间记录对 ADHD 是注定弃用的方案,又是「把系统建在最弱部件上」的反面教材)。

但可观测性有自己的坑,两条都要标:**其一,数据引发的自我攻击**——对账单对 ADHD 者可能是羞耻弹药(「看,我果然废物」),Inflow 篇的叙事层问题在这里复发;用法纪律:**看数据找模式,不找罪证**——正确问法是「哪个时段/哪个入口漏得最多,上什么工具」,不是「我怎么这么烂」;**其二,测量不是干预**——RescueTime 自己不改变任何行为(它是日志,不是控制器),指望「看到数据就自然变好」会失望;它的价值在给其他工具当瞄准镜——**先观测,后干预,用数据验证干预有没有用**(这最后一步,恰好就是自我实验篇的闭环)。

## 边界

同构强度 A 级:时间感知偏差与 ADHD 的自我监控困难有实验文献,可观测性作为 agent 工程的前置层是行业共识,「无日志不优化」的同构两侧扎实;RescueTime 无 ADHD 对照研究,A 判给机制。声明:自动追踪涉及隐私(尤其工作设备),配置前读清数据条款;若对账单持续引发强烈自我攻击,先处理叙事层再看数据。

## 今天就能试的 3 件事

1. **开一周静默记录**:装上自动追踪,一周内不看不改——先攒数据,别急着审判。
2. **周末做一次对账**:先凭印象写下「我以为的时间分配」,再开报告对比——校准误差就是你的盲区地图。
3. **用数据选一个干预**:找出头号漏点(app+时段),给它配对应工具(门禁/排程)——两周后用数据验证。

「时间莫名其妙就没了」不是修辞,是你的系统没开日志——**agent 工程的第一课不是优化,是先让一切可见**。看不见的漏,永远堵不上;看见了,一半就已经堵上了。
