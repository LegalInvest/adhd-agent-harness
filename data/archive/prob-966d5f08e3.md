# 为什么用 Structured 治 ADHD 的注意力涣散，和给 agent 套 上下文工程 是一回事？

> 上下文工程系列讲过五个维度(内容/时间/呈现/底噪/社会);Structured 这个看起来极简的日程 app,展示的是一个贯穿所有维度的架构决策:**串行化(serialization)**。它把一天呈现为一条自上而下的单列时间轴——不是看板、不是矩阵、不是多列日历,就是一条线:现在做这个,然后做那个。这个「简陋」恰恰是它对 ADHD 注意涣散的核心疗效,而它在 agent 工程里的对应物,是上下文工程最基础的一条纪律:**并发是污染之母,单线程是最便宜的净化**。

先讲 agent 侧为什么怕并发。把多个任务塞进同一个上下文窗口交错处理,是 agent 失败的经典配方:任务 A 的中间状态和任务 B 的指令互相稀释注意权重,模型开始张冠李戴(Reclaim 篇讲过执行隔离)——所以成熟编排的默认形态是**串行执行+显式切换**:一次只有一个活跃任务,切换时存档、清空、加载。不是因为并行不诱人,是因为**在有限注意的系统里,并行的真实代价是每条线都在给其他线交污染税**。

ADHD 侧的对应,藏在「涣散」的一个具体形态里:**多线程幻觉**。ADHD 的联想式认知天然倾向同时挂起多件事(做着 A 想到 B,顺手开个 C 的标签),主观感受是「我在推进好几件事」,客观结果是切换成本吃掉大部分产出——注意研究对任务切换代价的测量一致显示:每次切换都有重启开销,而 ADHD 的切换恢复更慢(目标维持篇);更隐蔽的是,**每条挂起的线程都在工作记忆里持续占位**(认知负荷篇的后台占用)。「忙了一天什么都没完成」的谜底,多数时候就是这笔没人记账的并发税。

Structured 的设计,逐条对应串行化工程:**①单列时间轴=强制串行**——界面上根本不存在「同时进行」的表达方式,一天被编译成一条执行队列,这不是功能少,是**用界面语法禁止并发**(正如单线程语言用语法消灭数据竞争);**②「现在」的唯一高亮**——当前时段的任务是屏幕上唯一的焦点(Tiimo 篇的单点显著性),等于给执行器一个明确的「活跃任务指针」;**③块与块之间的显式边界**——上一块结束、下一块开始,切换是有仪式的(勾掉→看下一个),对应 agent 切换时的「存档-清空-加载」,而不是任务间的无边界渗漏。

它的 ADHD 增益还有一层容易被漏看:**串行化把「选择」从执行时刻移除了**。涣散的高发点之一是「做完一件事,下一件做什么?」——这个开放选择对 ADHD 是漂移窗口(挑最有趣的而不是最该做的,或者干脆在选择中瘫痪);时间轴把选择前置到了排程时刻,执行时只剩「照队列走」——**决策与执行分离,正是 planner-executor 架构的雏形**(这个话题下一批展开)。

边界两条:**其一,串行计划的脆性**——一天被一条线排满,前面一块崩了(它会的),后面全线连坐,容易触发全有全无的放弃;解法是块间留缓冲、并接受「重排不是失败」(Reclaim 篇的自愈逻辑,这里要手动做)。**其二,串行化不解决点火**——队列告诉你现在该做什么,不负责让你开始做;启动仍是另一个模块的事(启动篇的四种故障)。

## 边界

同构强度 B 级:任务切换代价与多任务的注意成本有厚实实验文献,agent 的串行执行与上下文隔离是工程实践,「串行化」的同构是清晰的功能对应;Structured 无 ADHD 对照研究,本文为机制分析。声明:日程工具是结构辅助;若一天的失控伴随情绪与睡眠的全面紊乱,评估优先于工具。

## 今天就能试的 3 件事

1. **把明天编译成一条线**:所有待办按时间排成单列(Structured 或纸笔)——禁止「同时」,只许「然后」。
2. **数一次并发税**:今天每次任务切换记一笔——晚上看看你付了多少次重启开销。
3. **给队列留两块缓冲**:计划里插入两段 30 分钟空白——前面崩了,在缓冲处重排,不许弃赛。

「我在同时推进五件事」是 ADHD 最贵的幻觉——**有限注意系统里,并行不是能力,是税**。一条单列时间轴的全部智慧就是:一次一件,然后下一件。
