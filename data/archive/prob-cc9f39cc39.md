# 为什么用 Structured 治 ADHD 的任务启动困难，和给 agent 套 function calling 工具调用 是一回事？

> function calling 系列收官篇,讲一个前五篇都绕着走的问题:调用和调用之间的东西。打包、递送、schema、环境、协议——每一篇都在保障「单次调用」的发生。但一天不是一次调用,是几十次调用组成的序列,而 ADHD 的一天最常见的崩法,不是某次调用失败,是**序列本身的断裂**:上午十点那次失败的启动,让十一点、十二点、整个下午像多米诺一样连环塌掉。Structured 这类时间轴 App 真正值钱的,是它维护的那条「序列」。

先看工程侧怎么处理调用序列。agent 执行多步任务时,单个工具调用失败是常态,**框架设计的重点从来不是「杜绝失败」,是「失败的隔离」**:每次调用有自己的边界,失败被捕获、记录,然后**控制流跳到下一步或重试逻辑——绝不允许一次失败污染整个执行链**。没有这层隔离的系统,第三步报错,后面七步全部搁浅,一次小故障变成整链崩溃。**序列的韧性,不取决于单步成功率,取决于失败的传播是否被切断。**

ADHD 的一天恰恰缺这层隔离。十点的启动失败后,发生的是一连串心理传播:「上午已经废了」→「今天节奏全乱了」→「干脆算了,明天重来」——**一次调用失败,通过「全天叙事」这个共享状态,污染了所有后续调用**。心理学里这叫「破堤效应」(见前面批次:一颗薯片破戒后干脆吃完整包),ADHD 的全或无思维让传播速度更快。注意病灶在哪:不是十点那次失败(失败很正常),是**失败与失败之间没有防火墙。**

Structured 的时间轴形态,恰好在结构上内置了防火墙。它把一天呈现为**一列离散的、边界清晰的块**:9:00 晨会、10:00 写报告、11:30 回邮件……每块独立存在、独立开始、独立结束。这个「离散化」看起来平平无奇,但对失败传播是关键的:**块和块之间有硬边界,一个块的失败在视觉上就停在那个块里**——10:00 的块塌了,时间轴上 11:30 的块原封不动地立在那里,带着自己的图标和自己的开始时刻,像什么都没发生过一样等你。**「今天全毁了」在一条离散时间轴上,是一个没有视觉对应物的叙事**——这就是结构对叙事的胜利。按「失败隔离」的逻辑用它:

**一、把一天切成互不依赖的块。** 排日程时检查依赖:尽量别让下午的块「必须建立在上午的产出上」——**耦合的序列,一处塌全线塌;解耦的序列,塌几块照样过完一天**。做不到全解耦,至少把「今天最重要的一块」排成不依赖任何前序的。

**二、失败的块,只做一件事:记一笔,翻页。** 块没启动/没做完,在块上标记一下(Structured 里勾不掉就让它过去),**然后眼睛看下一个块**——禁止当场复盘、禁止重排全天、禁止「找回损失」式的赶工(那是失败在向后传播的三种伪装)。复盘统一放在晚上,失败日志攒着一起看。

**三、每个块的开始,是一次全新调用。** 11:30 的块响了——**它不知道 10:00 发生过什么,你也别告诉它**。新块的启动只需要新块自己的条件(参数、环境、点火),不需要「先消化掉上午的挫败」这个前置任务。这是整套方法最反直觉也最有效的一条:**允许下一次启动带着上一次的失败直接发生,内疚可以晚点再处理,启动不等它。**

**四、晚上看「块完成率”,别看「完美天数」。** 一天十块完成六块=正常且不错的一天(工程里 60% 的单步成功率配上好的隔离,任务照样完成);**用完成率替代「今天行/今天废」的二元叙事**,数据会教你破堤效应有多撒谎。

## 边界

同构强度 B 级:失败隔离与控制流设计是真实的工程范式,全或无思维与破堤效应有心理学文献,ADHD 的情绪对执行的级联影响有研究支撑,Structured 本身无对照研究。声明:若「一次失败全天崩」伴随强烈的自我攻击、或情绪的翻覆远超事件本身,那可能涉及情绪调节的更深层议题(ADHD 常见共病),值得专业评估——时间轴隔离的是失败的传播,不是情绪的源头。

## 今天就能试的 3 件事

1. **把明天排成离散块**:检查依赖,至少让最重要的块不依赖任何前序。
2. **预演「翻页动作」**:下次块失败,标记→看下一块→照常启动,内疚延迟到晚上处理。
3. **今晚开始记块完成率**:一周后看平均值——你会发现「废掉的一天」里,通常也完成了一半。

单次启动的技术,前五篇给齐了;这一篇给的是让一天扛得住失败的架构——**块与块之间的防火墙,比任何一个块的成功都重要**。启动会继续失败,这是常态;但从今天起,失败该在它发生的那个块里,就地埋葬。
