# 为什么用 Speechify 治 ADHD 的任务启动困难，和给 agent 套 function calling 工具调用 是一回事？

> 有一种启动困难专属于一类任务:读。四十页的报告、律师函、说明文档——你不是不想读,是「读」这个动作本身启动不了:眼睛在第一段来回打滑,读了三遍没进脑子,挫败感先于内容抵达。Tiimo 那篇讲过 schema 适配(换呈现格式降低解析门槛),这一篇把同一个思想推到底:**如果一种模态整体不适配,就换一条 IO 通道。** Speechify 做的事情,工程上叫多模态路由:同一份内容,换一个模型读得动的格式送进去。

先看工程侧。多模态系统的一个基本设计:**不同类型的输入,路由到不同的处理通道**——图像走视觉编码器,音频走语音通道,没人会把图片逐像素转成文字硬塞给文本模型。更进一步,当某个通道处理某类输入的效果差时,成熟的做法是**转换模态再路由**:表格转成结构化文本、语音转成文字、文字合成为语音——转换本身廉价,收益是让每份输入走它被处理得最好的通道。**通道选择是架构决策,不是能力评判——没有「更高贵」的模态,只有匹配或不匹配的路由。**

ADHD 侧,阅读这条通道的拥堵有具体成分:**持续视觉注意的维持困难**(长文本要求眼睛做长程的、自控式的线性扫描——恰恰是维持性注意的重灾区)、**回跳与丢行**(注意瞬断后找不回位置,重读消耗翻倍)、**默读的节奏自由度太高**(没有外部节拍,走神后文本不会推着你走)。再叠加一个高共病率的事实:ADHD 与阅读障碍(dyslexia)的共病相当常见——很多人的「读不动」是双重通道税。而同一个人听播客、听书两小时不累:**内容处理能力完好,是视觉文本这条 IO 通道在堵车。**

Speechify 这类 TTS 工具做的就是模态转换+重路由:任何文本(网页、PDF、扫描件)一键转成自然语音,变速播放,边听边高亮跟随。「读四十页」变成「听四十页」,启动门槛从视觉扫描的重税区,搬到听觉通道的顺风路。按「模态路由」的逻辑用:

**一、先做通道审计,再决定路由。** 一周内留意:哪类内容你读得动(短消息?代码?),哪类一读就滑(长报告?合同?)——**路由规则按类型定**:读得动的照读,堵的类型默认走听。全量转听觉是过度矫正,听觉通道也有自己的弱项(跳读、回查、扫结构都不便)。

**二、双通道并行,是被低估的王牌。** 边听边看(眼睛跟着高亮走):**听觉提供节拍(推着你走,走神后把你拽回来),视觉提供锚点(随时知道在哪)**——两条通道互补掉彼此的短板,对「回跳丢行」尤其有效。这也是很多阅读障碍辅助方案的标准配置。

**三、变速要为理解服务,不为效率表演。** ADHD 听语速偏快的内容反而更专注(信息密度低于阈值时,注意力会溜去找别的刺激)——**调到「刚好需要跟紧」的速度**;但复杂内容该降速就降速,3 倍速听完没进脑子,等于给「假完成」交税。

**四、把「转成语音」做进启动流程。** 那份拖了一周的长文档,今天的第一步不是「开始读」,是「丢进 TTS,戴上耳机,散步听」——**换通道本身就是一次点火动作**:走路+听觉输入的组合,比钉在椅子上瞪文档,对这颗大脑友好得多。

## 边界

同构强度 B 级:多模态路由是真实的系统设计,ADHD 的持续注意与阅读维持困难有文献支撑,TTS 辅助对阅读障碍人群有辅助技术研究基础,Speechify 本身无 ADHD 对照研究。声明:若阅读困难从小学起持续存在且远超注意力问题所能解释,值得做阅读障碍的专业评估——那是独立的、可被正式支持的状况(考试延时、辅助技术权利);换通道是聪明的代偿,确诊是应得的权益,两个都要。

## 今天就能试的 3 件事

1. **把那份拖着的长文档转成语音**:今天用「散步+耳机」的方式启动它。
2. **试一次双通道**:边听边看高亮,感受「节拍+锚点」和裸读的差距。
3. **写下你的路由表**:哪类内容默认读、哪类默认听、哪类双通道——贴在工作区。

读不动不是理解力的判决,是路由的误配——**内容要进的是你的脑子,不是非得经过你最堵的那条路**。换条通道,四十页的墙,常常就变成了四十分钟的散步。
