# ADHD 程序员视角：为什么「治好 ADHD 的工作记忆容量小、边做边忘」和「让 LLM 不跑飞」其实是同一道工程题？

> 程序员对「上下文装不下」有最切身的体感:debug 到第五层调用栈,忘了第一层要查什么;改到一半被拉去开会,回来对着自己的代码一脸陌生。有趣的是,你天天给 LLM 设计的外部记忆方案,恰好就是你自己最需要的那套。

先看编程对工作记忆的压榨方式。写代码是持续的多层状态持有:当前函数的逻辑、它在整个调用链里的位置、刚才为什么改那行、还有三个「顺手要改」的地方悬在脑子里。**debug 是重灾区**:排查是一棵深度优先的假设树,每往下钻一层,上层的「待验证假设」就得在工作台上多躺一会儿——ADHD 的工作台浅,钻到第三层,第一层的假设已经掉了,于是重复排查、漏掉线索、或者干脆忘了最初要解决什么。再加上打断:一条 Slack、一个会议,深度状态瞬间清零,重建要 20 分钟起——而 ADHD 的重建成本更高、重建意愿更容易直接崩到「算了先刷会儿手机」。

而你在工作里给 LLM agent 修的正是同款问题:上下文窗口有限→外部状态文件、任务队列、scratchpad(草稿区)、每步落盘的执行日志。**把你写进 agent 框架里的每个组件,给自己也部署一份**:

## 给自己部署 scratchpad

物理便签或固定的 scratch 文件,存三类易失状态:**debug 假设栈**(每个假设一行:「怀疑 X→验证方法→结果」,钻下一层前先落盘当前层——你的深度优先搜索从此带栈回溯,不再靠脑子持栈);**「顺手要改」队列**(写代码时冒出的所有「这里也该改」,一律入队不当场改——当场改是 ADHD 程序员最大的跑偏入口:改 A 时看到 B,改 B 时看到 C,两小时后 A 还开着膛);**断点票**(任何离开键盘前 10 秒写一行:「正在改 X,下一步 Y,注意 Z」——回来后的重建成本从 20 分钟降到 30 秒)。

## 让工具链当你的外部记忆

**小步提交,commit message 写「为什么」**。大块未提交的变更是悬在工作台上的巨石;小步提交把状态持续落盘,message 里写动机(「为什么这么改」——三个月后 git blame 时,你会拥抱当年的自己)。**测试是可执行的记忆**:「这个函数的边界条件是什么」不该靠记,写成测试,永久落盘且自动复查。**TODO 注释+lint 收口**:代码里的 `TODO` 要能被工具汇总(否则就是写进海里),每周扫一次。

## 会议与打断的防御工事

程序员的日历默认对深度工作充满敌意。可谈判的防线:**打断分级**(Slack 免打扰+固定批处理时段回消息;真急事自有电话进来);**会议前的 5 分钟落盘仪式**(写断点票再去开会);**争取整块时间**(和团队约定「上午不排会」之类的深度时段——用「我在深度时段产出翻倍」的实绩去谈,比用诊断去谈有效)。

## 一个职业级彩蛋

ADHD 程序员被工作记忆逼出来的这套「一切外置」的习惯,恰好是**优秀工程实践本身**:小步提交、测试覆盖、文档化决策、可复现环境——神经典型的同事靠自律偶尔做到,你靠生存刚需天天做到。不少 ADHD 工程师最终以「工程规范好得反常」著称,原因无他:**对你,规范不是美德,是呼吸机——所以你从不断电。**

## 边界

同构强度 A 级:scratchpad/状态外置是标准 agent 架构且有效性有研究支撑,ADHD 工作记忆缺陷证据扎实,程序员场景的映射几乎是同一张图纸。提醒:长期高打断环境+deadline 压力下的 ADHD 程序员是 burnout 高危组合,若已出现持续的启动瘫痪和情绪耗竭,那不是流程能修的,请专业支持;药物对工作场景症状的改善有较强证据,值得和医生讨论。

## 今天就能试的 3 件事

1. **开一个 scratch 文件**:今天 debug 时练「假设栈落盘」——每层一行,钻之前先写。
2. **立「顺手要改」队列规则**:冒出来就入队,当前任务完成才领取下一项。
3. **写一张断点票**:今天第一次被打断前抢先体验——离开键盘前 10 秒,三行状态。

你早就知道答案了——你给 agent 写的每一行状态管理代码,都是在替它解决你自己的问题。现在把架构图转 180 度:scratchpad、小步提交、断点票,给自己也来一套。毕竟,你是自己最重要的那个 agent。
