# 为什么用 Routinery 治 ADHD 的注意力涣散，和给 agent 套 上下文工程 是一回事？

> 早晨出门,一个神经典型者的流程是全自动的:洗漱、穿衣、拿包、出门,意识几乎不参与。同样的流程对很多 ADHD 者,每天都是一次现场即兴:下一步干嘛?→ 顺路看见手机 → 刷了十分钟 → 想起来还没刷牙 → 去卫生间路上被别的钩走——**日常例程的每一步之间,都是一个涣散的缺口**。Routinery 这类例程 app 做的事:把流程定义成步骤序列,每步配时,执行时一步一提示,像放幻灯片一样把你从头推到尾。这个机制在 agent 工程里有一个精确的名字:**把高频流程从「每次现场生成」改为「调用预编译模板」——prompt 模板化与工作流固化,上下文工程的省钱头等舱**。

先讲 agent 侧的逻辑。让 LLM 每次从零推理一个高频流程(比如标准的代码审查步骤),既贵又不稳定——每次生成的步骤略有不同,偶尔漏关键项;工程共识是**高频流程固化为模板/工作流**:步骤序列预先定义,运行时只做填空执行——把「推理」从热路径上移走,换来速度、稳定性和更干净的上下文(推理占用的窗口空间也省了)。**能模板化的绝不现场生成**——这是上下文预算管理的第一课。

ADHD 侧的对应,要先纠正一个误解:例程崩溃不是「懒」,是**程序性自动化的形成障碍**。神经典型者重复一个流程几十次后,它沉淀为自动程序(基底节接管,前额叶下班);ADHD 的这条沉淀通道更弱——研究对习惯形成与程序性学习的观察显示,ADHD 者往往重复数百次仍未自动化,**每天的例行流程都还在走「昂贵的前额叶现场推理」路径**:每一步都要想、要决策、要抗干扰——而决策间隙正是自下而上捕获的黄金窗口(注意调度篇)。「为什么别人出门不用想,我要想?」——因为别人在跑编译好的程序,你在逐行解释执行。

Routinery 的机制,就是把编译工作外包给系统:**①流程定义一次,执行调用无限次**——「早晨例程」的步骤序列写死在 app 里,执行时零决策(下一步永远由系统报,不由你想)——**决策缺口被填平,涣散最大的进入点关闭**;**②每步配时+倒计时**——单步有明确的时间容器(刷牙 3 分钟在倒数),对抗单步内的漂移与超时(在浴室神游二十分钟的那种);**③步骤间自动流转**——上一步勾掉,下一步自动弹出,不给「接下来干嘛」留任何开放时刻。三条合起来,效果等于给日常流程装了一个外置的程序执行器——**你的前额叶从流程管理里解放出来,涣散失去了它最日常的猎场**。

两个诚实的提醒:**其一,模板会腐烂**——新奇效应对例程 app 同样生效(用了三周,提示音变成背景噪音),对策是把例程当消耗品维护:定期改顺序、换提示、精简步骤(模板也要重构,工程师都懂);**其二,例程只覆盖「已知流程」**——它治的是高频固定流程里的涣散,对开放性任务(写作、决策)无能为力,那是别的模块的战场;而且例程定义本身别贪多——**先固化一个最痛的流程(通常是早晨),跑稳了再扩**,一次上五个例程的结局是五个一起弃。

## 边界

同构定位(本文未做正式 A/B/C 分级):ADHD 的程序性学习与习惯形成差异有研究基础(证据仍在积累),prompt 模板化是工程标准实践,「预编译 vs 现场推理」的同构是清晰的功能对应;Routinery 无对照研究。声明:例程崩溃若伴随整体功能损害,评估优先;例程工具不适合作为强迫倾向者的仪式放大器——若「必须完美执行流程」带来痛苦,方向就反了。

## 今天就能试的 3 件事

1. **挑一个最痛的流程编译它**:通常是早晨出门——写下步骤+每步分钟数,明早照着跑一遍。
2. **数一次决策缺口**:今天的例行流程里,你「想下一步干嘛」了几次?——每一次都是本可关闭的涣散入口。
3. **给旧例程做一次重构**:已经无感的例程,改顺序或删两步——模板腐烂了就翻新,别弃用。

别人不用想就能出门,不是因为意志强,是因为**他们的流程编译过了,你的还在解释执行**——编译器可以外置,这就是例程 app 的全部原理,也是 harness 给 agent 上模板的全部原理。
