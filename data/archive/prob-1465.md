# Structured 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> Structured 是日程类 App 里形态最简单的一个:一条从早到晚的垂直时间轴,任务是轴上的块,没有 Motion 的算法调度,没有 Reclaim 的习惯引擎——它几乎什么都不替你做。但恰恰是这种简单,让它成为观察「重锚定」这件事的最佳标本:**它的全部价值,取决于你和这条时间轴之间的「注入节奏」——什么时候写它、什么时候看它、什么时候改它。** 同一个 App,注入节奏对的人得到一条全天的锚链,节奏错的人得到一张晚上才想起来的废图。而节奏的错法,和 system prompt 的管理失误一一对应。

先立正确的注入节奏。system prompt 管理的黄金实践是个循环:**会话开始时注入→执行中保持在场→关键节点校对→会话结束后修订**。映射到 Structured 的一天:**早晨五分钟排今天的轴(注入)→白天时间轴挂在可见处(在场)→每完成一块勾掉(校对)→晚上两分钟看「计划 vs 实际」(修订)**。四个节拍,缺哪个,哪里漏。误用四种:

**误用一:晚上排明天,早上不再看(注入时机错位)。** 睡前雄心勃勃排好明天,早上醒来直接被现实卷走,轴一眼没看——**prompt 写好了,忘了随请求发送**。更微妙的问题:睡前的你和早晨的你不是同一个状态,前者habitually 高估后者(睡前排的 7 点晨跑,是对早晨的你的一厢情愿)。修法:**排轴的动作放在早晨,或至少早晨花六十秒「确认注入」**——过一遍昨晚的安排,砍掉睡前的浪漫主义,轴才对今天生效。

**误用二:轴只在手机里,不在视野里(注入后不在场)。** 打开 App 才能看到的时间轴,对 ADHD 约等于不存在——「打开看一眼」本身就是需要被提醒的动作,递归了。工程对应:prompt 在数据库里躺着,没进上下文窗口。修法:**小组件放主屏第一屏(显示当前块+下一块)、手表表盘常驻**——让时间轴出现在你无论如何都会看到的地方;每次解锁手机的瞬间,就是一次微注入。

**误用三:计划赶不上变化时,弃轴而不是改轴。** 十点被临时会议打穿,轴和现实脱节——**从此刻起你按混沌模式运行,轴成了「今天失败」的纪念碑**。工程对应:上下文和现实冲突后不做修正,后续生成全部基于过期信息。修法(弹性规划的纪律,再强调一次):**脱节的那一刻,花九十秒重排剩余的块**——删、缩、移;轴的权威不来自「从不被改」,来自「永远反映当前的真实计划」。可以被快速修订的锚才是活锚。

**误用四:只用轴,不复盘轴(修订环节缺失)。** 每天排、每天执行(或执行不完),但从不回看「计划与实际的差距」——**于是同样的错误参数(高估上午产能、低估通勤耗时)日复一日地被写进新的轴**。工程对应:prompt 从不根据失败案例迭代,同一个坑每天掉一次。修法:晚上两分钟,只问两个问题——**今天哪个块崩了?是参数错(时长/时段)还是意外(下次预留缓冲)?**把答案直接改进明天的轴;复盘不是反省仪式,是参数更新。

## 边界

同构强度 B 级:注入节奏与上下文管理是真实的工程实践,ADHD 的计划-执行脱节与时间参数失真有文献支撑,Structured 本身无对照研究。声明:时间轴是结构层;排得再好也不解决块内的启动(见启动系列)与能量供给——若每天的轴都在为「根本排不进生活的总量」腾挪,那是负载问题,值得被正视而不是被更精巧地排布。

## 今天就能试的 3 件事

1. **明早花五分钟排轴**(或确认昨晚的),砍掉一切浪漫主义参数。
2. **小组件上主屏**:当前块+下一块,今天配置好。
3. **今晚做第一次九十秒复盘**:哪个块崩了,参数怎么改——直接写进明天。

最简单的工具最诚实:它不会替你锚定,只会如实暴露你的注入节奏——**早晨写、全天见、崩了改、晚上修**。四个节拍打齐,一条朴素的时间轴,就够撑起一整天的秩序。
