# ADHD 程序员视角：为什么「治好 ADHD 的状态日内大幅波动、不稳定」和「让 LLM 不跑飞」其实是同一道工程题？

> ADHD 程序员是这道工程题里最特殊的解题人:**你们每天的本职工作,就是给不可靠组件设计可靠系统**——重试、超时、降级、幂等、监控告警,这些肌肉记忆你们全有;讽刺的是,下班后管理自己时,多数人把整套工程学扔在工位上,回家继续用「明天我一定早点开始」这种裸调用管理自己。这一篇不用从头讲原理,只做一件事:**把你已经会的可靠性工程,逐条映射到你自己身上**——毕竟你就是那个高方差系统,而你恰好是专业的。

映射表逐行过。**「状态好坏」→ 资源可用性,先加监控**:没有 metrics 的系统没法优化——给自己装最简监控:每天三次状态打分(1–5),两周后你会拿到自己的负载曲线(几点是高峰、周几容易崩、睡眠怎样影响次日)——**你不会在没有监控的服务上做容量规划,凭什么在自己身上做?**;**「深度工作」→ 高优先级任务,要资源预留**:你的高峰窗口(多数程序员实测在上午或深夜)是稀缺计算资源,复杂设计与难 bug 只在这个窗口调度;code review、写文档、修 CI、清 backlog 这类低认知任务,压进低谷时段——**把 P0 任务调度到资源低谷,是你绝不会对生产系统做的事**;**「状态崩了」→ 触发降级预案,不是抛异常给用户**:预先写好你的 degraded mode 任务单(整理依赖、补注释、清理分支、跑跑腿),状态崩的下午照单执行——**优雅降级的核心是「预案写在事故之前」**,对人同理:崩溃当下的大脑没有能力现场规划;**「昨天的超专注」→ 突发流量,不是新基线**:超专注冲了一夜写完一个模块,最危险的后续是把它当作「我的真实产能」来做 sprint 承诺——**你不会按峰值 QPS 做容量承诺,别按超专注日估算工期**;估算一律用历史均值(含烂日子的那种),然后像对待所有工期一样再乘个系数;**「hyperfocus 跑偏」→ 没有超时控制的循环**:超专注是双刃剑——四小时优化一个无关紧要的脚本、重构没人要求的模块,是 ADHD 程序员的经典事故;给深潜加 watchdog:番茄钟不是用来打断专注的,是**每 90 分钟弹一个「我还在主线任务上吗」的检查点**——允许继续,但必须是显式决定而不是无意识滑行(目标漂移的自检,agent 跑长任务的标准配置);**「写一半的代码」→ 未提交的事务,收工必须落盘**:ADHD 程序员的次日启动困难,一半是因为昨天的状态没序列化——收工仪式:git commit(哪怕 WIP)+ 给明天的自己留三行(做到哪/卡在哪/下一步)+ 未完成思路写进 TODO 注释——**明早的你是另一个无状态实例,今天不落盘,明天就冷启动**(四行笔记的程序员版,你们本来就有最好的工具链)。

程序员场景还有两个特有的放大器要点名。**其一,中断的代价对你是平方级的**:上下文重建对深度编码状态尤其昂贵(任务恢复研究的方向,对程序员群体有专门讨论),而 ADHD 的恢复成本更高——所以 IM 通知、随手瞄一眼的群、开着的社交 tab,对你不是「分心」是「事务回滚」;深度窗口内通知全关,不是自律,是资源隔离。**其二,你的职业环境是最好的伪装同时是最好的掩护**:程序员文化对「怪异的作息」「戴耳机勿扰」「用一堆工具管理自己」宽容度极高——**别的行业的 ADHD 者要争取的 accommodations,你的行业默认发放**——用足它:异步沟通、灵活时段、文档优先,这些工程团队的最佳实践,恰好都是 ADHD 友好设计。

## 边界

同构强度 A 级:可靠性工程(监控/降级/超时/幂等)是软件工程实体,表现变异性与中断恢复成本有认知研究基础,程序员场景的双侧结构对应完整;个人策略为实践翻译。声明:若「崩溃时段」伴随持续情绪问题或倦怠,专业评估优先;本文不替代职业健康支持。

## 今天就能试的 3 件事

1. **给自己装监控**:今天起每日三次状态打分,记进任何地方——两周后做一次你自己的容量规划。
2. **写好 degraded mode 任务单**:五件低状态也能做的工程杂务——下次崩的下午,照单执行,不硬扛 P0。
3. **今晚执行落盘仪式**:WIP commit + 给明天的三行 + TODO 注释——明早的冷启动成本,今晚付掉。

你给每个不可靠组件都设计过重试、降级和监控,**唯独对自己裸跑**。你就是你架构里方差最大的那个服务——用你吃饭的那套工程学对待它,这可能是你做过投产比最高的一次可靠性改造。
