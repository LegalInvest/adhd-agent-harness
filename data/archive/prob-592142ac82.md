# 为什么用 Claude 治 ADHD 的卡在执行与落地，和给 agent 套 外部编排调度层 是一回事？

> 用 Claude 对付「卡在执行与落地」的 ADHD 用户,很快会发现一个吊诡的现象:**对话越精彩,落地越可疑**。你把项目困境丢给 Claude,它给出漂亮的拆解、周密的计划、鼓舞人心的建议——你合上电脑,感觉「今天推进了很多」——**但现实世界里没有任何原子被移动过**:文件还是没写,电话还是没打。这个现象在 agent 工程里有精确的对应物:**一个只有 LLM 没有编排层(orchestration layer)的系统**——模型每次调用都能产出优秀的「下一步该做什么」,但没有任何组件负责把输出转换成动作、跟踪执行状态、在中断后恢复——**推理在空转,因为没有传动轴**。

对应关系拆到组件级。agent 的编排调度层至少做四件事,每件都恰好是「Claude+ADHD 用户」这个组合缺掉的:**①状态持有**——编排层记住任务进行到哪一步了;而 Claude 的对话是无状态的(新会话即失忆),ADHD 用户的工作记忆同样易失——**两个健忘系统对话,状态没有家**;②**动作绑定**——编排层把「计划的步骤」绑定到「实际的工具调用」;而聊天窗口里的计划没有绑定任何现实动作——「周三写初稿」停留为一行字,没有变成日历事件、没有闹钟、没有触发器(1762 篇的教训:停留在对话窗口里的计划视为不存在);③**执行监督**——编排层检查每步是否真的完成、失败了重试;而 Claude 不会知道你昨天到底写没写(除非你说,而你不说);④**中断恢复**——编排层从断点继续;而 ADHD 用户离开三天后回来,面对的是一条冷掉的聊天记录和一样冷掉的项目。**诊断一句话:Claude 是一颗优秀的规划核心,但「执行与落地」的瓶颈从来不在规划质量,在编排——你缺的不是更聪明的建议,是传动系统**。

解法因此不是「更好地提问」,而是**在 Claude 外面手工搭一层编排**,四个组件逐个补:**①给状态一个家**:每个项目维护一份固定的「项目状态文档」(当前步骤/已完成/卡点),每次和 Claude 讨论前把它粘进对话(上下文注入),讨论后更新它——**状态存在文档里,不存在对话里,更不存在脑子里**;②**建立「对话→日历」的强制转换**:每次 Claude 给出计划,结束前追加一个固定指令——「把以上计划压缩成三个带日期的日历条目」——然后**当场录入日历,不录入不许关窗口**(动作绑定的人肉版,把仪式做成硬规则);③**让 Claude 当执行监督**:固定每天/隔天一次的「汇报会话」——粘贴状态文档+一句「昨天完成了 X,没完成 Y」——让它基于事实调整计划;**汇报的价值一半在它的调整,一半在「有个东西在等我汇报」的问责压力**(human-in-the-loop 的反转:人是 executor,AI 是 supervisor);④**断点协议**:每次会话结束让 Claude 生成三行「断点摘要」(做到哪/下一步/注意什么)存进状态文档——三周后回来,重启成本从「考古整条聊天记录」降为「读三行」。

边界照例钉牢:**Claude 是规划与监督组件,不是治疗**——「执行与落地」的困难若伴随广泛的功能损害,正式评估与治疗是地基,工具是地基上的脚手架;这套手工编排本身有维护成本(1764 篇的维护税),预期它会周期性失守,失守后用重启协议,不用羞耻。

## 边界

同构强度 B 级:执行功能的「意图-行动断裂」有研究方向,编排调度层是 agent 工程实体,「规划核心缺传动系统」的结构对应清晰;四组件方案为实践翻译,无对照研究。声明:Claude 未被证实为 ADHD 干预手段,本文方案为自组织策略;显著功能损害请寻求专业评估。

## 今天就能试的 3 件事

1. **建第一份项目状态文档**:当前步骤/已完成/卡点,三栏——从今天起,和 Claude 的每次讨论以它开头和结尾。
2. **立「不进日历不关窗」规则**:下次 Claude 给出计划,让它压成三个带日期的条目,录完日历才算讨论结束。
3. **约一次汇报会话**:明天固定时间,向 Claude 汇报「完成了什么、没完成什么」——让它当你的执行监督。

Claude 的建议再好,也只是引擎的轰鸣——**没有编排层,马力到不了轮子**。状态文档、日历绑定、汇报会话、断点摘要:四个零件装上,对话才开始驱动现实。
