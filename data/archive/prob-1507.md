# Routinery 之于 ADHD，就像 重锚定 system prompt 之于 LLM——但有人用错了

> function calling 那篇讲了 Routinery 当「工作流引擎」:把固定序列打包成宏,消灭接缝决策。这一篇换一个视角看同一个 App——重锚定系列的收官视角。前面十篇讲的注入载体(原则页、日历、时间轴、音景、宣告、brief、宪法区、锚便签、每日模板)有一个共同的软肋:**它们都需要在某个时刻「被看见」,而看见之后是否照做,还是悬着的**。例程是唯一的例外:它不展示你的原则,它**把原则直接编译成了每天必然执行的动作序列**——晨间例程里有「看重心页十秒」这一步,方向注入就不再依赖任何自觉。例程是被编译执行的 system prompt。但编译这条路,有它自己的深坑。

先看「编译」的含义。声明的原则(「我要重视健康」)和执行的例程(晨间例程第四步:吃药+接水)之间,隔着一次翻译:**把价值观翻译成固定时刻的固定动作**。翻译完成后,原则的兑现不再需要每天重新决策——这对决策昂贵的 ADHD 是最高级的省电模式:**最重要的事,恰恰不该依赖每天的状态,该被编译进流程。** 误用四种:

**误用一:例程和原则脱钩(编译产物没有源代码)。** 很多人的例程是抄来的:名人的晨间流程、生产力博主的模板——**每一步都执行了,但没有一步来自你自己的原则**;跑了三个月,生活没有朝任何你在乎的方向移动。工程对应:一段没人知道对应什么需求的祖传代码,跑得很稳,不知道在为谁服务。修法:**给例程做一次「反编译」——每一步写出它服务的原则**;写不出来的步骤,要么删,要么承认「它只是看起来很自律」。

**误用二:原则改了,例程没重编译(过期的可执行文件)。** 换了工作、换了作息、有了新目标——原则层早变了,例程还在跑旧版本:**每天 6:30 起床跑步的例程,忠实地服务着一个已经不存在的生活**。这比没有例程更隐蔽,因为「我有例程且在执行」的感觉太好了,好到掩盖了「执行的是过时指令」。修法:**原则修订会(Saner 篇的月度十五分钟)加一个议程:现有例程还对齐吗?**——prompt 更新了,记得重新编译。

**误用三:把一切都编译(过度自动化)。** 尝到例程甜头后的滑坡:晨间、午间、晚间、周末……**清醒时间的大半被例程覆盖**——但生活里有大量本该保持弹性的部分(社交、玩、发呆),编译它们等于把弹性也写死;而且例程越多,总违约率越高,系统的权威被摊薄。工程对应:什么都塞进 system prompt 的膨胀病,换了个马甲。修法:**只编译两类东西——最重要的(重心相关)和最容易漏的(吃药、备份、账单)**;其余留给自由意志,自由意志也需要跑道。

**误用四:违约后修例程,不查状态(错怪了代码)。** 例程连续崩,第一反应是重新设计例程——换顺序、改时长、换 App——**但很多时候崩的原因根本不在例程层:睡眠崩了、药物调整期、情绪低谷**。工程对应:模型输出变差,工程师疯狂改 prompt,其实是底层模型出了问题——prompt 调优治不了模型故障。修法:例程崩溃超过三天,先查状态层(睡眠、饮食、情绪、用药变化),**状态异常时的正确动作不是优化例程,是启用「最低配模式」**(只保吃药+吃饭+睡觉三步的灾难版例程),等状态回来再恢复全量。

## 边界

同构强度 B 级:例程结构化是 ADHD 行为干预的常规组件,「编译」是把行为科学的习惯自动化用工程语言重述,Routinery 本身无对照研究。声明:例程的前提是可编译的生活——作息被外部力量(轮班、育儿、疾病)高度打散的人,先争取结构的可能性,再谈结构的优化;状态层的持续异常(长期失眠、情绪低谷)请专业评估,那不是任何例程设计能覆盖的辖区。

## 今天就能试的 3 件事

1. **反编译你的例程**:每步标出服务的原则,标不出的删掉或降级。
2. **把一条重要原则编译进去**:比如「重心页十秒」加进晨间例程第二步。
3. **写好你的灾难版例程**:状态崩的日子只跑三步——哪三步,现在就定。

原则说一千遍,不如编译一次——**让最重要的事住进每天必然发生的序列里,让坚持不再消耗坚持**。但记得回头看源代码:例程跑得再稳,也要确认它服务的,还是今天的你。
